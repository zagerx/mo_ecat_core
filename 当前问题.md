# 当前问题：从站类型未区分

## 1. 问题背景

当前 `EcatController::Initialize()` 的简化流程如下：

```cpp
auto slave_infos = master_.ScanSlaves();
// ...
node_manager_.Initialize(master_, slave_infos);
```

`SlaveNodeManager::Initialize()` 内部为每个从站创建一个 `SlaveNode`：

```cpp
for (const auto &info : slave_infos) {
    SlaveConfig config = MakeDefaultSlaveConfig(info.slave_id);
    nodes_.emplace_back(std::make_unique<SlaveNode>(master, config, info));
}
```

`MakeDefaultSlaveConfig()` 对所有从站返回**完全相同的配置**：一组默认的 CiA402 位置/速度/力矩 PDO 映射，以及一个 `ServoAxis`。

## 2. 当前代码的问题

**所有从站被当成同一个类型的设备处理**，没有根据实际设备类型做区分。

具体表现为：

1. **不区分伺服驱动器和 IO 模块**
   - IO 模块没有 `target_position`、`actual_position` 等概念
   - 但代码里依然会尝试读写这些 PDO 条目
   - 找不到对应条目时只是静默跳过，没有类型级别的处理

2. **不区分不同厂家的伺服驱动器**
   - 不同厂家（如 Panasonic、Mitsubishi、Inovance）的 PDO 映射不同
   - 默认的 16 字节输出 / 16 字节输入映射不一定适用

3. **后续扩展困难**
   - 如果以后给伺服驱动器加 CiA402 状态机，IO 模块也会被错误地当成电机来管理
   - 如果要支持模拟量模块、编码器模块等，无法优雅扩展

4. **上层无法知道设备类型**
   - 上层只能通过索引或名字访问 `SlaveNode`
   - 无法判断这个从站是电机还是 IO，从而无法做类型-specific 的操作

## 3. 造成这个问题的原因

- `SlaveInfo` 里已经包含 `vendor_id`、`product_id`、`name` 等可用于识别设备的信息
- 但 `SlaveNodeManager::Initialize()` 完全没有使用这些信息
- 当前只有 `MakeDefaultSlaveConfig()` 一个配置生成函数，没有按类型分发

## 4. 解决目标

在扫描从站之后、创建 `SlaveNode` 之前，**根据从站信息识别设备类型**，然后：

- 为不同类型的设备生成不同的 PDO 映射
- 为不同类型的设备挂载不同的业务对象（如 `ServoAxis`、`DigitalIo`、`AnalogIo`）
- 让上层能够按类型访问和操作从站

## 5. 方案设计

### 5.1 增加 `SlaveType` 枚举

```cpp
enum class SlaveType {
    Unknown,      // 未知设备
    Servo,        // 伺服驱动器
    DigitalIo,    // 数字 IO 模块
    AnalogIo,     // 模拟量 IO 模块
    // 后续可扩展：Encoder, ForceSensor, etc.
};
```

### 5.2 增加从站类型识别器

```cpp
class SlaveTypeDetector {
public:
    static SlaveType Detect(const SlaveInfo &info);
};
```

识别策略可以基于：

- **Vendor ID + Product ID**：最准确，但需要维护一张设备表
- **从站名称关键字**：如名字里含 " servo "、" io "、" dig " 等
- **配置文件**：通过 YAML/JSON 配置 vendor_id/product_id 到类型的映射

示例（基于关键字）：

```cpp
SlaveType SlaveTypeDetector::Detect(const SlaveInfo &info) {
    const std::string name = ToLower(info.name);
    if (name.find("servo") != std::string::npos ||
        name.find("motor") != std::string::npos) {
        return SlaveType::Servo;
    }
    if (name.find("dio") != std::string::npos ||
        name.find("digital") != std::string::npos) {
        return SlaveType::DigitalIo;
    }
    if (name.find("aio") != std::string::npos ||
        name.find("analog") != std::string::npos) {
        return SlaveType::AnalogIo;
    }
    return SlaveType::Unknown;
}
```

### 5.3 增加 `SlaveConfigFactory`

不同类型生成不同的 `SlaveConfig`：

```cpp
class SlaveConfigFactory {
public:
    static SlaveConfig Create(SlaveType type, int slave_id);
};
```

示例：

```cpp
SlaveConfig SlaveConfigFactory::Create(SlaveType type, int slave_id) {
    switch (type) {
    case SlaveType::Servo:
        return MakeServoSlaveConfig(slave_id);
    case SlaveType::DigitalIo:
        return MakeDigitalIoSlaveConfig(slave_id);
    case SlaveType::AnalogIo:
        return MakeAnalogIoSlaveConfig(slave_id);
    default:
        return MakeDefaultSlaveConfig(slave_id);
    }
}
```

### 5.4 重构 `SlaveNode` 的业务对象挂载方式

当前 `SlaveNode` 直接包含一个 `ServoAxis`：

```cpp
class SlaveNode {
    // ...
    ServoAxis axis_;
};
```

这样所有从站都被迫有一个 `ServoAxis`，对 IO 模块不合理。

改为：

```cpp
class DeviceProfile {
public:
    virtual ~DeviceProfile() = default;
    virtual SlaveType GetType() const = 0;
    virtual void UpdatePdoOutput(...) = 0;
    virtual void UpdatePdoInput(...) = 0;
};

class ServoProfile : public DeviceProfile {
    ServoAxis axis_;
    // ...
};

class DigitalIoProfile : public DeviceProfile {
    // ...
};

class SlaveNode {
    // ...
    std::unique_ptr<DeviceProfile> profile_;
};
```

`SlaveNode` 只负责 EtherCAT 通信状态、SDO、PDO 映射，具体的业务对象通过 `DeviceProfile` 挂载。

### 5.5 `SlaveNodeManager::Initialize()` 按类型创建节点

```cpp
bool SlaveNodeManager::Initialize(EcMaster &master,
                                  const std::vector<SlaveInfo> &slave_infos)
{
    Clear();
    for (const auto &info : slave_infos) {
        SlaveType type = SlaveTypeDetector::Detect(info);
        SlaveConfig config = SlaveConfigFactory::Create(type, info.slave_id);

        if (!info.name.empty()) {
            config.name = info.name;
            config.axis_config.name = info.name;
        }

        auto node = std::make_unique<SlaveNode>(master, config, info);
        node->SetDeviceProfile(DeviceProfileFactory::Create(type));
        nodes_.push_back(std::move(node));
    }
    return true;
}
```

## 6. 预期效果

### 对上层应用

```cpp
auto *node = controller.GetSlaveNodeManager().GetNode(0);
if (node && node->GetType() == SlaveType::Servo) {
    auto *servo = node->GetProfile<ServoProfile>();
    servo->GetAxis().SetTargetPosition(1.0);
}
```

### 对系统架构

```text
扫描从站
    │
    ▼
SlaveTypeDetector 识别类型
    │
    ├── Servo      → ServoProfile + ServoAxis
    ├── DigitalIo  → DigitalIoProfile
    ├── AnalogIo   → AnalogIoProfile
    └── Unknown    → GenericProfile
```

## 7. 实施步骤建议

1. **第一步**：增加 `SlaveType` 枚举和 `SlaveTypeDetector`
   - 先基于从站名称关键字做简单识别
   - 后续再扩展为 vendor_id/product_id 表

2. **第二步**：把 `SlaveConfig` 的生成从单一函数改为工厂
   - `MakeDefaultSlaveConfig()` 改名为 `MakeServoSlaveConfig()`
   - 增加 `MakeDigitalIoSlaveConfig()`、`MakeAnalogIoSlaveConfig()`

3. **第三步**：把 `ServoAxis` 从 `SlaveNode` 中解耦
   - 引入 `DeviceProfile` 接口
   - `ServoAxis` 放到 `ServoProfile` 里
   - `SlaveNode` 通过 `profile_` 指针持有业务对象

4. **第四步**：更新 `SlaveNodeManager::Initialize()`
   - 识别类型 → 生成配置 → 创建节点 → 挂载对应 Profile

5. **第五步**：更新文档和测试

## 8. 注意事项

- 当前阶段不需要实现所有设备类型，先实现 `Servo` 和 `Unknown` 即可
- `DigitalIo` / `AnalogIo` 可以保留为框架，等实际硬件接入时再补
- 类型识别策略越准确越好，但初期用名称关键字最简单
- 不兼容旧代码，旧的 `MakeDefaultSlaveConfig()` 直接改名或删除
