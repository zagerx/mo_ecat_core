# 当前问题与详细解决方案

> 本文档整合了两个密切相关的问题：
> 1. PDO 映射来源问题（原方案 B）
> 2. 主站缺乏生命周期状态机、从站操作失败处理不完善的问题
>
> 最终目标：通过 CoE SDO 动态读取 PDO 映射，并用显式状态机管理主站/从站生命周期，确保每一步可检查、可回滚、可排错。

---

## 1. 问题总结

### 1.1 PDO 映射完全硬编码

`SlaveConfigFactory::CreateConfig()` 根据 `SlaveType` 分发到四个 `Make*Config()` 方法，所有 `cfg.mapping` 都是手写常量，不依赖 ESI / EEPROM / CoE。

| 从站类型 | output_bytes | 输出条目 | input_bytes | 输入条目 |
|---|---|---|---|---|
| Servo | 10 | target_position, target_velocity, target_torque | 10 | position_actual_value, velocity_actual_value, torque_actual_value |
| DigitalIo | 1 | 无条目 | 1 | 无条目 |
| AnalogIo | 4 | 无条目 | 4 | 无条目 |
| Unknown | 0 | 空 | 0 | 空 |

关键证据：
- `CreateConfig()` 只用 `info.name`，`info.output_bytes` / `info.input_bytes` 等字段被忽略。
- `SlaveNode::ConfigurePdoMapping()` 当前是空壳。
- `SetPdoMapping()` 提供运行时替换入口，但无调用点。

### 1.2 主站缺少生命周期状态机

当前初始化流程是一条直线，没有显式状态，失败只靠单个 `bool` 返回：

```
EcatController::Initialize()
  └─ master_.Initialize()      // ecx_init()
  └─ master_.ScanSlaves()      // ecx_config_init()  → 从站在 INIT
  └─ node_manager_.Initialize() // 创建 SlaveNode，调用空壳 ConfigurePdoMapping()
```

问题：
- `ScanSlaves()` 之后从站处于 **INIT**，邮箱不可用，无法执行 SDO。
- `ConfigureProcessData()` / `ConfigureDc()` 已存在但**从未被调用**。
- `StartOperation()` 直接请求 OP，没有确认前置状态（SAFEOP）。
- `Stop()` 中 `RequestSafeOpState()` / `RequestInitState()` 失败也不处理。
- `RequestState()` 内部不做前置状态校验，也不区分"超时"和"从站报错"。

### 1.3 失败处理薄弱

- 多从站场景下，部分成功、部分失败时没有记录和回滚。
- 状态转换失败后，系统可能停留在"半初始化"状态。
- 没有统一的地方确认"所有从站是否都到达了目标状态"。

---

## 2. 设计目标

1. **动态 PDO 映射**：通过 CoE SDO 读取 `0x1C12/0x1C13/0x1600+/0x1A00+`，自动构建 `PdoMapping`。
2. **显式状态机**：为 `EcatController`/`EcMaster` 建立 INIT → PREOP → SAFEOP → OP 的生命周期。
3. **前置条件检查**：每个操作前确认当前状态合法。
4. **失败可恢复**：状态转换失败可重试、可回滚；SDO 读取失败可回退到硬编码映射。
5. **最小侵入**：尽量不改动 `DeviceProfile` 等现有业务接口，集中改 `EcMaster`、`EcatController`、`SlaveNode`。

---

## 3. 主站生命周期状态机

### 3.1 状态定义

在 `EcatController` 层维护一个高层生命周期状态：

```cpp
enum class ControllerState {
    kUninitialized,   // 未初始化
    kInitDone,        // ecx_init 成功
    kScanned,         // ecx_config_init 成功，从站在 INIT
    kPreOp,           // 所有从站已到达 PREOP，可执行 SDO
    kPdoConfigured,   // PDO 映射已读取并写入 SlaveNode
    kSafeOp,          // ecx_config_map_group 成功，从站在 SAFEOP
    kDcConfigured,    // DC 配置完成
    kOperational,     // 所有从站在 OP，周期运行中
    kError            // 出现不可恢复错误
};
```

`EcMaster` 层保留对 SOEM 实际 EtherCAT 状态的读取和切换能力，但不强制维护高层状态。

### 3.2 状态转换图

```text
                Initialize()
Uninitialized ────────────────→ InitDone

                  ScanSlaves()
InitDone ───────────────────────→ Scanned
                                     │
                                     │ RequestPreOpState()
                                     ▼
                                  PreOp
                                     │
                                     │ ConfigurePdoMapping()
                                     ▼
                              PdoConfigured
                                     │
                                     │ ConfigureProcessData()
                                     ▼
                                  SafeOp
                                     │
                                     │ ConfigureDc()
                                     ▼
                              DcConfigured
                                     │
                                     │ StartOperation()
                                     ▼
                              Operational
                                     │
                                     │ Stop() / 错误
                                     ▼
                               Uninitialized / Error
```

### 3.3 转换规则（前置条件 + 允许操作）

| 当前状态 | 允许操作 | 前置条件 | 失败策略 |
|---|---|---|---|
| Uninitialized | `Initialize()` | 网口未绑定 | 报错，保持 Uninitialized |
| InitDone | `ScanSlaves()` | `ecx_init()` 成功 | 关闭 SOEM，回到 Uninitialized |
| Scanned | `RequestPreOpState()` | 已扫描到从站 | 尝试回滚到 INIT |
| PreOp | `ConfigurePdoMapping()` | 所有从站 PREOP | 回退到硬编码映射，状态可到 PdoConfigured |
| PdoConfigured | `ConfigureProcessData()` | 映射已确定 | 回滚到 PREOP |
| SafeOp | `ConfigureDc()` | IOmap 已建立 | 可继续（DC 非必需） |
| DcConfigured | `StartOperation()` | 所有从站 SAFEOP | 回滚到 SAFEOP |
| Operational | `Stop()` | 周期线程已停止 | 尽量回到 INIT |
| Error | `Shutdown()` | 任意 | 清理资源 |

## 4. 初始化流程重构

### 4.1 目标调用链

```cpp
bool EcatController::Initialize(const EcMasterConfig &config)
{
    if (!master_.Initialize(config)) return false;
    state_ = ControllerState::kInitDone;

    auto slave_infos = master_.ScanSlaves();
    if (slave_infos.empty()) { Shutdown(); return false; }
    state_ = ControllerState::kScanned;

    // ★ 关键新增：先进入 PREOP，邮箱才可用
    if (!master_.RequestPreOpState()) {
        LOG_ERROR << "Failed to enter PREOP";
        Shutdown();
        return false;
    }
    state_ = ControllerState::kPreOp;

    // 创建 SlaveNode，此时 ConfigurePdoMapping() 内部可用 SDO
    if (!node_manager_.Initialize(master_, slave_infos)) {
        Shutdown();
        return false;
    }
    state_ = ControllerState::kPdoConfigured;

    // ★ 关键修复：接入这两个已存在但未调用的函数
    if (!master_.ConfigureProcessData()) {
        Shutdown();
        return false;
    }
    state_ = ControllerState::kSafeOp;

    if (!master_.ConfigureDc()) {
        // DC 非致命，可继续
        LOG_WARN << "DC configuration skipped/failed";
    }
    state_ = ControllerState::kDcConfigured;

    initialized_ = true;
    return true;
}
```

### 4.3 RequestStateWithRetry 实现要点

```cpp
bool EcMaster::RequestStateWithRetry(int slave, uint16_t state, int max_retries)
{
    std::lock_guard<std::mutex> lock(soem_mutex_);

    for (int i = 0; i <= max_retries; ++i) {
        // 1. 先读取实际状态
        ecx_readstate(&ctx_);
        uint16_t actual = ctx_.slavelist[slave].state;

        // 2. 如果已经在目标状态，直接成功
        if (actual == state) return true;

        // 3. 如果处于 ERROR 状态，先 ACK
        if (actual & EC_STATE_ERROR) {
            ctx_.slavelist[slave].state = (actual & 0x0f) | EC_STATE_ACK;
            ecx_writestate(&ctx_, slave);
            ecx_statecheck(&ctx_, slave, (actual & 0x0f), EC_TIMEOUTSTATE);
            continue;
        }

        // 4. 请求目标状态
        ctx_.slavelist[slave].state = state;
        ecx_writestate(&ctx_, slave);
        ecx_statecheck(&ctx_, slave, state, EC_TIMEOUTSTATE);

        // 5. 确认
        if (ctx_.slavelist[slave].state == state) {
            return true;
        }

        LOG_WARN << "State request to 0x" << std::hex << state
                 << " failed for slave " << slave
                 << ", actual=0x" << actual
                 << ", retry " << (i + 1) << "/" << max_retries;
    }

    return false;
}
```

### 4.4 CheckAllSlavesInState 实现要点

```cpp
bool EcMaster::CheckAllSlavesInState(uint16_t state)
{
    std::lock_guard<std::mutex> lock(soem_mutex_);
    ecx_readstate(&ctx_);

    bool all_ok = true;
    for (int i = 1; i <= ctx_.slavecount; ++i) {
        if (ctx_.slavelist[i].state != state) {
            LOG_ERROR << "Slave " << i << " [" << ctx_.slavelist[i].name
                      << "] state=0x" << std::hex << ctx_.slavelist[i].state
                      << ", expected=0x" << state;
            all_ok = false;
        }
    }
    return all_ok;
}
```

## 5. ConfigurePdoMapping 实施方案（方案 B）

### 5.1 调用位置

`SlaveNodeManager::Initialize()` 中，创建 `SlaveNode` 后显式调用：

```cpp
bool SlaveNodeManager::Initialize(EcMaster &master, const std::vector<SlaveInfo> &slave_infos)
{
    Clear();
    SlaveTypeDetector detector;
    SlaveConfigFactory config_factory;

    for (const auto &info : slave_infos) {
        const SlaveType type = detector.Detect(info);
        SlaveConfig config = config_factory.CreateConfig(info, type);

        auto node = std::make_unique<SlaveNode>(master, config, info);

        // ★ 新增：动态读取 PDO 映射
        if (!node->ConfigurePdoMapping()) {
            LOG_WARN << "Slave " << info.slave_id << " [" << info.name
                     << "] failed to configure PDO mapping, fallback to factory default";
            // 失败时保留 factory default，不阻断初始化
        }

        nodes_.emplace_back(std::move(node));
    }

    return true;
}
```

> 注意：此处失败不阻断初始化，而是回退到硬编码默认值。这样可以在新旧行为之间平滑过渡，也便于调试验证。

### 5.2 CoE SDO 读取流程

对每个从站执行以下步骤：

```text
RxPDO（输出）
1. SDO Read 0x1C12:0x00  → RxPDO 使能数量 N
2. for i = 1 .. N:
      SDO Read 0x1C12:i  → 第 i 个 RxPDO 索引（如 0x1600, 0x1601）
3. for 每个 RxPDO 索引 PDO_idx:
      SDO Read PDO_idx:0x00  → 该 PDO 映射条目数 M
      for j = 1 .. M:
          SDO Read PDO_idx:j  → 映射条目值（4 字节）

TxPDO（输入）
4. SDO Read 0x1C13:0x00  → TxPDO 使能数量 N'
5. for i = 1 .. N':
      SDO Read 0x1C13:i  → 第 i 个 TxPDO 索引（如 0x1A00, 0x1A01）
6. for 每个 TxPDO 索引 PDO_idx:
      SDO Read PDO_idx:0x00  → 该 PDO 映射条目数 M'
      for j = 1 .. M':
          SDO Read PDO_idx:j  → 映射条目值（4 字节）
```

### 5.3 映射条目解码

每个条目是 4 字节（32 位），编码如下：

```text
Bit 31..16 : Index    (对象字典索引，如 0x6040)
Bit 15..8  : Subindex (子索引，如 0x00)
Bit 7..0   : BitLen   (位长度，如 0x10 = 16bit)
```

示例：
- `0x60400010` → Index=0x6040, Subindex=0x00, BitLen=16 → Controlword
- `0x60640020` → Index=0x6064, Subindex=0x00, BitLen=32 → Position actual value

### 5.4 构建 PdoMapping

当前 `PdoEntry` 是字节精度的（`offset` + `size`，单位字节）。为了支持位精度 PDO，建议扩展：

```cpp
struct PdoEntry {
    std::string name;        // 语义名称，如 "controlword"
    uint16_t index = 0;      // 对象字典索引
    uint8_t subindex = 0;    // 子索引
    int bit_offset = 0;      // 在 PDO buffer 中的位偏移
    int bit_length = 0;      // 位长度
};
```

同时 `ComputeOutputSize` / `ComputeInputSize` 需要按位计算最大占用：

```cpp
size_t SlaveNode::ComputeOutputSize(const PdoMapping &mapping)
{
    int max_bit = 0;
    for (const auto &entry : mapping.outputs) {
        max_bit = std::max(max_bit, entry.bit_offset + entry.bit_length);
    }
    size_t size = (max_bit + 7) / 8;
    if (size == 0) {
        size = static_cast<size_t>(mapping.output_bytes);
    }
    return size;
}
```

> 如果暂时不想改动 `PdoEntry`，也可以先按字节对齐累加，将 bit_length 映射为最近的字节 size。但这会丢失位精度，可能不适用于所有从站。

### 5.5 名称生成

SDO 读取得到的是 index/subindex/bitlen，没有语义名称。为了兼容现有 `DeviceProfile`（如 `ServoProfile` 按 `"target_position"` 查找），需要建立 index→name 的映射表。

建议新增一个**对象字典名称表**：

```cpp
// 在 slave_node 或独立文件中
const std::unordered_map<uint32_t, std::string> kCoEObjectNames = {
    {0x60400000, "controlword"},
    {0x60410000, "statusword"},
    {0x607A0000, "target_position"},
    {0x60640000, "position_actual_value"},
    {0x60FF0000, "target_velocity"},
    {0x606C0000, "velocity_actual_value"},
    {0x60710000, "target_torque"},
    {0x60770000, "torque_actual_value"},
    // ...
};
```

生成 name 的函数：

```cpp
std::string GetPdoEntryName(uint16_t index, uint8_t subindex)
{
    uint32_t key = (static_cast<uint32_t>(index) << 16) | (subindex << 8);
    auto it = kCoEObjectNames.find(key);
    if (it != kCoEObjectNames.end()) return it->second;

    std::ostringstream oss;
    oss << "obj_" << std::hex << std::setfill('0') << std::setw(4) << index
        << "_" << std::setw(2) << static_cast<int>(subindex);
    return oss.str();
}
```

### 5.6 ConfigurePdoMapping 伪代码

```cpp
bool SlaveNode::ConfigurePdoMapping()
{
    // 1. 前置检查：只有 CoE 从站才读取
    if (!(info_.mbx_proto & ECT_MBXPROT_COE)) {
        LOG_INFO << "Slave " << info_.name << " does not support CoE, keep factory default";
        return true;  // 不是错误
    }

    // 2. 确认从站已在 PREOP
    uint16_t state = GetCurrentState();
    if (state != EC_STATE_PRE_OP) {
        LOG_WARN << "Slave " << info_.name << " not in PREOP (state=0x" << std::hex << state
                 << "), trying to switch...";
        if (!RequestState(EC_STATE_PRE_OP)) {
            LOG_ERROR << "Failed to bring slave " << info_.name << " to PREOP";
            return false;
        }
    }

    // 3. 读取 RxPDO
    PdoMapping new_mapping;
    if (!ReadPdoAssignment(/*tx=*/false, new_mapping.outputs, new_mapping.output_bytes)) {
        return false;
    }

    // 4. 读取 TxPDO
    if (!ReadPdoAssignment(/*tx=*/true, new_mapping.inputs, new_mapping.input_bytes)) {
        return false;
    }

    // 5. 验证字节数与 SlaveInfo 是否一致（可选但推荐）
    if (new_mapping.output_bytes != static_cast<int>(info_.output_bytes) ||
        new_mapping.input_bytes != static_cast<int>(info_.input_bytes)) {
        LOG_WARN << "Slave " << info_.name << " SDO mapping bytes (out=" << new_mapping.output_bytes
                 << ", in=" << new_mapping.input_bytes << ") differ from SlaveInfo (out="
                 << info_.output_bytes << ", in=" << info_.input_bytes << ")";
    }

    // 6. 应用到本节点
    SetPdoMapping(new_mapping);
    LOG_INFO << "Slave " << info_.name << " PDO mapping configured: out="
             << new_mapping.output_bytes << "B, in=" << new_mapping.input_bytes << "B";
    return true;
}
```

### 5.7 ReadPdoAssignment 伪代码

```cpp
bool SlaveNode::ReadPdoAssignment(bool tx, std::vector<PdoEntry> &entries, int &total_bytes)
{
    uint16_t assign_index = tx ? 0x1C13 : 0x1C12;
    entries.clear();
    total_bytes = 0;

    // 读取 PDO 数量
    uint8_t n_pdo = 0;
    if (!SdoRead(assign_index, 0, &n_pdo, sizeof(n_pdo))) {
        return false;
    }

    int bit_offset = 0;
    for (uint8_t i = 1; i <= n_pdo; ++i) {
        uint16_t pdo_index = 0;
        if (!SdoRead(assign_index, i, &pdo_index, sizeof(pdo_index))) {
            return false;
        }

        uint8_t n_entries = 0;
        if (!SdoRead(pdo_index, 0, &n_entries, sizeof(n_entries))) {
            return false;
        }

        for (uint8_t j = 1; j <= n_entries; ++j) {
            uint32_t mapping_value = 0;
            if (!SdoRead(pdo_index, j, &mapping_value, sizeof(mapping_value))) {
                return false;
            }

            uint16_t index = (mapping_value >> 16) & 0xFFFF;
            uint8_t subindex = (mapping_value >> 8) & 0xFF;
            uint8_t bitlen = mapping_value & 0xFF;

            PdoEntry entry;
            entry.name = GetPdoEntryName(index, subindex);
            entry.index = index;
            entry.subindex = subindex;
            entry.bit_offset = bit_offset;
            entry.bit_length = bitlen;
            entries.push_back(entry);

            bit_offset += bitlen;
        }
    }

    total_bytes = (bit_offset + 7) / 8;
    return true;
}
```

## 6. 错误处理与回滚策略

### 6.1 状态转换失败

| 失败场景 | 处理策略 |
|---|---|
| INIT → PREOP 失败 | 重试 3 次；若仍失败，尝试切回 INIT，然后 `Shutdown()`，初始化失败 |
| PREOP → SAFEOP 失败（ConfigureProcessData） | 切回 PREOP；记录错误；初始化失败 |
| SAFEOP → OP 失败 | 切回 SAFEOP；`StartOperation()` 返回 false |
| 运行中从站掉回 SAFEOP | `CheckSlaveStates()` 发现后，标记 `operational_=false`，通知上层 |
| Stop() 时 OP → SAFEOP/INIT 失败 | 记录错误，但仍继续清理资源，避免资源泄漏 |

### 6.2 SDO 读取失败

- 单个从站 `ConfigurePdoMapping()` 失败 → 回退到 `SlaveConfigFactory` 默认映射，继续初始化。
- 这样可以在验证阶段打印对比信息，也保证系统可用。
- 当日志中不再出现 fallback 时，说明方案 B 已稳定。

### 6.3 部分从站失败

多从站场景下，状态请求是广播的。如果 `CheckAllSlavesInState()` 发现部分从站未到达目标状态：

1. 打印每个失败从站的实际状态和 AL status code。
2. 如果失败从站是 Unknown 类型且非关键，可配置为"忽略"。
3. 否则整体失败，回滚到上一稳定状态。

### 6.4 新增 Shutdown 方法

```cpp
void EcatController::Shutdown()
{
    if (state_ == ControllerState::kOperational) {
        master_.RequestSafeOpState();
    }
    if (state_ >= ControllerState::kPreOp) {
        master_.RequestInitState();
    }

    node_manager_.Clear();
    master_.Close();

    initialized_ = false;
    operational_ = false;
    state_ = ControllerState::kUninitialized;
}
```

`EcatController::Stop()` 改为调用 `Shutdown()`，保证清理路径唯一。

### 7.3 PdoEntry 结构变更

```cpp
struct PdoEntry {
    std::string name;        // 语义名称
    uint16_t index = 0;      // CoE 对象索引
    uint8_t subindex = 0;    // CoE 子索引
    int bit_offset = 0;      // 位偏移
    int bit_length = 0;      // 位长度

    // 便捷函数
    int byte_offset() const { return bit_offset / 8; }
    int bit_shift() const { return bit_offset % 8; }
    int byte_size() const { return (bit_length + 7) / 8; }
};
```

### 7.4 DeviceProfile 访问适配

`ServoProfile::EncodePdoOutput` / `DecodePdoInput` 需要按位读取/写入：

```cpp
// 读取示例
int32_t ReadInt32LE(const std::vector<uint8_t> &buf, const PdoEntry &entry)
{
    int byte_off = entry.byte_offset();
    int shift = entry.bit_shift();
    int64_t value = 0;
    for (int i = 0; i < entry.byte_size() + (shift ? 1 : 0); ++i) {
        value |= static_cast<int64_t>(buf[byte_off + i]) << (i * 8);
    }
    value >>= shift;
    // sign extend if needed
    return static_cast<int32_t>(value);
}
```

> 如果确认项目当前所有 PDO 都是字节对齐的，可以先简化：仅支持 `bit_offset % 8 == 0` 的情况，否则报错。这样实现成本低，且不影响现有 Renesas RA8 伺服。

## 9. 风险与注意事项

1. **SOEM `ecx_config_map_group()` 内部也会读 PDO 映射**
   - SOEM 的 `ecx_config_map_group()` 内部会调用 `ecx_map_coe_soe()`，该函数会先 `statecheck(PREOP)`，然后调用 `ecx_readPDOmap()`。
   - 但我们已经在调用 `ecx_config_map_group()` 之前显式进入 PREOP 并读取了映射，所以不会冲突。
   - 注意：`ecx_config_map_group()` 会覆盖 `ctx_.slavelist[slave].Obits/Ibits/Obytes/Ibytes`，这些值应该与我们读取的一致。

2. **状态切换不要与周期线程并发**
   - `RunOneCycle()` 和 `RequestState()` 都访问 SOEM 上下文，当前已有 `soem_mutex_` 保护。
   - 状态切换期间应确保周期线程尚未启动或未在运行。

3. **非 CoE 从站**
   - DigitalIo/AnalogIo 可能没有 CoE 邮箱，应保留工厂默认映射。
   - `ConfigurePdoMapping()` 中先做 `mbx_proto & ECT_MBXPROT_COE` 检查。

4. **位精度 vs 字节精度**
   - 如果暂时只支持字节对齐，需在文档和代码中明确。
   - 长期应完整支持位精度，以兼容更多 IO 从站。

5. **SDO 读取超时**
   - 建议单次 SDO 超时 10~50ms，整个映射读取过程可能耗时数百毫秒。
   - 初始化阶段可以接受，不影响运行期实时性。

---

## 10. 总结

当前项目存在两个核心问题：

1. **PDO 映射硬编码**，限制了对不同从站的适应能力。
2. **主站缺少状态机**，导致初始化流程不清晰、失败处理薄弱。

本方案通过以下方式解决：

- 在 `EcatController` 中建立 **INIT → PREOP → SAFEOP → OP** 的显式生命周期状态机。
- 在 `ScanSlaves()` 之后显式进入 **PREOP**，为 SDO 读取创造条件。
- 在 `SlaveNode::ConfigurePdoMapping()` 中通过 CoE SDO 读取 `0x1C12/0x1C13/0x1600+/0x1A00+`，自动构建 `PdoMapping`。
- 失败时回退到硬编码默认映射，保证系统可用。
- 接入原本已存在但未被调用的 `ConfigureProcessData()` 和 `ConfigureDc()`。
- 增加状态转换重试、ACK 错误处理、实际状态确认等错误处理机制。

下一步建议从**阶段 1（状态机框架）**开始实施，先确保主站生命周期管理正确，再逐步加入 SDO 读取功能。

## 11. 方案评审与修订建议

基于当前项目代码和 SOEM `ec_config.c` 的实现，本文档前面的总体方向是正确的，但实施优先级需要进一步调整：**短期不应优先重复实现 SOEM 已经具备的总线层 PDO/FMMU/IOmap 配置能力，而应先接入 SOEM 的 `ecx_config_map_group()`，再在业务层解决 PDO 条目语义映射问题。**

### 11.1 当前方案中正确的判断

#### 11.1.1 PDO 业务映射确实是硬编码

当前代码中：

- `SlaveConfigFactory::CreateConfig()` 只使用了 `info.name`。
- `MakeServoConfig()` 固定使用 10 字节 output / 10 字节 input。
- `SlaveNode::ConfigurePdoMapping()` 当前只是打印日志并返回 `true`。
- `SlaveNodeManager::Initialize()` 当前没有调用 `ConfigurePdoMapping()`。
- `EcatController::Initialize()` 当前没有调用 `ConfigureProcessData()` / `ConfigureDc()`。

因此，“当前项目业务层 PDO 映射完全硬编码”这个结论是准确的。

#### 11.1.2 主站初始化流程确实不完整

当前 `EcatController::Initialize()` 的主流程为：

```cpp
master_.Initialize(config);
auto slave_infos = master_.ScanSlaves();
node_manager_.Initialize(master_, slave_infos);
initialized_ = true;
```

但没有调用：

```cpp
master_.ConfigureProcessData();
master_.ConfigureDc();
```

这会导致 SOEM 的 IOmap、FMMU、SM、`ctx_.slavelist[slave].outputs` / `inputs` 指针等没有被完整配置。后续 `EcMaster::WriteOutput()` / `ReadInput()` 依赖这些指针，如果没有 `ecx_config_map_group()`，周期 PDO 读写就没有可靠基础。

因此，**接入 `ConfigureProcessData()` 是当前最优先要做的修复。**

### 11.2 需要修正的关键点

#### 11.2.1 不应把“自己 SDO 读取 PDO 映射”放在第一优先级

前文方案 B 提到：

> 在 `SlaveNode::ConfigurePdoMapping()` 中通过 SDO 读取 `0x1C12/0x1C13/0x1600+/0x1A00+`，自动构建 `PdoMapping`。

这个方向长期是有价值的，但短期优先级应下调。原因是 SOEM 的 `ecx_config_map_group()` 内部已经包含 PDO 映射读取和 IOmap 配置逻辑。

在 `third_party/SOEM/src/ec_config.c` 中可以看到相关流程：

- `ecx_config_map_group()`
- `ecx_map_coe_soe()`
- `ecx_readPDOmap()`
- `ecx_readPDOmapCA()`

也就是说，SOEM 在配置过程数据时会读取 CoE PDO map，并计算：

- 每个从站的 `Obits` / `Ibits`
- 每个从站的 `Obytes` / `Ibytes`
- SyncManager / FMMU 配置
- IOmap 中的 inputs / outputs 指针
- group 的 expected WKC

因此，短期更合理的路径是：

```text
先接入 SOEM 已有的 ecx_config_map_group()
再从 SOEM 结果中同步/校验字节数和 IOmap 信息
最后再考虑是否需要自己读取条目级业务语义映射
```

#### 11.2.2 需要区分“两类 PDO 映射”

当前讨论中容易把两层概念混在一起：

| 层级 | 负责方 | 关注内容 | 当前建议 |
|---|---|---|---|
| **EtherCAT 总线层 PDO 映射** | SOEM | SM/FMMU、IOmap、inputs/outputs 指针、Obytes/Ibytes、WKC | 使用 `ecx_config_map_group()` |
| **业务语义层 PDO 映射** | 本项目 | `target_position` / `statusword` 等业务变量在 PDO buffer 中的 offset/size | 当前硬编码，后续可通过 CoE SDO 条目读取改进 |

一句话：

> **SOEM 负责“PDO 数据怎么进入 IOmap”，本项目负责“IOmap 里的字节代表哪个业务变量”。**

当前真正缺的是第二层：业务语义层的 `PdoMapping`。因此后续如果读取 `0x1C12/0x1C13/0x1600+/0x1A00+`，其目的应是构建业务语义映射，而不是替代 SOEM 的 IOmap 配置流程。

#### 11.2.3 “ScanSlaves 后一定处于 INIT，邮箱不可用”需要改成更谨慎的表述

前文中写到：

> `ScanSlaves()` 之后从站处于 INIT，邮箱不可用，无法执行 SDO。

这个判断不够严谨。SOEM 的 `ecx_config_init()` 内部会执行从站发现和基础配置，实际状态应以 `ctx_.slavelist[i].state` 为准。不同 SOEM 流程和从站表现下，扫描后状态不应简单假定为 INIT。

更稳妥的表述应为：

> `ScanSlaves()` 后没有显式确认所有从站已到 PREOP，因此不能假设 SDO 一定可用。若要执行 SDO，应显式请求并确认 PREOP。

#### 11.2.4 `PdoEntry` 不建议短期直接替换为纯 bit_offset/bit_length 结构

前文建议将：

```cpp
struct PdoEntry {
    std::string name;
    int offset;
    int size;
};
```

直接替换为：

```cpp
struct PdoEntry {
    std::string name;
    uint16_t index;
    uint8_t subindex;
    int bit_offset;
    int bit_length;
};
```

长期看这个设计更通用，但短期会牵连较大，需要同时修改：

- `slave_config_factory.cpp`
- `slave_node.cpp`
- `device_profile.cpp`
- `WriteRaw()` / `ReadRaw()`
- 所有现有硬编码映射

更稳妥的短期兼容方案是保留现有字段，并增量增加 CoE 元信息：

```cpp
struct PdoEntry {
    std::string name;

    // 兼容现有代码：字节偏移和字节大小
    int offset = 0;
    int size = 0;

    // 新增：来自 CoE PDO Mapping 的对象字典信息
    uint16_t index = 0;
    uint8_t subindex = 0;
    int bit_offset = 0;
    int bit_length = 0;
};
```

并且第一阶段只支持字节对齐条目：

```cpp
if (bit_offset % 8 != 0 || bit_length % 8 != 0) {
    // 暂不支持非字节对齐 PDO，回退到工厂默认映射
    return false;
}

entry.offset = bit_offset / 8;
entry.size = bit_length / 8;
```

这样既保留了后续支持位精度的扩展空间，又不会破坏当前 `DeviceProfile` 的读写逻辑。

### 11.3 更推荐的实施顺序

#### 阶段 1：先修主流程，不碰 PDO 条目解析

优先级最高的是让 SOEM 正常完成总线层过程数据配置。

推荐初始化流程调整为：

```text
master.Initialize()
master.ScanSlaves()
master.ConfigureProcessData()
master.ConfigureDc()
刷新 slave_infos / 重新读取 GetSlaveInfo()
node_manager.Initialize()
```

其中 `ConfigureProcessData()` 会调用 SOEM 的 `ecx_config_map_group()`，完成 IOmap、FMMU、SM、inputs/outputs 指针、WKC 等配置。

注意：`ScanSlaves()` 返回的 `SlaveInfo` 是 `ecx_config_init()` 后立即读取的。如果 `ConfigureProcessData()` 后 `ctx_.slavelist[i].Obytes/Ibytes` 被更新，应新增一个刷新步骤，例如：

```cpp
std::vector<SlaveInfo> EcMaster::GetAllSlaveInfos() const;
```

或者在 `ConfigureProcessData()` 后重新逐个调用 `GetSlaveInfo(i)`。

#### 阶段 2：让 `SlaveConfigFactory` 至少参考 SOEM 的字节数

在不解析 PDO 条目的情况下，也应利用 SOEM 结果做基本校验：

- 对 Servo：硬编码映射大小与 `info.output_bytes/input_bytes` 不一致时打印 `WARN`。
- 对 DigitalIo / AnalogIo / Unknown：可优先使用 `info.output_bytes/input_bytes` 作为 buffer 大小。
- 如果 SOEM 字节数为 0，则保留工厂默认值。

这样可以快速发现“业务硬编码映射”和“从站实际 PDO 大小”不一致的问题。

#### 阶段 3：实现条目级业务语义映射读取，但不替代 SOEM IOmap

在 `SlaveNode::ConfigurePdoMapping()` 或独立 `PdoMappingReader` 中读取：

- `0x1C12`：RxPDO Assignment
- `0x1C13`：TxPDO Assignment
- `0x1600+`：RxPDO Mapping
- `0x1A00+`：TxPDO Mapping

但这一步的目标是构建本项目业务层结构：

```cpp
PdoMapping {
    outputs: name/index/subindex/offset/size
    inputs:  name/index/subindex/offset/size
}
```

也就是服务于：

```cpp
DeviceProfile::EncodePdoOutput();
DeviceProfile::DecodePdoInput();
```

而不是重新配置 EtherCAT 的 SM/FMMU/IOmap。

#### 阶段 4：再逐步增强状态机和错误恢复

完整的 `PreOp / PdoConfigured / SafeOp / DcConfigured` 状态机方向正确，但短期可以先轻量化：

```cpp
enum class ControllerState {
    Uninitialized,
    Initialized,
    SlavesScanned,
    ProcessDataConfigured,
    Operational,
    Error
};
```

等主流程跑通后，再细化为更完整的 EtherCAT 生命周期状态。

### 11.4 修订后的核心结论

当前方案建议修订为：

> 短期不要重复实现 SOEM 已有的 PDO/FMMU/IOmap 配置能力。首先应接入 `EcMaster::ConfigureProcessData()`，让 SOEM 完成总线层 PDO 映射和 IOmap 建立；然后用 SOEM 得到的 `Obytes/Ibytes` 校验或修正 `SlaveConfigFactory` 的 buffer 大小；最后，如需摆脱硬编码业务字段，再通过 CoE SDO 读取对象条目，构建业务语义层 `PdoMapping`。

最终分工应明确为：

```text
SOEM：负责总线层 PDO 映射、SM/FMMU、IOmap、WKC。
本项目：负责业务语义层 PDO 映射，即对象字典条目到业务变量名的对应关系。
```

这是当前项目中更清晰、风险更低、实现成本更可控的演进路线。
