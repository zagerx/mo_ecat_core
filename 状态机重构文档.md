# 状态机重构文档

## 1. 前因后果

### 1.1 当前状态机是如何演进到今天的

`EcatController` 最初被设计为一个**函数调用式状态机**：

```cpp
controller.Initialize(config);   // Uninitialized -> Maintenance
controller.StartOperation();     // Maintenance -> Operational
controller.Stop();               // Operational -> Uninitialized
```

为了支持"主站初始化后等待外部命令，再逐步推进"的交互模式，后续把 `Initialize()` 拆分为三个更细粒度的接口：

```cpp
controller.InitializeAdapter(config);  // Uninitialized -> AdapterReady
controller.Scan();                     // AdapterReady -> Scanned
controller.EnterMaintenance();         // Scanned -> Maintenance
```

这三个接口底层仍然走 `TransitionTo(ControllerState target)`，由 `TransitionTo()` 统一处理：

- 前置状态检查
- 转换表合法性检查
- 自动顺序前进或后退
- 失败回滚或进入 `kError`
- 日志记录

同时，为了把维护操作（SDO 诊断、参数写入、状态巡检）从主状态机中解耦，引入了 `EcatActivity` 框架。Activity 在主站某个状态下执行，但不改变 `ControllerState`。

### 1.2 为什么现在需要再次重构

当前代码已经能工作，但暴露了两个核心问题：

**问题一：状态机不够直观**

当前状态迁移逻辑分散在 `DoInit()` / `DoScan()` / `DoPreOp()` / `DoPdoConfigure()` 等私有方法中，外部通过 `Scan()` / `EnterPreOp()` 等函数触发。对于希望"一眼看到主站当前在做什么、接下来该做什么"的开发者来说，这种设计不够直观。

**问题二：缺少周期性任务调度**

当前 `EcatController` 提供了 `RunOneCycle()` 和 `CheckSlaveStates()`，但程序里没有一个地方**周期性地调用它们**。比如：

- `kMaintenance` 状态下，应该周期性 `CheckSlaveStates()`
- `kOperational` 状态下，应该每个周期 `RunOneCycle()` + `CheckSlaveStates()`

这些周期性任务没有归属，导致 PDO 阶段无法自然跑起来，也让 Maintenance 阶段的掉线检测只能依赖手动命令触发。

### 1.3 用户的诉求

用户希望主状态机是经典的 **switch-case 主循环** 结构：

```cpp
while (!shutdown) {
    switch (state) {
    case kAdapterReady:  /* ... */ break;
    case kScanned:       /* ... */ break;
    case kMaintenance:   /* ... */ break;
    case kOperational:   /* ... */ break;
    }
}
```

诉求包含两点：

1. **直观性**：所有状态下做什么，集中在一个 switch-case 里。
2. **周期性任务**：主线程持续运行，在不同状态下执行各自的周期任务。

## 2. 当前问题详细分析

### 2.1 问题一：状态机不直观

当前 `EcatController` 的状态迁移是"命令触发式"的：

```cpp
// EcatApplication.cpp
void OnScan() {
    if (controller_.GetState() != kAdapterReady) return;
    controller_.Scan();  // 内部 TransitionTo(kScanned)
}
```

从 `OnScan()` 看，只能知道"scan 命令会触发 Scan()"，但看不到：

- `kAdapterReady` 状态下除了等 scan，还能做什么？
- `kScanned` 状态下必须做什么才能推进？
- 所有状态下共同的任务（如日志、监控）在哪里统一处理？

状态处理逻辑被封装在 `TransitionTo()` / `DoXxx()` 的调用链中，阅读代码时需要不断在多个函数间跳转。

### 2.2 问题二：周期性任务没有主循环

当前 `main.cpp` 的逻辑：

```cpp
while (!shutdown) {
    std::string command;
    if (queue->TimedPop(command, 100)) {
        app->HandleCommand(command);  // 处理命令
    }
}
```

这个循环只处理命令，没有根据当前 `ControllerState` 执行任何周期性任务。`CheckSlaveStates()` 只在 `OnInspect()` 等 Activity 里被调用，不是周期性的。

后果：

- `kMaintenance` 阶段从站掉线无法及时发现。
- `kOperational` 阶段不会自动跑 PDO 周期，必须额外创建周期线程。
- 后续想增加"心跳检查""状态统计"等周期性工作没有统一入口。

### 2.3 问题三：Activity 与状态机循环的关系不清晰

当前 Activity 由 `EcatApplication::HandleCommand()` 直接调用 `controller_.ExecuteActivity()` 执行。Activity 执行期间，主循环阻塞在 `HandleCommand()` 内部。这带来两个问题：

1. Activity 执行期间无法响应退出信号（虽然当前 Activity 很快，可以接受）。
2. Activity 执行期间无法执行周期性任务（如 CheckSlaveStates）。

如果将来 Activity 变长（如 FOE 升级），这种设计会不够用。

---

## 3. 解决方案

### 3.1 核心思想

**不要推翻 `TransitionTo()` / `DoXxx()`，而是在应用层增加一个显式的 switch-case 主循环。**

把状态机拆成两个层次：

| 层次 | 负责 | 当前对应代码 |
|---|---|---|
| **状态迁移层** | 状态转换的原子步骤、失败回滚、合法性检查 | `EcatController::TransitionTo()` / `DoXxx()` |
| **状态运行层** | 当前状态下做什么、周期性任务、命令分发 | 新增 `EcatApplication::Run()` switch-case 主循环 |

这样：

- `TransitionTo()` 继续保证状态迁移的原子性和安全性。
- switch-case 主循环解决直观性和周期性任务调度问题。
- 两层职责清晰，互不替代。

### 3.2 目标架构

```text
main.cpp
    │
    ▼
EcatApplication::Run()          // switch-case 主循环
    │
    ├── 读取命令（stdin，100ms 超时）
    │
    ├── switch (controller_.GetState())
    │       ├── kAdapterReady:  处理 scan / stop 命令
    │       ├── kScanned:       处理 config / stop 命令
    │       ├── kMaintenance:   周期性 CheckSlaveStates() + 处理 diagnose/param/inspect/start/stop 命令
    │       ├── kOperational:   周期性 RunOneCycle() + CheckSlaveStates()
    │       ├── kError:         只能处理 stop 命令
    │       └── default:        无操作
    │
    └── controller_.TransitionTo() / ExecuteActivity() 执行实际迁移或 Activity
```

### 3.3 关键设计决策

#### 决策 1：保留 `TransitionTo()`

不改造 `EcatController` 内部为 switch-case，原因：

- `TransitionTo()` 已经稳定处理了状态迁移的原子性、回滚、查表检查。
- 改造成 switch-case 会把这些逻辑打散，容易引入 bug。
- `DoXxx()` 每个步骤都是一次性的（如 scan 是一次性操作），不适合放在循环里反复执行。

#### 决策 2：switch-case 只表达"当前状态下做什么"

switch-case 里不写具体迁移动作，而是：

- 调用 `controller_.Scan()` / `EnterMaintenance()` / `StartOperation()` / `Stop()` 触发迁移。
- 调用 `controller_.CheckSlaveStates()` / `RunOneCycle()` 执行周期任务。
- 调用 `controller_.ExecuteActivity()` 执行 Activity。

这样 switch-case 保持简洁，复杂逻辑仍在 `EcatController` 中。

#### 决策 3：命令驱动 + 周期任务并存

主循环每轮做两件事：

1. **尝试读取命令**（带超时，不阻塞退出）。
2. **根据当前状态执行周期任务**（如果该状态有周期任务）。

例如 `kMaintenance` 状态下，即使没有新命令，每 100ms 也会执行一次 `CheckSlaveStates()`。

#### 决策 4：状态只表示稳定工况

`kPdoConfigured` / `kSafeOp` / `kDcConfigured` 等过渡步骤不暴露为 `ControllerState`，而是作为 `StartOperation()` 的内部子步骤。这样：

- 状态数量从 9 个减少到 6 个；
- 用户视角更清晰；
- switch-case 主循环更简洁。

## 4. 具体实现方案

### 4.1 新增 `EcatApplication::Run()` 主循环

```cpp
void EcatApplication::Run()
{
    LOG_INFO << "Main state machine started. Type 'help' for commands, 'exit' to quit.";

    while (!shutdown_requested_) {
        // 1. 读取命令（100ms 超时，不阻塞退出）
        std::string command;
        const bool has_command = ReadCommand(command, 100);

        if (has_command && (command == "exit" || command == "quit")) {
            RequestShutdown();
            break;
        }

        // 2. 根据当前 ControllerState 执行状态机逻辑
        switch (controller_.GetState()) {
        case ControllerState::kAdapterReady:
            HandleAdapterReadyState(has_command ? &command : nullptr);
            break;

        case ControllerState::kScanned:
            HandleScannedState(has_command ? &command : nullptr);
            break;

        case ControllerState::kMaintenance:
            HandleMaintenanceState(has_command ? &command : nullptr);
            break;

        case ControllerState::kOperational:
            HandleOperationalState(has_command ? &command : nullptr);
            break;

        case ControllerState::kError:
            HandleErrorState(has_command ? &command : nullptr);
            break;

        default:
            break;
        }
    }

    LOG_INFO << "Main state machine stopped.";
}
```

### 4.2 各状态处理函数

#### `kAdapterReady`

```cpp
void EcatApplication::HandleAdapterReadyState(const std::string *command)
{
    if (command == nullptr) {
        return;
    }

    if (*command == "scan") {
        controller_.Scan();
    } else if (*command == "stop") {
        controller_.Stop();
    } else if (*command == "help") {
        OnHelp();
    } else {
        LOG_ERROR << "Command '" << *command << "' not allowed in AdapterReady state";
    }
}
```

#### `kScanned`

```cpp
void EcatApplication::HandleScannedState(const std::string *command)
{
    if (command == nullptr) {
        return;
    }

    if (*command == "config") {
        controller_.EnterMaintenance();
    } else if (*command == "stop") {
        controller_.Stop();
    } else if (*command == "help") {
        OnHelp();
    } else {
        LOG_ERROR << "Command '" << *command << "' not allowed in Scanned state";
    }
}
```

#### `kMaintenance`

```cpp
void EcatApplication::HandleMaintenanceState(const std::string *command)
{
    // 周期性任务：检查从站状态
    controller_.CheckSlaveStates();

    if (command == nullptr) {
        return;
    }

    if (*command == "diagnose") {
        OnDiagnose();
    } else if (command->rfind("param", 0) == 0) {
        // 解析 param <index> <subindex> <value>
        OnParam(args);
    } else if (*command == "inspect") {
        OnInspect();
    } else if (*command == "start") {
        controller_.StartOperation();
    } else if (*command == "stop") {
        controller_.Stop();
    } else if (*command == "help") {
        OnHelp();
    } else {
        LOG_ERROR << "Unknown command: " << *command;
    }
}
```

#### `kOperational`

```cpp
void EcatApplication::HandleOperationalState(const std::string *command)
{
    // 周期性 PDO 周期 + 状态检查
    controller_.RunOneCycle();
    controller_.CheckSlaveStates();

    if (command != nullptr && *command == "stop") {
        controller_.Stop();
    }
}
```

> 注意：`kOperational` 状态下以周期任务为主，命令处理可以简化为只处理 `stop`。如果需要在线修改参数，后续可扩展。

#### `kError`

```cpp
void EcatApplication::HandleErrorState(const std::string *command)
{
    if (command != nullptr && *command == "stop") {
        controller_.Stop();
    } else if (command != nullptr) {
        LOG_ERROR << "In Error state, only 'stop' is allowed";
    }
}
```

### 4.3 命令读取接口

当前 `main.cpp` 从 `stdin` 读取。可以把读取逻辑封装进 `EcatApplication`：

```cpp
class EcatApplication {
public:
    bool Initialize(const EcMasterConfig &config);
    void Run();          // 主循环入口
    void RequestShutdown();

private:
    bool ReadCommand(std::string &command, int timeout_ms);
    // ...

    EcatController controller_;
    std::atomic<bool> shutdown_requested_{false};
};
```

`ReadCommand()` 使用 `poll()` 实现带超时的 stdin 读取：

```cpp
bool EcatApplication::ReadCommand(std::string &command, int timeout_ms)
{
    struct pollfd pfd {};
    pfd.fd = STDIN_FILENO;
    pfd.events = POLLIN;

    int ret = poll(&pfd, 1, timeout_ms);
    if (ret <= 0) {
        return false;
    }

    if (!std::getline(std::cin, command)) {
        RequestShutdown();  // EOF / Ctrl+D
        return false;
    }

    return true;
}
```

### 4.4 main.cpp 改造

```cpp
namespace {
mo_ecat::EcatApplication *g_app = nullptr;

void OnSignal(int) {
    if (g_app != nullptr) {
        g_app->RequestShutdown();
    }
}

void RegisterShutdownSignals() {
    std::signal(SIGINT, OnSignal);
    std::signal(SIGTERM, OnSignal);
}
} // namespace

int main(int argc, char *argv[])
{
    // ... 初始化 config

    auto app = std::make_unique<mo_ecat::EcatApplication>();
    if (!app->Initialize(config)) {
        return 1;
    }

    g_app = app.get();
    RegisterShutdownSignals();

    app->Run();

    app->Shutdown();
    g_app = nullptr;
    return 0;
}
```

## 6. 预期收益

1. **状态机更直观**：打开 `EcatApplication::Run()` 就能看懂主站整体运行流程。
2. **周期性任务有归属**：`CheckSlaveStates()` / `RunOneCycle()` 自然挂在对应状态下。
3. **命令合法性更清晰**：每个状态下允许什么命令，集中写在对应 `HandleXxxState()` 里。
4. **状态数量精简**：从 9 个状态减少到 6 个，降低心智负担。
5. **后续扩展更容易**：
   - 增加新状态只需新增一个 case。
   - 增加周期任务只需在对应 case 里加一行。
   - 替换命令通道（stdin → socket）只需改 `ReadCommand()`。
6. **不破坏现有状态迁移安全性**：`EcatController::TransitionTo()` 仍然负责原子迁移和错误回滚。

---

## 7. 风险与应对措施

| 风险 | 应对措施 |
|---|---|
| `HandleOperationalState()` 中 `RunOneCycle()` 阻塞导致命令无法响应 | 当前阶段 PDO 周期由主线程执行，命令响应有周期延迟；后续可拆分为独立周期线程 |
| `ReadCommand()` 带超时实现复杂 | 使用 `poll()` 实现 stdin 超时读取 |
| Activity 执行期间无法做周期任务 | 当前 Activity 同步执行且很快；后续如需要可扩展为 Activity 分片执行或独立线程 |
| 状态处理函数膨胀 | 随着命令增多，可进一步拆分为子函数；必要时引入命令表 + 回调注册机制 |

---

## 8. 与现有 Activity 框架的关系

本重构不改变 Activity 框架：

- `EcatActivity` 仍然封装维护操作。
- `ControllerState` 仍然由 `TransitionTo()` 统一管理。
- `EcatApplication` 只是在更高层增加一个 switch-case 主循环，决定何时调用 `controller_.ExecuteActivity()`。

Activity 仍然遵守：

- **不改变 `ControllerState`**
- **失败策略由 Activity 自己声明**
- **同步执行，同一时刻只有一个 Activity 在运行**

---

## 9. 结论

当前 `TransitionTo()` 式状态机在**状态迁移的原子性和安全性**上做得很好，但缺少**显式主循环**和**周期性任务调度**。本次重构不推翻底层状态迁移机制，而是在 `EcatApplication` 层引入 switch-case 主循环，把"当前状态下做什么"和"周期性任务"显性化，同时将 `ControllerState` 精简为 6 个稳定状态，隐藏 PDO/SafeOp/DC 过渡态。

这样既满足了用户对直观性的要求，又保留了现有状态机的可靠性，同时为后续 PDO 阶段奠定了清晰的调度入口。
