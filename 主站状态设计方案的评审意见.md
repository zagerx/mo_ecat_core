# 主站状态设计方案的评审意见

## 1. 评审范围

本评审基于以下文件和当前代码状态：

- `主站状态设计方案.md`
- `src/ec_controller/ec_controller.h`
- `src/ec_controller/ec_controller.cpp`
- `src/ec_master/ec_master.h`
- `src/slave_node/slave_node.h`

评审重点包括：

1. 文档对当前问题的分析是否准确。
2. “主站生命周期状态 + EcatActivity 独立维护活动”的设计是否合理。
3. 该方案与当前代码的衔接是否清晰。
4. 后续实现时需要补充或修正的关键点。

---

## 2. 总体结论

`主站状态设计方案.md` 的总体方向是正确的。

当前项目已经完成了较好的第一步：`EcatController` 中已经建立了线性主站生命周期状态机，包括：

- `ControllerState` 枚举；
- `TransitionTo(ControllerState target)` 统一状态转换入口；
- `DoInit()` / `DoScan()` / `DoPreOp()` / `DoPdoConfigure()` / `DoSafeOp()` / `DoDcConfigure()` / `DoOperational()` / `DoShutdown()` 拆分；
- `kAllowedTransitions` 状态转换表；
- `EnterErrorState()` 错误状态入口；
- 移除 `initialized_` / `operational_` 冗余标志，改为基于 `state_` 判断。

在这个基础上，文档提出的“**主站生命周期状态 + EcatActivity 横向维护活动**”是一个合理的扩展方向。它能够避免把 FOE 升级、SDO 诊断、单站复位、参数标定、热插拔等维护场景全部塞进 `ControllerState`，从而避免主状态机膨胀成复杂网络。

不过，文档中的 Activity 执行模型、并发模型、主站状态与单站状态的关系还需要进一步收敛。尤其是以下两点建议重点修正：

1. **Activity 不应改变主站生命周期状态，因此 `GetTargetState()` 设计意义不大，建议移除。**
2. **当前阶段不建议直接引入后台线程式 Activity 执行器，应优先采用同步执行模型，避免 SOEM 上下文并发访问风险。**

---

## 3. 对文档问题分析部分的评价

### 3.1 “主站状态与从站状态混为一谈”的判断基本准确

文档第 2.1 节指出：当前 `ControllerState` 描述的是整站生命周期，隐含“所有从站同进同退”的假设。

这个分析是准确的。

当前 `EcatController` 的状态转换确实以全站为单位，例如：

```cpp
TransitionTo(kPreOp)
TransitionTo(kOperational)
DoShutdown()
```

这些操作本质上描述的是“整站处于哪个主流程阶段”，而不是某个单独从站的实际 EtherCAT 状态。

但实际工程中，单站维护场景非常常见：

- 单个从站进入 BOOTSTRAP 做 FOE；
- 单个从站重启；
- 单个从站故障恢复；
- 部分从站参数标定；
- 某个从站临时退出 OP，而其他从站继续运行。

因此，主站状态机不应承担所有单站状态变化的表达职责。

### 3.2 “线性状态机无法表达维护活动”的判断准确

文档认为 FOE 升级、SDO 诊断、单站复位、参数标定等不是主生命周期中的“下一个状态”，而是横向活动。这个判断非常重要。

如果将这些维护行为加入 `ControllerState`，会出现类似：

```cpp
kFoEUpgrading
kSdoDiagnosing
kSlaveResetting
kCalibrating
kHotSwapping
```

这会带来三个问题：

1. 状态枚举快速膨胀；
2. `kAllowedTransitions` 转换表越来越复杂；
3. 正常运行流程与维护流程耦合。

因此，文档提出将维护操作抽象为 `EcatActivity` 是合理的。

### 3.3 PREOP 阶段掉线检测问题成立，但优先级应下调

文档第 2.4 节指出当前 `CheckSlaveStates()` 只在 `kOperational` 时检查 OP 掉线，PREOP 等阶段不检测。

这个现象存在。当前代码中：

```cpp
void EcatController::CheckSlaveStates()
{
    master_.CheckSlaveStates();

    if (state_ == ControllerState::kOperational) {
        if (!master_.CheckAllSlavesInState(EC_STATE_OPERATIONAL)) {
            EnterErrorState("Slave dropped out of OPERATIONAL");
        }
    }
}
```

不过，从当前项目阶段看，PREOP 阶段主要用于初始化验证和 SDO 通信，不存在 PDO 周期线程持续运行的强实时需求。因此 PREOP 掉线检测可以作为增强项，但不应优先于状态机稳定性、Activity 同步模型、单站状态控制等基础能力。

## 4. 对“主站生命周期状态 + EcatActivity”方案的评价

### 4.1 二维设计方向正确

文档提出将系统拆成两个维度：

1. **纵向：主站生命周期状态**
   - `kUninitialized`
   - `kInitDone`
   - `kScanned`
   - `kPreOp`
   - `kPdoConfigured`
   - `kSafeOp`
   - `kDcConfigured`
   - `kOperational`
   - `kError`

2. **横向：维护活动 `EcatActivity`**
   - FOE 升级；
   - SDO 诊断；
   - 参数标定；
   - 单站复位；
   - EEPROM 烧录；
   - 错误恢复；
   - 拓扑诊断。

这个模型比把所有维护状态塞进 `ControllerState` 更清晰。

建议保留该总体设计。

### 4.2 Activity 应被定义为“不改变 ControllerState 的横向操作”

文档目前的 `EcatActivity` 接口包含：

```cpp
virtual ControllerState GetTargetState() const = 0;
```

这个接口建议删除。

理由是：

- 主站生命周期状态只应由 `EcatController::TransitionTo()` 改变；
- Activity 内部可以改变单个从站的 EtherCAT 状态，但不应直接改变 `ControllerState`；
- 对 FOE 升级这类局部活动而言，主站状态通常始终保持 `kOperational`，不存在“Activity 结束后主站进入另一个生命周期状态”的需求；
- 如果允许 Activity 声明目标主站状态，容易模糊职责边界。

建议接口改为：

```cpp
class EcatActivity {
public:
    virtual ~EcatActivity() = default;

    virtual const char *GetName() const = 0;
    virtual bool CanStart(ControllerState state) const = 0;
    virtual bool Execute() = 0;
};
```

如需 Activity 失败后进入 `kError`，应由 `EcatController` 在执行结果返回后统一处理，而不是 Activity 私自修改 `state_`。

## 5. 对 Activity 执行器设计的建议

### 5.1 不建议当前阶段直接使用后台线程执行 Activity

文档第 3.4 节提出：

```cpp
std::unique_ptr<EcatActivity> current_activity_;
std::thread activity_thread_;
std::atomic<bool> activity_running_{false};
```

并由后台线程执行维护活动。

这个设计在长期看有价值，但当前阶段不建议直接采用。原因如下。

#### 5.1.1 SOEM 上下文并发访问风险

当前 `EcMaster` 中通过 `soem_mutex_` 保护 SOEM 上下文：

```cpp
mutable std::mutex soem_mutex_;
```

虽然单个 `EcMaster` 方法内部加锁，但如果 Activity 后台线程、PDO 周期线程、状态监控线程同时调用不同 SOEM 操作，仍然会带来复杂的时序问题，例如：

- Activity 正在把某个从站切到 BOOTSTRAP；
- 周期线程仍在 `RunOneCycle()`；
- 状态监控线程正在 `CheckAllSlavesInState()`；
- `Stop()` 同时触发 `TransitionTo(kUninitialized)`。

即便互斥锁能避免数据竞争，也不能保证业务时序正确。

#### 5.1.2 Stop 等待 Activity 存在阻塞风险

文档规定：

> Stop 时需等待当前活动结束，再执行 DoShutdown。

如果 Activity 卡在 FOE 写入、等待从站重启、等待链路恢复等操作中，`Stop()` 可能长时间阻塞，甚至无法退出。

#### 5.1.3 当前代码没有 Activity 取消机制

文档第 7.2 节提到急停应能打断普通 Activity，但当前设计中没有：

- cancellation token；
- timeout policy；
- activity priority；
- emergency interrupt path；
- Stop 强制中断策略。

因此，一开始就引入后台线程会放大复杂度。

### 5.2 建议第一阶段采用同步 Activity 执行模型

建议先实现同步接口：

```cpp
bool EcatController::ExecuteActivity(std::unique_ptr<EcatActivity> activity)
{
    if (!activity) {
        return false;
    }

    if (activity_running_) {
        LOG_ERROR << "Activity already running";
        return false;
    }

    if (!activity->CanStart(state_)) {
        LOG_ERROR << "Activity " << activity->GetName()
                  << " cannot start in state " << StateToString(state_);
        return false;
    }

    activity_running_ = true;
    LOG_INFO << "Activity started: " << activity->GetName();

    const bool ok = activity->Execute();

    LOG_INFO << "Activity finished: " << activity->GetName()
             << ", result=" << (ok ? "ok" : "failed");
    activity_running_ = false;

    return ok;
}
```

同步模型的优点：

- 语义简单；
- 不引入线程生命周期问题；
- 不需要立即设计取消/抢占；
- 更容易调试；
- 与当前 SOEM 封装模型更匹配。

如果后续确实需要异步执行，可以在同步模型稳定后再加：

```cpp
std::future<bool> StartActivityAsync(...);
```

而不是一开始就将 Activity 绑定到内部线程。

## 6. 对当前 EcatController 状态机实现的评审

### 6.1 当前状态机框架基本可用

当前代码已经完成了文档中第 8 节描述的大部分内容：

- `TransitionTo()` 已实现；
- `DoStepTo()` 已实现；
- 各状态的 `DoXxx()` 已拆分；
- `DoShutdown()` 已根据当前状态分级回滚；
- `IsInitialized()` / `IsOperational()` 已基于 `state_` 判断。

这是后续引入 Activity 的良好基础。

### 6.2 `TransitionTo()` 的前进路径没有检查转换表

当前 `TransitionTo()` 对“前进状态”采用按枚举值自动递增：

```cpp
if (static_cast<int>(target) > static_cast<int>(state_)) {
    while (state_ != target) {
        ControllerState next =
            static_cast<ControllerState>(static_cast<int>(state_) + 1);
        if (!DoStepTo(next)) {
            return false;
        }
    }
    return true;
}
```

这意味着从 `kUninitialized` 调用 `TransitionTo(kOperational)` 会自动走完整条链路：

```text
Uninitialized → InitDone → Scanned → PreOp → PdoConfigured → SafeOp → DcConfigured → Operational
```

这个行为对 `StartOperation()` 来说可能方便，但它绕过了 `kAllowedTransitions` 对前进路径的约束。

建议文档补充说明：

- `TransitionTo(target)` 是“目标状态驱动”的接口，允许自动按线性路径前进；
- `kAllowedTransitions` 当前主要用于后退或特殊转换；
- 如果希望严格约束每一步前进，也应对 `next` 调用转换表检查。

更稳妥的实现建议：前进时也调用 `IsTransitionAllowed(state_, next)`，避免未来插入非线性状态后出现隐患。

### 6.3 `StartOperation()` 当前前置条件过窄

当前实现：

```cpp
bool EcatController::StartOperation()
{
    if (state_ != ControllerState::kPreOp) {
        LOG_WARN << "StartOperation called in invalid state " << StateToString(state_);
        return false;
    }

    return TransitionTo(ControllerState::kOperational);
}
```

该实现要求必须从 `kPreOp` 调用 `StartOperation()`。但从状态机设计看，后续如果系统已经处于：

- `kPdoConfigured`
- `kSafeOp`
- `kDcConfigured`

理论上也可以继续进入 OP。

建议后续将 `StartOperation()` 改为允许从多个中间状态继续前进，或者明确它的语义是“从 PREOP 开始执行完整 PDO 启动流程”。

### 6.4 `DoDcConfigure()` 对 DC 失败的策略需要重新确认

当前实现：

```cpp
if (!master_.ConfigureDc()) {
    EnterErrorState("Failed to configure DC");
    return false;
}
```

但早期方案中曾提到 DC 可能是非必需能力。当前 `EcMasterConfig` 也包含：

```cpp
bool use_dc = true;
```

建议明确策略：

- 如果 `use_dc == true` 且 DC 配置失败，是否必须进入 Error？
- 如果部分从站不支持 DC，是否允许跳过？
- 是否需要区分“DC 禁用”和“DC 配置失败”？

文档中应补充这一点，否则后续 PDO 阶段容易出现策略不一致。

### 6.5 `CheckSlaveStates()` 与文档建议还未一致

文档建议 PREOP / SAFEOP / OP 都检测，而当前代码只检测 OP。

建议文档将该项标记为“后续增强”，不要暗示已经实现。同时建议先实现无线程版本：由外部或主循环定期调用，而不是立即在 `EcatController` 内部新增监控线程。

## 7. 对 SlaveNode 单站控制设计的建议

### 7.1 `SlaveNode::current_state_` 不应作为真实状态来源

当前 `SlaveNode` 中存在：

```cpp
uint16_t current_state_ = 0;
```

文档第 7.5 节已经指出需要实时跟踪实际状态，建议增加 `RefreshActualState()`。

这个建议是必要的。

推荐后续约定：

- `SlaveNode::GetCurrentState()` 应从 `EcMaster::ReadActualState(slave_id)` 获取真实状态；
- `current_state_` 如果保留，只能作为缓存，且必须明确失效策略；
- Activity 执行前后必须刷新从站真实状态。

### 7.2 单站 BOOTSTRAP 控制应落在 EcMaster 层

文档中提到：

```cpp
bool SlaveNode::RequestState(uint16_t state);
```

支持任意 EtherCAT 状态，包括 BOOTSTRAP。

这在接口层面合理，但实际底层应复用：

```cpp
EcMaster::RequestStateWithRetry(int slave, uint16_t state, int max_retries)
```

并建议在 `EcMaster` 层增加清晰封装：

```cpp
bool RequestBootstrapState(int slave);
```

这样可以统一 ACK ERROR、重试、状态确认和日志输出。

### 7.3 FOE 接口建议先放在 EcMaster，再由 SlaveNode 转发

文档建议：

```cpp
bool SlaveNode::FoEWrite(...);
```

从业务调用角度看合理。但底层实现应该优先放在 `EcMaster`：

```cpp
bool EcMaster::FoEWrite(uint16_t slave,
                        const std::string &filename,
                        const std::vector<uint8_t> &data,
                        int timeout_us);
```

然后 `SlaveNode::FoEWrite()` 只是转发：

```cpp
return master_.FoEWrite(info_.slave_id, filename, data, timeout_us);
```

这样更符合当前架构：

- `EcMaster` 负责 SOEM API；
- `SlaveNode` 负责单站语义封装；
- `Activity` 负责流程编排。

## 8. 对维护场景清单的建议

文档第 4 节列出了大量维护场景，覆盖面很完整，但建议分层落地，不要一次性实现过多抽象。

建议按复杂度分成三类：

### 8.1 第一阶段：只读/低风险 Activity

优先实现：

- `SdoDiagnosticsActivity`
- `FirmwareAuditActivity`
- `StateInspectionActivity`

特点：

- 不改变从站状态；
- 不影响 PDO；
- 不需要 BOOTSTRAP；
- 适合验证 Activity 框架。

### 8.2 第二阶段：单站状态切换 Activity

再实现：

- `SlaveResetActivity`
- `ErrorRecoveryActivity`
- `SdoParameterActivity`

特点：

- 需要单站 `RequestState()`；
- 可能影响目标从站 PDO；
- 需要状态刷新和失败恢复策略。

### 8.3 第三阶段：高风险 Activity

最后实现：

- `FoEUpgradeActivity`
- `EepromUpdateActivity`
- `HotSwapActivity`

特点：

- 涉及 BOOTSTRAP、EEPROM、固件写入；
- 失败后可能需要人工恢复；
- 需要完整超时、重试、恢复和审计日志。

## 9. 推荐修订文档中的关键设计点

建议对 `主站状态设计方案.md` 做以下修订。

### 9.1 修改 EcatActivity 接口

将：

```cpp
virtual ControllerState GetTargetState() const = 0;
```

删除。

并明确：

> Activity 不直接改变 `ControllerState`；主站状态只能由 `EcatController::TransitionTo()` 修改。

### 9.2 将后台线程执行改为“分阶段设计”

建议文档中改成：

第一阶段：同步执行

```cpp
bool ExecuteActivity(std::unique_ptr<EcatActivity> activity);
```

第二阶段：如确有需要，再增加异步执行

```cpp
bool StartActivityAsync(std::unique_ptr<EcatActivity> activity);
bool CancelActivity();
bool IsActivityRunning() const;
```

### 9.3 明确 Activity 与 PDO 周期的互斥策略

文档目前提到“升级 A 站时 B/C/D 站仍在 OP 跑 PDO”，这是理想状态，但工程上复杂度较高。

建议增加保守策略：

第一阶段 Activity 执行期间暂停 PDO 周期或禁止在 OP 下执行高风险 Activity。

后续再支持局部跳过目标从站 PDO 的高级模式。

### 9.4 明确 Activity 失败策略

建议为 Activity 增加失败策略枚举：

```cpp
enum class ActivityFailurePolicy {
    kKeepControllerState,
    kEnterError,
    kShutdown,
};
```

这样可以避免所有 Activity 失败都直接进入 `kError`。

例如：

- SDO 诊断失败：只记录日志，不进 Error；
- 单站恢复失败：可进 Error；
- FOE 升级失败：根据配置决定保持状态、进 Error 或停机。

### 9.5 明确 PREOP 监控线程不是当前必需项

建议将“内部启动监控线程”调整为后续增强。当前阶段可以先提供 `CheckSlaveStates()` 能力，由上层或测试程序主动调用。

## 10. 建议的落地顺序

结合当前代码状态，推荐按以下顺序推进：

1. **稳定当前主站状态机**
   - 修正 `TransitionTo()` 前进路径是否也应查转换表；
   - 明确 `StartOperation()` 的允许起点；
   - 明确 DC 失败策略。

2. **补强单站状态能力**
   - `SlaveNode::RefreshActualState()`；
   - `EcMaster::RequestBootstrapState(int slave)`；
   - `SlaveNode::RequestState()` 统一走 `RequestStateWithRetry()`。

3. **实现同步版 Activity 框架**
   - `EcatActivity` 基类；
   - `EcatController::ExecuteActivity()`；
   - `activity_running_` 互斥标志；
   - 不引入后台线程。

4. **实现第一个低风险 Activity**
   - 推荐 `SdoDiagnosticsActivity`；
   - 读取 `0x1000`、`0x1018:01`、`0x1018:02`、`0x1018:03`；
   - 验证 Activity 框架和日志链路。

5. **再实现单站恢复类 Activity**
   - `ErrorRecoveryActivity`；
   - `SlaveResetActivity`。

6. **最后实现 FOE 升级**
   - 增加 `EcMaster::FoEWrite()`；
   - 增加 BOOTSTRAP 状态切换；
   - 增加超时、失败恢复、审计日志。

7. **根据实际需求再考虑异步 Activity / 监控线程 / 优先级抢占**

---

## 11. 最终建议

`主站状态设计方案.md` 可以作为后续架构演进的主方向保留，但建议在进入实现前做一次修订：

1. 保留“主站生命周期状态 + EcatActivity 独立维护活动”的总体设计。
2. 删除 `EcatActivity::GetTargetState()`，明确 Activity 不改变主站状态。
3. 将 Activity 执行器从后台线程模型改为“先同步、后异步”的分阶段方案。
4. 补充 Activity 与 PDO 周期、Stop、错误恢复之间的互斥和失败策略。
5. 将 PREOP 监控线程标记为后续增强，而非当前阶段必须项。
6. 将 FOE 升级放到较后阶段，先用 SDO 诊断类 Activity 验证框架。

总体评价：

> 方案方向正确，问题分析准确，Activity 抽象值得采用；但当前文档中的异步执行器和状态回归设计偏激进，建议先以同步、低风险、单站可控的方式落地，再逐步扩展到 FOE、热插拔、异步调度和优先级抢占。
