# 主站状态设计方案

> 版本：v3.2（边界收缩后：`ActivityScheduler` / `ProcessDataEngine` 已从 `EcatController` 独立）

> 说明：本文是早期扁平状态机设计方案，记录了当前实现的历史背景。后续状态机架构演化以 `docs/planning/主站状态机分层演化方案.md` 为准，目标是逐步迁移到“五大主状态 + 进入/稳定/退出 + 准备态内部阶段”的分层模型。

## 1. 背景

`EcatController` 维护一条线性状态机，用于描述 EtherCAT 主站从网卡绑定到 OP 运行的正常运行流程：

```
Uninitialized → AdapterReady → Scanned → Maintenance → Operational → Error
```

其中 `Maintenance` 对应从站 PREOP 阶段，是 SDO 维护、诊断、参数标定的主战场；`Operational` 对应 PDO 周期运行阶段。

> **设计原则**：状态只表示稳定工况。PDO 配置、SafeOp、DC 同步等作为 `StartOperation()` 的内部过渡步骤，不暴露为独立状态。

该设计能够满足当前阶段“初始化到 PREOP”以及后续 PDO 周期运行的需求，同时为 FOE 升级、单站复位、在线诊断、参数标定等维护类操作通过 `EcatActivity` 框架提供支撑。

---

## 2. 当前状态机的优势与边界

### 2.1 优势

- **状态语义清晰**：每个状态对应一个明确的稳定工况。
- **转换关系简单**：线性迁移，命令驱动。
- **与 Activity 框架正交**：维护操作不改变 `ControllerState`。
- **周期任务归属明确**：
  - `kMaintenance`：周期性 `ProcessDataEngine::CheckSlaveStates()`
  - `kOperational`：周期性 `ProcessDataEngine::RunOnce()` + `ProcessDataEngine::CheckSlaveStates()`

### 2.2 边界

- 所有从站必须同时满足期望状态，主站才能维持在 `kMaintenance` 或 `kOperational`。
- 单站 BOOTSTRAP（如 FOE 升级）需要 Activity 内部临时改变该从站状态，但 Activity 不改变 `ControllerState`。
- 当前阶段不进入 `kOperational`，PDO 周期运行作为后续阶段启用。

---

## 3. 推荐方案：6 状态主站生命周期 + 独立维护活动

推荐采用“**主站生命周期状态 + EcatActivity 独立维护活动**”的二维设计。

### 3.1 核心思想

- **主站状态机（纵向）**：保持简单、清晰的线性生命周期，只描述主站正常运行阶段。主站状态只能由 `EcatController::TransitionTo()` 改变。
- **维护活动（横向）**：把 FOE 升级、SDO 诊断、单站复位、参数标定等操作封装为独立的 `EcatActivity`，在主站某个状态下执行，不污染主站状态机。
- **从站独立状态**：每个从站可以独立进入 BOOTSTRAP、INIT、PREOP 等状态，不受其他从站束缚。

关键原则：**Activity 不改变 `ControllerState`**。Activity 内部可以改变单个从站的 EtherCAT 状态，但主站生命周期状态只由 `TransitionTo()` 管理。Activity 失败后是否进入 `kError`，由 `EcatController` 根据失败策略统一处理。

### 3.2 主站状态机保留线性流程

```cpp
enum class ControllerState {
    kUninitialized,
    kAdapterReady,
    kScanned,
    kMaintenance,
    kOperational,
    kError
};
```

状态转换通过 `TransitionTo()` 统一入口管理。主站状态只回答一个问题：**主站当前处于正常生命周期的哪个稳定阶段？**

#### 3.2.1 前进路径受转换表约束

相邻公开状态之间的转换由调用方（如 `EcatApplication` 或便捷接口 `Initialize()`）按顺序调用 `TransitionTo()` 完成，每步都通过 `kAllowedTransitions` 查表校验。例如：

- `InitializeAdapter()` → `Uninitialized -> AdapterReady`
- `Scan()` → `AdapterReady -> Scanned`
- `EnterMaintenance()` → `Scanned -> Maintenance`

`StartOperation()` 是一个**复合转换**，内部顺序执行 PDO 配置、SafeOp、DC 配置、Op，但只有最终到达 `kOperational` 时才更新 `ControllerState`。任一子步骤失败进入 `kError`。

非法转换会被 `TransitionTo()` 拒绝并记录错误。

#### 3.2.2 `StartOperation()` 语义

`StartOperation()` 当前语义为：**从 `kMaintenance` 开始，执行完整的 PDO 启动流程直至 `kOperational`**。

内部子步骤失败时进入 `kError`，不暴露中间状态给上层。

#### 3.2.3 DC 配置失败策略

`DoDcConfigure()` 中需要明确 DC 失败的策略：

- 如果 `use_dc == false`，直接跳过，不视为失败；
- 如果 `use_dc == true` 且所有从站都支持 DC，配置失败才进入 `kError`；
- 如果部分从站不支持 DC，是否允许跳过，需要后续根据实际需求确定。

### 3.3 引入 EcatActivity 抽象

```cpp
class EcatActivity {
public:
    virtual ~EcatActivity() = default;

    // 活动名称，用于日志
    virtual const char *GetName() const = 0;

    // 是否允许在给定主站状态下启动
    virtual bool CanStart(ControllerState state) const = 0;

    // 执行活动，返回是否成功
    virtual bool Execute() = 0;

    // 活动失败后的策略
    virtual ActivityFailurePolicy GetFailurePolicy() const {
        return ActivityFailurePolicy::kEnterError;
    }
};
```

注意：接口中**没有** `GetTargetState()`。Activity 不直接改变 `ControllerState`，因此不存在“目标主站状态”。

### 3.4 Activity 失败策略

为 Activity 增加失败策略枚举，避免所有 Activity 失败都直接进入 `kError`：

```cpp
enum class ActivityFailurePolicy {
    kKeepControllerState,  // 只记录日志，保持当前主站状态
    kEnterError,           // 进入 kError，等待 Stop 后恢复
    kShutdown,             // 自动调用 Stop() 回到 Uninitialized
};
```

例如：

- SDO 诊断失败：`kKeepControllerState`
- 单站恢复失败：`kEnterError`
- FOE 升级失败：根据配置决定

### 3.5 Activity 执行器：先同步，后异步

当前阶段不建议直接引入后台线程执行 Activity，原因：

- SOEM 上下文并发访问风险高；
- `Stop()` 等待 Activity 结束存在阻塞风险；
- 当前没有 Activity 取消、抢占、优先级机制。

因此第一阶段采用同步执行模型，由独立的 `ActivityScheduler` 负责调度：

```cpp
class ActivityScheduler {
public:
    explicit ActivityScheduler(EcatController &controller);
    bool Execute(std::unique_ptr<EcatActivity> activity);
    bool IsRunning() const;

private:
    EcatController &controller_;
    std::atomic<bool> running_{false};
};

bool ActivityScheduler::Execute(std::unique_ptr<EcatActivity> activity)
{
    if (!activity) {
        return false;
    }

    if (running_.exchange(true)) {
        LOG_ERROR << "Activity already running";
        return false;
    }

    if (!activity->CanStart(controller_.GetState())) {
        LOG_ERROR << "Activity " << activity->GetName()
                  << " cannot start in state "
                  << EcatController::StateToString(controller_.GetState());
        running_ = false;
        return false;
    }

    LOG_INFO << "Activity started: " << activity->GetName();
    const bool ok = activity->Execute();
    LOG_INFO << "Activity finished: " << activity->GetName()
             << ", result=" << (ok ? "ok" : "failed");
    running_ = false;

    if (!ok) {
        switch (activity->GetFailurePolicy()) {
        case ActivityFailurePolicy::kEnterError:
            controller_.RequestErrorState(std::string("Activity failed: ") + activity->GetName());
            break;
        case ActivityFailurePolicy::kShutdown:
            controller_.Stop();
            break;
        case ActivityFailurePolicy::kKeepControllerState:
        default:
            LOG_WARN << "Activity failed but keeping controller state";
            break;
        }
    }

    return ok;
}
```

`EcatController` 不再持有 `activity_running_`，也不再提供 `ExecuteActivity()`。后续如确实需要异步执行，可在 `ActivityScheduler` 内部扩展：

```cpp
std::future<bool> StartActivityAsync(std::unique_ptr<EcatActivity> activity);
bool CancelActivity();
```

### 3.6 从站独立状态控制

`SlaveNode` 提供完整的单站状态控制能力，但底层状态切换应复用 `EcMaster::RequestStateWithRetry()`：

```cpp
class SlaveNode {
public:
    // 请求任意 EtherCAT 状态，包括 BOOTSTRAP
    bool RequestState(uint16_t state);

    // 刷新并返回从站真实状态
    uint16_t RefreshActualState();

    // FOE 写入（升级固件），底层由 EcMaster 实现
    bool FoEWrite(const std::string &filename, const std::vector<uint8_t> &data,
                  int timeout_us = 60000);

    // 等待从站重新上线（升级后复位）
    bool WaitForReboot(int timeout_us = 30000000);
};
```

建议在 `EcMaster` 层增加清晰封装：

```cpp
bool EcMaster::RequestBootstrapState(int slave);
bool EcMaster::FoEWrite(uint16_t slave, const std::string &filename,
                        const std::vector<uint8_t> &data, int timeout_us);
```

`SlaveNode` 只做转发和单站语义封装：

```cpp
bool SlaveNode::FoEWrite(const std::string &filename,
                         const std::vector<uint8_t> &data, int timeout_us)
{
    return master_.FoEWrite(info_.slave_id, filename, data, timeout_us);
}
```

### 3.7 活动与主站状态的协调规则

| 规则 | 说明 |
|---|---|
| **Activity 不改变 `ControllerState`** | 主站状态只能由 `TransitionTo()` 改变 |
| **运行时活动互斥** | 由 `ActivityScheduler` 保证：一个 Activity 运行期间，不允许启动另一个 Activity |
| **状态前置检查** | 活动通过 `CanStart()` 声明允许启动的主站状态 |
| **失败策略可配置** | 通过 `GetFailurePolicy()` 决定失败后是保持状态、进 Error 还是 Shutdown |
| **Activity 与 PDO 周期互斥（第一阶段保守策略）** | 高风险 Activity 执行期间暂停 PDO 周期或禁止在 OP 下执行；低风险只读 Activity 可与 PDO 并发 |
| **Stop 等待活动** | 同步模型下 `Stop()` 由调用方在活动结束后执行；异步模型下需要强制取消机制 |

## 4. 典型维护场景与分阶段落地

FOE 升级只是众多维护场景中的一种。以下是 EtherCAT 主站在实际工程中需要面对的主要维护场景，按复杂度分三阶段落地。

### 4.1 第一阶段：只读/低风险 Activity

| 场景 | 说明 | 对应 Activity |
|---|---|---|
| **SDO 诊断** | 读取对象字典、固件版本、错误码 | `SdoDiagnosticsActivity` |
| **固件版本巡检** | 批量读取 `0x100A` 或厂商自定义索引，比对版本 | `FirmwareAuditActivity` |
| **状态巡检** | 读取 AL status、ESC 寄存器 | `StateInspectionActivity` |

特点：

- 不改变从站状态；
- 不影响 PDO；
- 不需要 BOOTSTRAP；
- 适合验证 Activity 框架。

### 4.2 第二阶段：单站状态切换 Activity

| 场景 | 说明 | 对应 Activity |
|---|---|---|
| **单站复位** | 某个从站软复位后重新上线 | `SlaveResetActivity` |
| **错误恢复** | 从站进入 ERROR 后 ACK 并尝试恢复 | `ErrorRecoveryActivity` |
| **SDO 参数读写** | 在线修改 PDO 映射、限位、控制参数 | `SdoParameterActivity` |
| **零点校准** | 伺服电机找零、编码器偏移写入 | `HomingActivity` |

特点：

- 需要单站 `RequestState()`；
- 可能影响目标从站 PDO；
- 需要状态刷新和失败恢复策略。

### 4.3 第三阶段：高风险 Activity

| 场景 | 说明 | 对应 Activity |
|---|---|---|
| **FOE 固件升级** | 通过 BOOTSTRAP 写入 `.bin` / `.hex` | `FoEUpgradeActivity` |
| **EEPROM 烧录** | 修改从站 EEPROM | `EepromUpdateActivity` |
| **热插拔/从站替换** | 运行中更换从站并重新配置 | `HotSwapActivity` / `SlaveReplaceActivity` |

特点：

- 涉及 BOOTSTRAP、EEPROM、固件写入；
- 失败后可能需要人工恢复；
- 需要完整超时、重试、恢复和审计日志。

### 4.4 这些场景的共同要求

1. **单站级独立控制**：很多操作只针对一个或部分从站，不能影响其他从站。
2. **状态可临时降级**：升级/标定往往需要目标从站从 OP → SAFEOP → PREOP/BOOTSTRAP。
3. **可与正常运行并发**：理想情况下，升级 A 站时，B/C/D 站继续运行。
4. **失败隔离**：A 站升级失败不应直接把整站拉进 Error，除非配置为必须全站停止。
5. **可组合/可编排**：比如“先停线 → 批量升级 3 个站 → collectively 回到 OP”。
6. **错误恢复策略多样**：有的失败可以重试，有的需要人工干预，有的需要整站停止。

## 5. FOE 升级场景示例

假设当前主站处于 `kOperational`，4 个从站都在 OP。收到升级从站 2 的指令：

```cpp
auto *node = controller.GetSlaveNodeManager().GetNode(2);  // 仅供底层周期/维护模块使用
auto activity = std::make_unique<FoEUpgradeActivity>(*node, "firmware.bin", firmware_data);
scheduler.Execute(std::move(activity));
```

> 第一阶段采用保守策略：执行 `FoEUpgradeActivity` 前，主站先 `TransitionTo(kMaintenance)` 或暂停 PDO 周期，避免 Activity 与 PDO 并发。后续再支持“OP 下只跳过目标从站”的高级模式。

`FoEUpgradeActivity::Execute()` 内部流程：

1. 保存从站 2 升级前的状态。
2. 请求从站 2 进入 SAFEOP（如主站仍在 OP）。
3. 请求从站 2 进入 INIT。
4. 请求从站 2 进入 BOOTSTRAP。
5. 通过 FOE 写入固件文件。
6. 校验写入结果。
7. 发送复位命令，等待从站重新上线。
8. 重新配置该从站（PDO 映射等）。
9. 请求从站 2 回到原来的运行状态。
10. 如有需要，主站恢复 OP。

整个过程中：
- 主站生命周期状态由 `TransitionTo()` 管理，不由 Activity 直接修改；
- 只有从站 2 经历 OP → SAFEOP → INIT → BOOTSTRAP → INIT → PREOP → SAFEOP → OP；
- 升级失败时根据 `GetFailurePolicy()` 决定保持状态、进 `kError` 或 Shutdown。

## 6. Maintenance/Operational 阶段从站掉线检测

`CheckSlaveStates()` 已放到 `ProcessDataEngine`，覆盖两个稳定状态：

```cpp
void ProcessDataEngine::CheckSlaveStates()
{
    master_.CheckSlaveStates();

    switch (controller_.GetState()) {
    case ControllerState::kMaintenance:
        if (!master_.CheckAllSlavesInState(EC_STATE_PRE_OP)) {
            controller_.RequestErrorState("Slave dropped out of PREOP");
        }
        break;
    case ControllerState::kOperational:
        if (!master_.CheckAllSlavesInState(EC_STATE_OPERATIONAL)) {
            controller_.RequestErrorState("Slave dropped out of OPERATIONAL");
        }
        break;
    default:
        break;
    }
}
```

在 `EcatApplication::Run()` 的主循环中：

- `kMaintenance` 状态下每个周期调用 `engine_.CheckSlaveStates()`。
- `kOperational` 状态下每个周期调用 `engine_.RunOnce()` + `engine_.CheckSlaveStates()`。

周期频率由 `main` 的调用频率决定（当前默认 1ms）。

## 7. Activity 方案支撑多场景时还需补强的设计点

### 7.1 Activity 依赖与编排

例如“批量升级”Activity 内部可能顺序执行多个 `FoEUpgradeActivity`；或者“停线升级”需要先执行 `SafeStopActivity`，再执行批量升级，最后恢复运行。需要支持 Activity 的组合或子任务编排。

### 7.2 Activity 优先级与取消

急停 / 安全停止应该能打断普通升级 Activity。Activity 执行器需要支持优先级抢占或取消机制。该能力在同步模型稳定后再引入。

### 7.3 部分失败策略

批量升级 4 个站，1 个失败时，是继续升级剩下 3 个，还是全部回滚？需要在 Activity 或调度层配置策略。

### 7.4 Activity 与 PDO 周期的并发

升级 A 站时，B/C/D 站仍在 OP 跑 PDO。需要确保 `ProcessDataEngine::RunOnce()` 不访问正在 BOOTSTRAP 的 A 站，或者在访问前跳过未就绪节点。

第一阶段建议采用保守策略：高风险 Activity 执行期间主站处于 `kMaintenance`，PDO 周期暂停。

### 7.5 从站状态缓存与同步

`SlaveNode` 需要实时跟踪自己当前实际状态，而不是依赖 `current_state_` 这个写入值。建议增加 `RefreshActualState()`。Activity 执行前后必须刷新从站真实状态。

### 7.6 全局活动 vs 局部活动

安全停止是所有从站的全局活动；FOE 升级是局部活动。调度策略、失败影响范围、资源锁粒度都不同，后续可能需要区分 `GlobalActivity` 和 `SlaveActivity`。

## 8. 与当前代码的衔接

当前代码已完成主站状态机框架重构和边界收缩（阶段 1~3），为 Activity 方案的落地打下了基础。具体已实现的内容包括：

- `ControllerState` 已精简为 6 个稳定状态：`kUninitialized` / `kAdapterReady` / `kScanned` / `kMaintenance` / `kOperational` / `kError`。
- `TransitionTo(ControllerState target)` 统一入口已实现，基于 `kAllowedTransitions` 校验，支持相邻转换和 `Maintenance -> Operational` 复合转换。
- `DoStepTo()`、`DoInit()`、`DoScan()`、`DoEnterMaintenance()`、`DoPdoConfigure()`、`DoSafeOp()`、`DoDcConfigure()`、`DoOperational()`、`DoShutdown()` 已拆分实现。
- `RequestErrorState()` 已实现，供 `ActivityScheduler` / `ProcessDataEngine` 请求进入 `kError`。
- `EcatApplication::Run()` 改为单步执行器，由 `main` 周期调用；基于 `ControllerState` 分发命令，不维护独立应用状态。
- `CommandReader` 抽象已引入，`StdinCommandReader` 实现非阻塞 stdin 读取。
- `ActivityScheduler` 已独立：负责 Activity 启动检查、同步执行、失败策略处理。
- `ProcessDataEngine` 已独立：负责 PDO 周期 `RunOnce()` 和从站状态监控 `CheckSlaveStates()`。
- `EcatController::ForEachSlaveNode()` 已提供，上层不再直接访问 `SlaveNodeManager`。
- `operational_` / `initialized_` 成员已移除，`IsOperational()` 和 `IsInitialized()` 基于 `state_` 判断。
- `DoShutdown()` 已根据当前 `state_` 精确回滚。

在此基础上，推荐按以下顺序继续落地 Activity 方案：

1. **补强单站状态能力**
   - `SlaveNode::RefreshActualState()`；
   - `EcMaster::RequestBootstrapState(int slave)`；
   - `SlaveNode::RequestState()` 统一走 `RequestStateWithRetry()`。

2. **扩展低风险 Activity**
   - 已实现 `SdoDiagnosticsActivity` / `SdoParameterActivity` / `StateInspectionActivity`；
   - 后续可增加 `FirmwareAuditActivity` 等只读 Activity。

3. **实现单站恢复类 Activity**
   - `ErrorRecoveryActivity`；
   - `SlaveResetActivity`。

4. **最后实现 FOE 升级**
   - 增加 `EcMaster::FoEWrite()`；
   - 增加 BOOTSTRAP 状态切换；
   - 增加超时、失败恢复、审计日志。

5. **根据实际需求再考虑异步 Activity / 监控线程 / 优先级抢占**
   - 异步化应在 `ActivityScheduler` 内部扩展，不影响 `EcatController` 状态机。

## 9. 为什么选这个方案，而不是其他方案

### 9.1 为什么不把维护状态塞进主状态机

如果把 `kFoEUpgrading`、`kDiagnostics` 等作为 `ControllerState` 的新值，主状态机会变成一张复杂的网，正常流程和维护流程相互纠缠。每新增一种维护操作，都要修改状态转换表和 `DoXxx()`，长期不可维护。

### 9.2 为什么不采用层次化状态机（HSM）

HSM 很适合复杂维护流程，但会显著增加代码复杂度。当前项目阶段还不确定维护流程的细节，过早引入 HSM 容易导致过度设计。Activity 方案可以在 HSM 变得必要时平滑迁移过去。

### 9.3 Activity 方案的优势

| 方面 | 说明 |
|---|---|
| **侵入性小** | 主站状态机保持简单线性，新增维护活动不需要改状态转换表 |
| **扩展性好** | FOE 升级、SDO 诊断、单站复位、参数标定各自一个 Activity 类 |
| **职责清晰** | 主站状态管生命周期，Activity 管维护操作，从站状态管单站行为 |
| **可测试** | 每个 Activity 可以独立单元测试 |
| **与现有架构兼容** | 不需要推翻 `SlaveNode`、`SlaveNodeManager` 等已有设计 |
| **可控的复杂度** | 先同步、后异步；先低风险、后高风险 |

## 10. 代价

- 需要新增 `EcatActivity` 基类、`ActivityFailurePolicy`、`ActivityScheduler` 等基础设施（`ActivityScheduler` 已落地）。
- `SlaveNode` 需要补充单站状态刷新、更 robust 的状态控制。
- `EcMaster` 需要增加 BOOTSTRAP 切换、FOE 写入等底层封装。
- 活动和 PDO 周期、Stop、错误恢复之间的互斥策略需要仔细设计。
- 当 Activity 种类增多后，可能需要进一步抽象出 `GlobalActivity` / `SlaveActivity`、Activity 优先级、子任务编排、异步执行等机制。

总体而言，这是当前阶段**复杂度与扩展性最平衡**的方案，既能支撑 FOE 升级，也能自然扩展到 SDO 诊断、伺服标定、热插拔、安全停止等后续场景。
