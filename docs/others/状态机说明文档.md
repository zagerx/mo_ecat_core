# EcatController 状态机说明文档

> 版本：v3.3（同步当前 `Dispatch(MasterAction)` 实现，并补充 `ReadyToRun` / `Fault` / `EmergencyStop`）

## 1. 概述

`EcatController` 是 EtherCAT 主站生命周期控制器，内部维护一个有限状态机（FSM），用于管理主站从网卡绑定到 OP 运行的完整流程，以及异常时的错误状态表达。

所有状态转换都通过统一入口 `Dispatch(MasterAction action)` 完成，集中处理动作合法性检查、转换执行、失败回滚和日志记录。

当前阶段（PREOP + SDO）主站生命周期走到 `kMaintenance` 即稳定，后续通过命令 `start` 或上层调用 `StartOperation()` 进入 PDO 阶段。

---

## 2. 状态定义

```cpp
enum class ControllerState {
    kUninitialized,   // 初始状态：未绑定网卡、未扫描从站
    kAdapterReady,    // SOEM 初始化完成，网卡已绑定
    kScanned,         // 从站扫描完成，已获取 SlaveInfo
    kMaintenance,     // 所有从站进入 PREOP，可执行 SDO 维护活动
    kReadyToRun,      // PDO / IOmap / SAFEOP / DC 已准备，等待进入 OP
    kOperational,     // 所有从站进入 OPERATIONAL，PDO 周期可运行
    kFault,           // 普通故障状态：只能 Stop() 或后续受控恢复
    kEmergencyStop    // 紧急安全事件：禁止自动恢复
};
```

> **当前设计原则**：状态只表示相对稳定的主站工况，不表示一次性动作。`kPdoConfigured` / `kSafeOp` / `kDcConfigured` 等步骤作为 `PrepareRun()` 的内部子步骤，不暴露为 `ControllerState`。

当前阶段（PREOP + SDO）只走到 `kMaintenance`，不进入 `kOperational`。

## 3. 状态转换规则

### 3.1 允许的动作表

```cpp
static const std::map<ControllerState, std::vector<MasterAction>> kAllowedActions = {
    {kUninitialized, {kInitializeAdapter}},
    {kAdapterReady,  {kScanSlaves, kStop}},
    {kScanned,       {kEnterMaintenance, kStop}},
    {kMaintenance,   {kPrepareRun, kStop}},
    {kReadyToRun,    {kStartOperation, kBackToMaintenance, kStop}},
    {kOperational,   {kBackToMaintenance, kStop}},
    {kFault,         {kStop}},
    {kEmergencyStop, {kStop}}
};
```

表中记录当前状态允许执行的动作。`kRequestFault` 和 `kRequestEmergencyStop` 是内部故障/急停请求动作，可由监控模块直接触发，不走普通动作表。

### 3.2 顺序前进与复合转换

相邻公开状态之间的转换通过公开接口触发，再由 `Dispatch(MasterAction)` 校验并执行。调用方（如 `EcatApplication` 或便捷接口 `Initialize()`）负责按顺序推进：

- 典型命令流：
  - `InitializeAdapter()` → `Uninitialized -> AdapterReady`
  - `Scan()` → `AdapterReady -> Scanned`
  - `EnterMaintenance()` → `Scanned -> Maintenance`
- `Initialize()` 便捷接口内部顺序调用上述三步，直接到达 `kMaintenance`。
- `PrepareRun()` 是一个**复合转换**：
  - `Maintenance -> (内部 PdoConfigure -> SafeOp -> DcConfigure) -> ReadyToRun`
- `StartOperation()` 只负责最后一步：
  - `ReadyToRun -> Operational`

`PdoConfigure` / `SafeOp` / `DcConfigure` 等内部步骤不单独暴露为 `ControllerState`，只有全部成功后才进入 `kReadyToRun`。

### 3.3 非法转换

- 从 `kUninitialized` 直接跳到 `kOperational` 会被拒绝。
- 从 `kFault` / `kEmergencyStop` 只能调用 `Stop()` 回到 `kUninitialized`；其他普通动作都会被拒绝。

## 4. 状态转换实现

| 方法 | 前置状态 | 作用 | 失败策略 |
|---|---|---|---|
| `DoInit()` | `kUninitialized` | `ecx_init()` 绑定网卡 | 保持 `kUninitialized` |
| `DoScan()` | `kAdapterReady` | `ecx_config_init()` 扫描从站，打印 SlaveInfo | 调用 `DoShutdown()` 回滚 |
| `DoEnterMaintenance()` | `kScanned` | 请求 PREOP 并确认，创建占位 SlaveNode | 调用 `DoShutdown()` 回滚 |
| `DoPrepareRun()` | `kMaintenance` | 依次执行 PDO 配置、IOmap/SAFEOP、DC 配置 | 进入 `kFault` |
| `DoPdoConfigure()` | `kMaintenance` | 调用 `SlaveNodeManager::ConfigureAll()` 填充配置 | 进入 `kFault` |
| `DoSafeOp()` | 内部过渡 | `ConfigureProcessData()` + 请求 SAFEOP 并确认 | 进入 `kFault` |
| `DoDcConfigure()` | 内部过渡 | `ConfigureDc()` 配置分布式时钟 | 进入 `kFault` |
| `DoOperational()` | `kReadyToRun` | 请求 OPERATIONAL 并确认 | 进入 `kFault` |
| `BackToMaintenance()` | `kReadyToRun` / `kOperational` | 请求回 PREOP 并确认 | 失败进入 `kFault` |
| `DoShutdown()` | 任意 | 根据当前状态逐级回滚，最终关闭 SOEM | 总是成功 |
| `RequestFault()` | 任意 | 设置 `state_ = kFault` 并记录原因 | — |
| `RequestEmergencyStop()` | 任意 | 设置 `state_ = kEmergencyStop` 并记录原因 | — |

### 4.1 回滚路径

`DoShutdown()` 根据当前状态精确回滚：

```text
Operational -> SAFE_OP -> PRE_OP -> INIT -> Close
ReadyToRun -> PRE_OP -> INIT -> Close
Maintenance -> PRE_OP -> INIT -> Close
Scanned / AdapterReady -> 直接 Close
Uninitialized -> 无操作
```

Maintenance 阶段停止时会先请求 PRE_OP，再请求 INIT，最后关闭 SOEM。

### 4.2 复合转换 Maintenance -> ReadyToRun

`PrepareRun()` 不经过公开的中间状态，而是通过 `DoPrepareRun()` 一次性完成：

```text
Maintenance
    ├── DoPdoConfigure()   // 配置 PDO 映射
    ├── DoSafeOp()         // 配置过程数据，进入 SAFEOP
    ├── DoDcConfigure()    // 配置分布式时钟
    └── state_ = kReadyToRun
```

任一子步骤失败都会调用 `RequestFault()`，`state_` 变为 `kFault`。成功前 `state_` 始终保持 `kMaintenance`。

### 4.3 最终进入 OPERATIONAL

`StartOperation()` 只从 `kReadyToRun` 请求进入 OP：

```text
ReadyToRun
    └── DoOperational()    // 进入 OPERATIONAL，state_ = kOperational
```

## 5. 公共接口

### 5.1 生命周期接口

```cpp
// 便捷接口：Uninitialized -> AdapterReady -> Scanned -> Maintenance
bool Initialize(const EcMasterConfig &config);

// 只初始化网卡/SOEM，到 AdapterReady
bool InitializeAdapter(const EcMasterConfig &config);

// AdapterReady -> Scanned
bool Scan();

// Scanned -> Maintenance
bool EnterMaintenance();

// Maintenance -> ReadyToRun（PDO / IOmap / SAFEOP / DC 准备）
bool PrepareRun();

// ReadyToRun -> Operational（PDO 阶段使用）
bool StartOperation();

// ReadyToRun / Operational -> Maintenance（真实请求回 PREOP）
bool BackToMaintenance();

// 安全停止，回到 kUninitialized
void Stop();
```

### 5.2 状态查询接口

```cpp
ControllerState GetState() const;                          // 获取当前状态
size_t GetSlaveCount() const;                              // 获取扫描到的从站数
static const char *StateToString(ControllerState state);   // 状态转字符串

bool IsInitialized() const;  // state_ != kUninitialized
bool IsOperational() const;  // state_ == kOperational
```

### 5.3 节点遍历与错误状态接口

```cpp
// 安全遍历所有 SlaveNode，不对外暴露 SlaveNodeManager
void ForEachSlaveNode(const std::function<void(SlaveNode &)> &callback);

// 请求进入故障状态（供 ActivityScheduler / ProcessDataEngine 调用）
void RequestFault(const std::string &reason);

// 请求进入紧急停止状态（供安全 IO / 外部急停 / 严重故障调用）
void RequestEmergencyStop(const std::string &reason);

// 仅供底层周期模块使用，EcatApplication 禁止调用
EcMaster &GetEcMaster();
SlaveNodeManager &GetSlaveNodeManager();
```

> **边界规则**：`RunOneCycle()` / `CheckSlaveStates()` / `ExecuteActivity()` / `IsActivityRunning()` 已分别移入 `ProcessDataEngine` 和 `ActivityScheduler`，不再由 `EcatController` 提供。

## 6. 使用示例

### 6.1 当前命令行交互用法

```bash
sudo ./ecat_scan enp0s31f6
```

启动后输入命令：

```text
scan
config
diagnose
inspect
stop
exit
```

对应的状态迁移：

```text
Uninitialized --InitializeAdapter--> AdapterReady --scan--> Scanned --config--> Maintenance
```

`diagnose` / `inspect` 等 Activity 在 `kMaintenance` 下执行，不改变 `ControllerState`。

### 6.2 编程式用法（后续 PDO 阶段）

```cpp
mo_ecat::EcatController controller;
mo_ecat::ActivityScheduler scheduler(controller);
mo_ecat::ProcessDataEngine engine(controller, controller.GetEcMaster(),
                                 controller.GetSlaveNodeManager());

// 走到 Maintenance
if (!controller.Initialize(config)) return 1;

// 可选：SDO 配置 PDO 映射（通过 scheduler.Execute(...) 执行 Activity）

// 准备到 SAFEOP/DC 就绪
if (!controller.PrepareRun()) return 1;

// 进入 OP
if (!controller.StartOperation()) return 1;
// controller.GetState() == kOperational

while (running) {
    engine.RunOnce();
    engine.CheckSlaveStates();
    if (!controller.IsOperational()) {
        // 进入 kFault，需要 Stop() 后恢复
        break;
    }
}

controller.Stop();
```

## 7. 异常处理

### 7.1 初始化阶段异常

`DoInit()`、`DoScan()`、`DoEnterMaintenance()` 失败时调用 `DoShutdown()`，最终回到 `kUninitialized`，不进入 `kFault`。

### 7.2 PDO 阶段异常

`PrepareRun()` 内部任一子步骤（PDO 配置、SafeOp、DC 配置）失败时进入 `kFault`。`StartOperation()` 请求 OP 失败时也进入 `kFault`。此时只能调用 `Stop()` 回到 `kUninitialized`。

### 7.3 运行中掉线

`CheckSlaveStates()` 已移到 `ProcessDataEngine`，覆盖两个稳定状态：

```cpp
void ProcessDataEngine::CheckSlaveStates()
{
    master_.CheckSlaveStates();

    switch (controller_.GetState()) {
    case ControllerState::kMaintenance:
        if (!master_.CheckAllSlavesInState(EC_STATE_PRE_OP)) {
            controller_.RequestFault("Slave dropped out of PREOP");
        }
        break;
    case ControllerState::kOperational:
        if (!master_.CheckAllSlavesInState(EC_STATE_OPERATIONAL)) {
            controller_.RequestFault("Slave dropped out of OPERATIONAL");
        }
        break;
    default:
        break;
    }
}
```

进入 `kFault` 后：

- `ProcessDataEngine::RunOnce()` 非 `kOperational` 时不执行 PDO 周期。
- `StartOperation()` 等转换被拒绝。
- 必须 `Stop()` 回到 `kUninitialized` 后重新初始化。

## 8. 注意事项

1. **当前阶段只到 `kMaintenance`**：命令行不调用 `start` / `StartOperation()`，不会进入 PDO 阶段。
2. **占位节点**：`kMaintenance` 时 `SlaveNode` 只是占位对象，不分配 PDO buffer，不创建 `DeviceProfile`。
3. **Activity 不改变状态**：所有维护 Activity 在 `kMaintenance` 下执行，由 `ActivityScheduler` 调度，失败策略由 Activity 自己声明。
4. **状态查询**：优先使用 `IsInitialized()` / `IsOperational()` 或 `GetState()`，不要直接依赖内部 `state_`。
5. **日志**：关键状态转换点会打印日志；扫描后打印一次完整 SlaveInfo，进入 Maintenance 后只打印 summary。
6. **过渡步骤不暴露**：PDO 配置、SafeOp、DC 配置是 `StartOperation()` 的内部实现细节，不对外暴露为独立状态。
7. **周期任务已剥离**：`RunOnce()` / `CheckSlaveStates()` 由 `ProcessDataEngine` 实现；`EcatController` 只负责状态迁移和状态保护。

## 9. 后续可扩展点

- 增加 `start` 命令，调用 `StartOperation()` 进入 PDO 阶段。
- `ProcessDataEngine::CheckSlaveStates()` 返回更详细的异常信息（从站 id、name、状态码）。
- 增加自动降级/重试策略，例如从 `kFault` 尝试回到 `kMaintenance` 而不是只能 Stop。
- 为 `ActivityScheduler` 增加异步执行 / 取消 / 优先级能力。
