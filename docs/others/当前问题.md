# 当前问题与后续演进优先级

> 本文基于当前源码阅读与架构分析整理，用于替代旧的边界收缩记录。当前框架整体方向正确，已经形成 `EcatApplication` 编排、`EcatController` 生命周期、`EcMaster` SOEM 封装、`ActivityScheduler` 维护任务、`ProcessDataEngine` 周期任务的分层。现阶段不建议推倒重来，优先处理稳定性、调试效率和后续扩展边界问题。

---

## 1. 总体判断

当前工程已经不是简单 SOEM 示例，而是具备分层雏形的 EtherCAT 主站框架。

| 项目 | 评价 |
|---|---|
| 架构方向 | 较好，值得继续沿用 |
| 当前代码可维护性 | 中上，模块尚未严重膨胀 |
| 后续扩展潜力 | 较高，已有 Activity / DeviceProfile / CommandReader 等扩展点 |
| 工程完备度 | 一般，配置、错误恢复、CiA402、动态 PDO 仍未完善 |
| 实时控制成熟度 | 偏早期，当前仍是 `main` 中 `while + sleep` 调度 |

核心结论：

```text
当前不需要推倒重来。
当前最重要的是先修几个基础稳定性问题，
然后再按 PDO/CiA402、单站维护、FOE、实时线程、配置外部化的顺序演进。
```

---

## 2. 当前必须立即解决的问题

这些问题修改成本小，但会直接影响稳定性、调试体验或后续扩展基础。

### ✅ 2.1 修正 `EcMaster::closed_` 资源生命周期语义

涉及文件：

- `src/ec_master/ec_master.h`
- `src/ec_master/ec_master.cpp`

当前问题：成员默认值是 `closed_ = true`，但构造函数中又设置为 `false`。这会导致对象刚构造、尚未 `Initialize()` 成功时，`Close()` 也可能认为 SOEM 已打开，从而调用 `ecx_close(&ctx_)`。

建议：

```cpp
EcMaster::EcMaster()
{
    std::memset(&ctx_, 0, sizeof(ctx_));
    std::memset(iomap_, 0, sizeof(iomap_));
    closed_ = true;
}

bool EcMaster::Initialize(const EcMasterConfig &config)
{
    config_ = config;

    if (!ecx_init(&ctx_, config_.ifname.c_str())) {
        LOG_ERROR << "ecx_init failed on " << config_.ifname << " (try sudo)";
        return false;
    }

    closed_ = false;
    LOG_INFO << "SOEM initialized on " << config_.ifname;
    return true;
}
```

优先级：**高**。

---

### ✅ 2.2 `ActivityScheduler::running_` 使用原子 `exchange()` 保证互斥

涉及文件：

- `src/activity/activity_scheduler.cpp`

当前代码：

```cpp
if (running_.load()) {
    LOG_ERROR << "Activity already running";
    return false;
}

running_.store(true);
```

单线程下问题不大，但未来命令输入、周期执行、维护 Activity 分线程后会有竞态窗口。

建议：

```cpp
if (running_.exchange(true)) {
    LOG_ERROR << "Activity already running";
    return false;
}
```

同时确保所有提前返回路径都会恢复 `running_.store(false)`。

优先级：**高**。

---

### ✅ 2.3 `EcatApplication::OnParam()` 增加异常处理与范围校验

涉及文件：

- `src/app/ecat_application.cpp`

当前直接调用 `std::stoul()`，用户输入非法参数时可能抛异常导致程序退出。例如：

```text
param abc 0 1
param 0xzz 0 1
param 0x123456789 0 1
```

建议增加：

1. `try/catch`。
2. `index <= 0xFFFF`。
3. `subindex <= 0xFF`。
4. `value <= 0xFFFFFFFF`。
5. 日志提示正确用法。

优先级：**高**。

---

### ✅ 2.4 `SlaveNode::RequestState()` 统一使用带重试的状态切换

涉及文件：

- `src/slave_node/slave_node.cpp`
- `src/ec_master/ec_master.cpp`

当前 `SlaveNode::RequestState()` 调用 `master_.RequestState()`，但 `EcMaster` 已经提供更稳健的 `RequestStateWithRetry()`。

建议：

```cpp
bool SlaveNode::RequestState(uint16_t state)
{
    if (!master_.RequestStateWithRetry(info_.slave_id, state)) {
        LOG_WARN << "SlaveNode " << info_.name << " failed to request state " << state;
        return false;
    }
    current_state_ = state;
    return true;
}
```

优先级：**高**。

---

### ✅ 2.5 `StateInspectionActivity` 中 AL status code 需要实时化或明确为缓存值

涉及文件：

- `src/activity/state_inspection_activity.cpp`
- `src/ec_master/ec_master.h/.cpp`
- `src/slave_node/slave_node.h/.cpp`

当前代码中 `actual_state` 是实时刷新值，但 `al_status_code` 来自 `SlaveInfo` 缓存，可能不是实时值。

建议二选一：

1. 增加 `EcMaster::ReadAlStatusCode(int slave)`，让 `inspect` 读取实时 AL status code。
2. 如果暂时不加接口，日志中明确写为 `cached_al_status_code`。

优先级：**中高**。

---

### ✅ 2.6 简化 `ProcessDataEngine::CheckSlaveStates()` 的重复状态读取

涉及文件：

- `src/cyclic/process_data_engine.cpp`
- `src/ec_master/ec_master.cpp`

当前 `ProcessDataEngine::CheckSlaveStates()` 先调用 `master_.CheckSlaveStates()`，随后 `CheckAllSlavesInState()` 内部又会 `ecx_readstate()`，造成重复读取。

建议短期删除 `ProcessDataEngine::CheckSlaveStates()` 中单独的：

```cpp
master_.CheckSlaveStates();
```

由 `CheckAllSlavesInState()` 统一刷新并判断。

优先级：**中**（影响较小，可顺手修）。

---

### ✅ 2.7 增加扫描后拓扑/身份校验

涉及文件：

- `src/ec_master/ec_master.h`
- `src/ec_controller/ec_controller.cpp`
- `src/ec_master/ec_master_config.h`（如不存在则新增）

当前问题：

`DoScan()` 只判断 `slave_infos.empty()`，不校验实际扫描到的从站数量和身份。现场可能出现：

- 预期 10 个从站，只扫到 5 个；
- 某个位置的从站型号/版本与预期不符；
- 从站顺序因接线或别名配置错误而对不上。

这些情况下如果继续进入 PREOP 乃至 OP，会导致后续 PDO 映射、CiA402 控制都建立在错误拓扑上。

建议：

1. 在 `EcMasterConfig` 或外部配置中增加 `expected_slave_count` 和 `expected_identities`（位置 → vendor_id / product_id / revision_id）。
2. `DoScan()` 完成后立即校验：

```cpp
if (slave_infos_.size() != config_.expected_slave_count) {
    LOG_ERROR << "Slave count mismatch: expected " << config_.expected_slave_count
              << ", found " << slave_infos_.size();
    DoShutdown();
    return false;
}

for (size_t i = 0; i < slave_infos_.size(); ++i) {
    const auto &expected = config_.expected_identities[i];
    const auto &actual   = slave_infos_[i];
    if (actual.vendor_id != expected.vendor_id ||
        actual.product_id != expected.product_id) {
        LOG_ERROR << "Slave[" << (i + 1) << "] identity mismatch: expected "
                  << expected.name << ", got " << actual.name;
        DoShutdown();
        return false;
    }
}
```

3. 校验失败时进入 `kError` 或停留在 `Scanned`，明确报告缺失/错位从站，禁止继续进入 `Maintenance`/`Operational`。

优先级：**高**。这是 PDO/CiA402 之前的安全闸门。

---

### ✅ 2.8 增强日志与现场调试能力

涉及文件：

- `src/utils/logger.h/.cpp`
- `src/ec_master/ec_master.cpp`
- `src/cyclic/process_data_engine.cpp`

当前问题：

- 进入 `kError` 时没有自动打印所有从站的快照（state / al_status_code）；
- PDO 周期运行后缺少周期时间、抖动、WKC mismatch 连续次数等统计；
- 日志级别不可通过命令或配置动态调整。

建议：

1. `RequestErrorState()` 或 `ProcessDataEngine::CheckSlaveStates()` 触发 Error 时，打印所有从站的 `slave_id`、`name`、`state`、`al_status_code`。
2. `EcMaster::RunOneCycle()` 增加周期统计：最小/最大/平均周期、连续 WKC mismatch 次数。
3. 支持通过命令 `loglevel debug|info|warn|error` 动态调整日志级别。

优先级：**中高**。对阶段 2/3 的稳定性验证非常关键。

## 3. 近期必须设计好的问题

这些问题不一定今天就改，但在进入 PDO、CiA402、FOE、单站维护之前必须明确。

### 3.1 全局主站状态与单站维护状态的冲突

当前 `ProcessDataEngine` 的逻辑是：

```text
Maintenance  -> 所有从站必须 PREOP
Operational  -> 所有从站必须 OP
```

这会和后续单站维护冲突，例如单站 FOE 升级进入 BOOTSTRAP、单站复位进入 INIT、单站错误恢复经历 PREOP/SAFEOP/OP 等。

建议后续引入每个从站自己的运行时状态：

```cpp
enum class SlaveHealth {
    Unknown,
    Online,
    Warning,
    Fault,
    Maintenance,
    Lost,
};

struct SlaveRuntimeState {
    uint16_t actual_state = 0;
    uint16_t expected_state = EC_STATE_INIT;
    bool state_check_exempt = false;
    std::string owner_activity;
    SlaveHealth health = SlaveHealth::Unknown;
};
```

`ProcessDataEngine` 后续不应只根据全局 `ControllerState` 一刀切，而应该逐站检查。

> 注意：阶段 2 做 CiA402 时就会遇到单站问题（例如某个从站 fault reset 会短暂离开 OP），因此 `SlaveRuntimeState` 的基础设施可以在阶段 2 开始搭建，不必等到阶段 3 才完整实现。

优先级：**近期高**。开始做单站 reset / recovery / FOE 前必须完成。

---

### 3.2 Activity 增加作用域与风险声明

当前 `EcatActivity` 只声明 `CanStart()` 和 `GetFailurePolicy()`，但后续调度器还需要知道：

- Activity 影响哪些从站？
- 是否会改变 EtherCAT 状态？
- 是否需要暂停 PDO？
- 是否允许在 OP 下执行？
- 失败影响范围是单站还是全站？

建议后续增加轻量元信息：

```cpp
enum class ActivityScope {
    Global,
    SingleSlave,
    MultiSlave,
};

enum class ActivityRiskLevel {
    ReadOnly,
    SdoWrite,
    StateChanging,
    FirmwareChanging,
};
```

优先级：**近期高**。实现 FOE、EEPROM、单站恢复前需要完成。

---

### 3.3 CiA402 / PDO 最小闭环

当前 `ServoAxis + ServoProfile` 还只是框架，缺少完整伺服控制闭环。

当前缺少：

- `control_word`
- `status_word`
- `mode_of_operation`
- `mode_of_operation_display`
- `error_code`
- CiA402 状态机
- fault reset / shutdown / switch on / enable operation 流程

推荐职责划分：

```text
ServoAxis
    只保存工程单位命令/反馈，不理解 EtherCAT。

CiA402Axis
    解析 status_word，生成 control_word，管理 CiA402 状态机。

ServoProfile
    负责 PDO 字节与 ServoAxis/CiA402Axis 之间的适配。
```

优先级：**近期高**。真正进入 OP 并尝试控制电机前必须完成。

---

### 3.4 PDO 周期中的 WKC 异常策略

当前 `EcMaster::RunOneCycle()` 只统计 WKC mismatch 次数，但不会触发任何策略。

建议增加连续异常判断：

```text
连续 3 次 WKC mismatch：记录 Warn
连续 10 次 WKC mismatch：请求进入 Error 或触发恢复策略
```

优先级：**近期中高**。开始持续 PDO 周期运行后必须补。

## 4. 可以后续逐步演进的问题

### 4.1 配置外部化

当前设备识别与配置主要硬编码在 `SlaveTypeDetector` 和 `SlaveConfigFactory`。短期设备少时可以接受。

当设备型号超过 2~3 种、同型号不同固件 PDO 不一致、现场需要频繁调整 scale/direction/limit 时，应推进 JSON/YAML 配置外部化。

优先级：**后续中**。

---

### 4.2 `CyclicRunner` 接入实时周期线程

当前 `CyclicRunner` 已实现但未接入，主循环仍是 `while + sleep_for(1ms)`。

进入真实 PDO 控制后，需要考虑独立周期线程、实时优先级、CPU 亲和性、周期抖动统计和 SOEM 访问互斥策略。

注意：接入 `CyclicRunner` 前，必须先明确 SOEM 访问模型。

优先级：**后续中高**。

---

### 4.3 SOEM 访问模型收敛

当前 `EcMaster` 用 mutex 保护 SOEM 上下文，短期单线程模型足够。

后续多线程后，需要在两种方向中选择：

1. 单 SOEM 线程 + 请求队列。
2. 保留 mutex + 增加 PDO / SDO / 状态切换 / Activity 等资源锁粒度。

建议短期使用资源锁过渡，但架构上预留向单 SOEM 线程演进的可能。

优先级：**后续中高**。

---

### 4.4 异步 Activity / 取消 / 优先级

当前 Activity 同步执行是合理的，简单且安全。不建议现在过早异步化。

当出现 FOE 升级耗时长、需要查询进度、需要取消 Activity、急停抢占普通维护任务时，再演进。

优先级：**后续中**。

---

### 4.5 CommandHandler / CommandRegistry

当前命令数量较少，`EcatApplication` 中 if/else 尚可维护。

当命令增多到 reset、recover、foe、axis move、axis home、pdo show、slave list 等规模时，应拆分命令处理器。

优先级：**后续中低**。

---

### 4.6 更细粒度的错误恢复策略

当前错误策略基本是 `RequestErrorState(reason) -> stop 回到 Uninitialized`，短期简单可靠。

后续多从站连续运行时，需要逐步引入单站 fault、warning、可恢复错误、不可恢复错误、自动 ACK、自动重新进入 OP、降级运行策略。

优先级：**后续中**。

---

## 5. 建议推进顺序

### 阶段 1：立即小修，保证基础稳定

```text
1. ✅ 修正 EcMaster::closed_ 生命周期语义
2. ✅ ActivityScheduler::running_ 改为 exchange()
3. ✅ OnParam 增加异常处理和范围校验
4. ✅ SlaveNode::RequestState 改用 RequestStateWithRetry()
5. ✅ inspect 的 AL status code 实时化
6. ✅ 增加扫描后拓扑/身份校验（expected_slave_count + expected_identities）
7. ✅ 增强日志与现场调试能力（Error 时从站快照、周期统计、loglevel 命令）
8. ✅ 简化 CheckSlaveStates 的重复 readstate（顺手修）
```

这一阶段不改变大架构，风险低，收益高。

---

### 阶段 2：补齐 PDO / CiA402 最小闭环

```text
1. 补充 CiA402 PDO 基础字段
2. 新增 CiA402Axis 或 CiA402StateMachine
3. ServoProfile 解析 status_word，生成 control_word
4. 增加 WKC 连续异常检测
5. 明确 start 进入 OP 后的最小控制流程
6. 开始搭建 SlaveRuntimeState / SlaveHealth（为单站 fault reset 和后续维护做准备）
7. 完善 PDO 阶段的日志与周期统计
```

这一阶段完成后，工程才真正具备“可控伺服主站”的基础。

---

### 阶段 3：补单站维护边界

```text
1. SlaveNode 增加 expected_state / actual_state / health
2. ProcessDataEngine 按节点检查状态
3. Activity 增加 scope / risk / affected slaves
4. 支持单站 reset / recovery
5. 再实现 FOE 升级
```

注意：**FOE 不建议在这些边界之前实现**，否则容易和全局状态检查冲突。

---

### 阶段 4：工程化增强

```text
1. 配置外部化
2. CommandRegistry
3. CyclicRunner 接入实时周期
4. SOEM 单线程访问模型或资源锁模型
5. 异步 Activity / 取消 / 进度
6. 更完整故障恢复策略
```

---

## 6. 当前架构维护红线

1. 不要让 `EcatController` 重新变胖。它只应负责主站生命周期状态迁移。
2. 不要让 `ControllerState` 表达所有过程。`kDiagnosing`、`kFoEUpgrading`、`kResettingSlave` 等应属于 Activity 或内部子步骤。
3. 不要把 CiA402 状态机塞进 `SlaveNode`。
4. 不要长期硬编码设备配置，设备类型和现场参数变多后应配置外部化。

---

## 7. 一句话结论

当前必须立即解决的是：

```text
资源生命周期、Activity 互斥、命令参数异常、状态切换一致性、
诊断数据准确性、扫描后拓扑校验、日志与调试能力、周期状态读取冗余。
```

可以后续逐步演进的是：

```text
CiA402、单站维护、FOE、配置外部化、实时周期线程、异步 Activity、完整错误恢复。
```

当前框架方向是正确的，不建议推倒重来；但如果不先补齐这些基础边界，后续做 FOE、CiA402、热插拔、实时周期线程时，复杂度会集中压到 `EcatApplication`、`EcatController` 和 `SlaveNode`，从而拉低长期可维护性。
