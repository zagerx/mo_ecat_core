# 主站状态机分层演化方案

## 1. 背景

当前 `EcatController` 使用一组扁平状态描述主站生命周期：

```cpp
enum class ControllerState {
    kUninitialized,
    kAdapterReady,
    kScanned,
    kMaintenance,
    kReadyToRun,
    kOperational,
    kFault,
    kEmergencyStop
};
```

这套状态在早期验证阶段是可用的。它把主站从“未初始化”推进到“PREOP 维护”和“OP 运行”的关键路径表达出来了，也避免把 `PDO 配置`、`SAFEOP`、`DC 配置` 这些一次性步骤全部暴露成公开状态。

但随着系统继续发展，这套状态命名和结构开始暴露两个问题：

1. 状态和动作不容易区分。
   `kScanned`、`kReadyToRun` 看起来像动作完成结果，而不是一个清晰的系统状态。使用者需要结合上下文才能知道它们属于状态，而不是动作。

2. 不同抽象层级混在同一个枚举里。
   `kAdapterReady`、`kScanned`、`kMaintenance`、`kReadyToRun` 本质上都是“准备态”内部的进度；而 `kOperational`、`kFault`、`kEmergencyStop` 是系统级大状态。它们放在同一层，会让状态机在后续扩展时越来越难读。

目前主站已经完成了 PREOP 维护路径验证，并解决了 EEPROM/SII 导致的 INIT -> PREOP 问题。下一阶段会逐步进入 PDO、SAFEOP、OP、故障恢复、单站维护等能力建设，因此状态机需要向更清晰、更可扩展的结构演化。

## 2. 目标

状态机演化的目标不是增加更多状态，而是把状态分层，让每一层只回答一个问题。

主状态回答：

```text
主站现在处于哪个系统阶段？
```

过渡阶段回答：

```text
主站是在进入这个阶段、稳定停留在这个阶段，还是正在退出这个阶段？
```

子阶段回答：

```text
在某个主状态内部，EtherCAT 具体走到了哪一步？
```

Activity/任务层回答：

```text
当前正在执行什么维护、诊断、升级或恢复任务？
```

这样可以避免把所有细节都塞进一个 `ControllerState` 枚举，也能让 UI、日志和上层控制逻辑更容易理解。

## 3. 当前问题分析

### 3.1 `AdapterReady / Scanned / Maintenance / ReadyToRun` 都属于准备态

从系统语义看，当前状态可以归类为：

| 当前状态 | 实际含义 | 所属大状态 |
|---|---|---|
| `kUninitialized` | 主站未完成基础初始化 | 初始化态 |
| `kAdapterReady` | SOEM 已绑定网卡 | 准备态 |
| `kScanned` | 已发现拓扑并读取从站信息 | 准备态 |
| `kMaintenance` | 所有从站处于 PREOP，可做 SDO 维护 | 准备态 |
| `kReadyToRun` | 已完成 PDO/IOmap/SAFEOP/DC，等待进入 OP | 准备态 |
| `kOperational` | 所有从站进入 OP，PDO 周期运行 | 运行态 |
| `kFault` | 普通故障 | 故障态 |
| `kEmergencyStop` | 安全类严重事件 | 紧急停止态 |

因此当前扁平枚举混合了两类概念：

```text
系统级主状态
准备态内部阶段
```

这会导致命名越来越难取。比如 `kReadyToRun` 既想表达“准备完成”，又隐含“从站已经 SAFEOP”，但名字本身并没有说明 EtherCAT 状态。

### 3.2 进入和退出过程没有显式表达

目前的状态只有稳定结果。例如：

```text
kMaintenance
kOperational
```

但从控制系统角度看，进入某个状态和稳定停留在某个状态不是一回事。

例如进入运行态时，内部会经历：

```text
PDO 配置
IOmap 配置
SAFEOP 请求
DC 配置
OP 请求
```

这段过程既不是稳定的准备态，也不是稳定的运行态。当前代码用函数调用表达了这个过程，但状态模型没有明确描述“正在进入运行态”。

退出也是同样问题。例如从 `kReadyToRun` 回到 `kMaintenance`，当前实现只修改主站状态，没有确保从站真实退回 PREOP。这个问题本质上也是因为“退出准备子阶段”的语义没有被状态机明确表达。

### 3.3 后续功能会放大扁平状态机的问题

后续可能加入：

```text
EEPROM 烧录
FoE 固件升级
单站诊断
单站重连
拓扑变化检测
运行中降级
故障恢复
急停恢复
```

如果每类能力都新增平级 `ControllerState`，状态机会快速膨胀。例如：

```text
kEepromUpdating
kFoEUpgrading
kRecovering
kDiagnostics
kReconnecting
```

这会让正常生命周期和维护任务纠缠在一起。更合理的方式是：主状态保持稳定，具体任务由 Activity/任务层表达。

## 4. 目标状态模型

建议把主站状态拆成三层：

```text
MasterMode        五大主状态
TransitionStage   进入中 / 稳定 / 退出中
PrepareStage      准备态内部 EtherCAT 阶段
```

### 4.1 五大主状态

```cpp
enum class MasterMode {
    kInit,              // 初始化态
    kPrepare,           // 准备态
    kRun,               // 运行态
    kFault,             // 故障态
    kEmergencyStop      // 紧急停止态
};
```

中文语义如下：

| 主状态 | 含义 |
|---|---|
| 初始化态 | 主站尚未完成基础连接，或者正在建立基础连接 |
| 准备态 | 主站已具备维护或运行准备能力，内部可处于网卡就绪、拓扑已发现、PREOP、SAFEOP 等子阶段 |
| 运行态 | 所有目标从站处于 OP，PDO 周期运行 |
| 故障态 | 普通故障，允许人工处理或受控恢复 |
| 紧急停止态 | 安全类严重事件，禁止自动恢复 |

### 4.2 进入/稳定/退出阶段

```cpp
enum class TransitionStage {
    kEntering,  // 正在进入某个主状态
    kStable,    // 已稳定处于某个主状态
    kExiting    // 正在退出某个主状态
};
```

这样可以表达：

```text
进入准备态
准备态
退出准备态

进入运行态
运行态
退出运行态

进入故障态
故障态
退出故障态
```

这比新增大量平级状态更清楚，也更容易给 UI 和日志展示。

### 4.3 准备态内部阶段

准备态内部需要保留 EtherCAT 细节，因为调试和运行控制仍然需要知道主站具体走到哪一步。

```cpp
enum class PrepareStage {
    kNone,
    kAdapterReady,          // 网卡就绪
    kTopologyDiscovered,    // 拓扑已发现
    kPreOpMaintenance,      // PREOP 维护
    kSafeOpReady            // SAFEOP 就绪
};
```

对应当前状态的映射为：

| 当前状态 | 新主状态 | 新过渡阶段 | 准备子阶段 |
|---|---|---|---|
| `kUninitialized` | `kInit` | `kStable` | `kNone` |
| `kAdapterReady` | `kPrepare` | `kStable` | `kAdapterReady` |
| `kScanned` | `kPrepare` | `kStable` | `kTopologyDiscovered` |
| `kMaintenance` | `kPrepare` | `kStable` | `kPreOpMaintenance` |
| `kReadyToRun` | `kPrepare` | `kStable` | `kSafeOpReady` |
| `kOperational` | `kRun` | `kStable` | `kNone` |
| `kFault` | `kFault` | `kStable` | `kNone` |
| `kEmergencyStop` | `kEmergencyStop` | `kStable` | `kNone` |

### 4.4 组合后的运行时状态

```cpp
struct MasterRuntimeState {
    MasterMode mode;
    TransitionStage transition;
    PrepareStage prepare_stage;
};
```

示例：

```text
{ kPrepare, kStable,   kPreOpMaintenance }  准备态 / 稳定 / PREOP维护
{ kRun,     kEntering, kNone }              进入运行态
{ kRun,     kStable,   kNone }              运行态
{ kRun,     kExiting,  kNone }              退出运行态
{ kFault,   kStable,   kNone }              故障态
```

## 5. 状态迁移语义

### 5.1 正常启动

```text
初始化态
  --进入准备态-->
准备态.网卡就绪
  --发现拓扑-->
准备态.拓扑已发现
  --进入PREOP-->
准备态.PREOP维护
  --准备SAFEOP-->
准备态.SAFEOP就绪
  --进入运行态-->
运行态
```

### 5.2 从运行态退回维护

```text
运行态
  --退出运行态-->
准备态.PREOP维护
```

内部应执行：

```text
OP -> SAFEOP -> PREOP
```

如果当前只是 `SAFEOP 就绪`，也必须真实请求：

```text
SAFEOP -> PREOP
```

不能只改主站内存状态。

### 5.3 停止关闭

```text
任意非初始化态
  --退出当前态-->
初始化态
```

内部按当前实际 EtherCAT 状态回滚：

```text
OP -> SAFEOP -> PREOP -> INIT -> Close
SAFEOP -> PREOP -> INIT -> Close
PREOP -> INIT -> Close
AdapterReady/TopologyDiscovered -> Close
```

### 5.4 故障和急停

普通故障：

```text
任意状态 -> 进入故障态 -> 故障态
```

紧急停止：

```text
任意状态 -> 进入紧急停止态 -> 紧急停止态
```

两者的区别：

| 状态 | 恢复策略 |
|---|---|
| 故障态 | 可允许人工确认后受控恢复 |
| 紧急停止态 | 禁止自动恢复，必须人工处理并重新建立安全条件 |

## 6. API 演化建议

### 6.1 第一阶段保留旧接口

短期不直接删除现有接口：

```cpp
bool InitializeAdapter(const EcMasterConfig &config);
bool Scan();
bool EnterMaintenance();
bool PrepareRun();
bool StartOperation();
bool BackToMaintenance();
void Stop();
```

新增只读状态接口：

```cpp
MasterRuntimeState GetRuntimeState() const;
```

旧接口继续工作，新接口用于 UI、日志和后续上层逻辑。

### 6.2 第二阶段引入更直观的新接口

后续可以逐步增加：

```cpp
bool EnterPrepare(const EcMasterConfig &config);
bool PrepareSafeOp();
bool EnterRun();
bool ExitRunToPrepare();
bool Shutdown();
void RaiseFault(const std::string &reason);
void TriggerEmergencyStop(const std::string &reason);
```

语义对应：

| 新接口 | 作用 |
|---|---|
| `EnterPrepare()` | 从初始化态进入准备态，内部完成网卡初始化、扫描、进入 PREOP |
| `PrepareSafeOp()` | 从 PREOP 维护推进到 SAFEOP 就绪 |
| `EnterRun()` | 从 SAFEOP 就绪进入 OP 运行 |
| `ExitRunToPrepare()` | 从运行态退回 PREOP 维护 |
| `Shutdown()` | 受控停机并关闭 SOEM |
| `RaiseFault()` | 进入普通故障态 |
| `TriggerEmergencyStop()` | 进入紧急停止态 |

### 6.3 第三阶段清理旧命名

当 UI、Activity、ProcessDataEngine 都切换到新状态模型后，再考虑移除或弱化：

```text
kAdapterReady
kScanned
kMaintenance
kReadyToRun
```

它们可以保留为 `PrepareStage`，不再作为主状态暴露。

## 7. Activity 与状态机边界

状态机不应该因为每一种维护任务而新增主状态。

例如以下任务：

```text
SDO 参数读写
PDO 映射检查
状态巡检
EEPROM 烧录
FoE 固件升级
单站诊断
单站恢复
```

应由 Activity/任务层表达。

推荐边界：

| 层级 | 职责 |
|---|---|
| `MasterMode` | 主站所处大阶段 |
| `TransitionStage` | 是否正在进入/退出大阶段 |
| `PrepareStage` | 准备态内部 EtherCAT 进度 |
| `Activity` | 当前执行的维护、诊断、升级或恢复任务 |
| `EcMaster` | SOEM 状态请求、PDO/DC 配置、SDO/EEPROM 访问 |

这样新增任务时，不需要扩展主状态枚举。

## 8. 推荐实施计划

### 阶段一：修正当前状态机语义

目标：不改变架构，先修现有问题。

任务：

1. 修复 `ReadyToRun -> Maintenance` 回退。
   当前实现只是设置 `state_ = kMaintenance`，应改为请求从站回到 PREOP 并确认。

2. 更新日志和注释。
   明确：

   ```text
   kMaintenance = PREOP维护
   kReadyToRun  = SAFEOP就绪
   ```

3. 更新旧文档中 `TransitionTo()` 的描述。
   当前代码已经使用 `Dispatch(MasterAction)`，文档应同步，避免误导。

验收标准：

```text
BackToMaintenance() 后所有从站真实处于 PREOP。
现有 InitializeAdapter -> Scan -> EnterMaintenance 流程不受影响。
```

### 阶段二：新增分层状态只读模型

目标：引入新状态表达，但不破坏旧接口。

任务：

1. 新增：

   ```cpp
   enum class MasterMode;
   enum class TransitionStage;
   enum class PrepareStage;
   struct MasterRuntimeState;
   ```

2. 新增：

   ```cpp
   MasterRuntimeState GetRuntimeState() const;
   ```

3. 将当前 `ControllerState` 映射到新状态模型。

4. 增加状态合法性校验函数，显式限制允许出现的组合。

5. UI 和日志优先显示新状态中文名。

验收标准：

```text
UI 可显示：初始化态 / 准备态.PREOP维护 / 准备态.SAFEOP就绪 / 运行态 / 故障态 / 紧急停止态。
旧接口和旧状态判断仍能编译并运行。
```

阶段二应同时定义合法组合，避免 UI 或日志出现没有意义的状态，例如：

```text
进入运行态 / SAFEOP就绪
故障态 / PREOP维护
紧急停止态 / 拓扑已发现
```

建议只允许以下组合起步：

| MasterMode | TransitionStage | PrepareStage |
|---|---|---|
| `kInit` | `kEntering` / `kStable` / `kExiting` | `kNone` |
| `kPrepare` | `kEntering` / `kStable` / `kExiting` | `kAdapterReady` / `kTopologyDiscovered` / `kPreOpMaintenance` / `kSafeOpReady` |
| `kRun` | `kEntering` / `kStable` / `kExiting` | `kNone` |
| `kFault` | `kEntering` / `kStable` / `kExiting` | `kNone` |
| `kEmergencyStop` | `kEntering` / `kStable` / `kExiting` | `kNone` |

### 阶段三：把进入/退出过程显式化

目标：让状态机能表达“进入中”和“退出中”。

任务：

1. 在复合转换开始时设置：

   ```text
   transition = kEntering
   ```

2. 转换成功后设置：

   ```text
   transition = kStable
   ```

3. 回退或关闭时设置：

   ```text
   transition = kExiting
   ```

4. 转换失败时进入 `kFault` 或 `kEmergencyStop`。

5. 过渡阶段禁止启动新的 Activity。
   `Activity::CanStart()` 应要求：

   ```text
   transition == kStable
   ```

6. 为进入/退出过程设计超时策略。
   如果状态长时间停在 `kEntering` 或 `kExiting`，应进入 `kFault` 并记录具体步骤。

验收标准：

```text
日志能看到“进入运行态 -> 运行态”或“退出运行态 -> 准备态.PREOP维护”。
失败时能明确知道失败发生在进入/退出哪个主状态的过程中。
```

### 阶段四：逐步替换旧接口

目标：让外部调用更贴近五大态。

任务：

1. 引入新接口：

   ```cpp
   EnterPrepare()
   PrepareSafeOp()
   EnterRun()
   ExitRunToPrepare()
   Shutdown()
   ```

2. 旧接口保留一段时间，内部转调新接口。

3. 更新 CLI、GUI、core_bridge 调用。

验收标准：

```text
上层代码不再直接依赖 kScanned / kMaintenance / kReadyToRun 这些旧主状态。
准备态内部阶段由 PrepareStage 表达。
```

### 阶段五：清理旧扁平状态

目标：完成分层状态机迁移。

任务：

1. 删除或内部化旧 `ControllerState`。
2. Activity 权限判断切换到 `MasterRuntimeState`。
3. ProcessDataEngine 只根据 `MasterMode::kRun` 执行 PDO 周期。
4. 维护类 Activity 根据 `MasterMode::kPrepare` 和具体 `PrepareStage` 判断是否可执行。

验收标准：

```text
主状态只剩五大态。
准备态内部细节由子阶段表达。
维护任务不会新增主状态。
```

## 9. 风险与注意事项

1. 不要一次性重构所有代码。
   当前 PREOP 路径刚验证通过，应避免在同一轮改动中同时修改状态机、UI、Activity 和 PDO 运行路径。

2. 进入/退出阶段不应长期停留。
   如果某个进入或退出过程卡住，应超时进入故障态，而不是无限停在 `kEntering` 或 `kExiting`。

3. 主状态不要表达具体任务。
   例如 EEPROM 烧录和 FoE 升级不应成为主状态，应作为 Activity。

4. UI 显示应优先使用中文语义。
   例如：

   ```text
   准备态 / PREOP维护
   准备态 / SAFEOP就绪
   运行态
   故障态
   紧急停止态
   ```

5. 底层 EtherCAT 状态仍然必须可见。
   主状态是业务抽象，但调试时仍需要显示每个从站的 INIT/PREOP/SAFEOP/OP 和 AL status code。

6. `PrepareStage` 不能替代真实从站状态。
   `PrepareStage` 是控制器内部认知，真实从站状态仍要通过总线读取。若巡检发现控制器认为处于 `PREOP维护`，但从站实际掉回 INIT 或进入错误状态，应进入 `kFault` 或执行受控恢复。

7. 分层状态不应直接依赖 SOEM 常量。
   `MasterMode` / `PrepareStage` 是业务抽象，不应直接绑定 `EC_STATE_*`。协议栈相关状态应封装在 `EcMaster` 内部，方便未来评估 SOEM -> IgH 或其他协议栈替换。

## 10. Activity 权限矩阵建议

分层状态落地后，Activity 不应只看旧的 `ControllerState`，而应基于 `MasterRuntimeState` 判断是否可以启动。

初始权限矩阵建议如下：

| Activity | 允许条件 | 说明 |
|---|---|---|
| `StateInspectionActivity` | `transition == kStable`，且 `mode == kPrepare` 或 `mode == kRun` | 巡检可在准备态和运行态执行 |
| `SdoDiagnosticsActivity` | `mode == kPrepare && prepare_stage == kPreOpMaintenance && transition == kStable` | 优先限定 PREOP 维护阶段 |
| `SdoParameterActivity` | `mode == kPrepare && prepare_stage == kPreOpMaintenance && transition == kStable` | 参数写入初期保持保守 |
| `PdoMappingActivity` | `mode == kPrepare && prepare_stage == kPreOpMaintenance && transition == kStable` | PDO 映射检查在 PREOP 最清晰 |
| `EepromUpdateActivity` | `mode == kPrepare && prepare_stage == kPreOpMaintenance && transition == kStable` | EEPROM 写入后需要冷启动或重新加载 |
| `FoEUpgradeActivity` | `mode == kPrepare && prepare_stage == kPreOpMaintenance && transition == kStable` | 初期不允许 OP 下升级 |

后续如果需要支持运行态下的在线诊断，可逐项放宽，而不是让所有 Activity 默认允许在 `kRun` 下启动。

## 11. 状态同步与巡检策略

分层状态机需要区分：

```text
控制器认为的状态
从站总线上真实状态
```

建议保留并强化 `ProcessDataEngine::CheckSlaveStates()` / `StateInspectionActivity` 的职责：

1. 在 `准备态 / PREOP维护` 下，定期确认目标从站仍处于 PREOP。
2. 在 `准备态 / SAFEOP就绪` 下，定期确认目标从站仍处于 SAFEOP。
3. 在 `运行态` 下，定期确认目标从站仍处于 OP。
4. 任一从站状态偏离期望时，记录从站 id、名称、实际 EtherCAT 状态、AL status code，并进入 `kFault`。

这样可以避免内部 `PrepareStage` 与真实总线状态长期不一致。

## 12. 故障与急停恢复策略

故障态和紧急停止态需要不同的恢复语义。

### 12.1 故障态

故障态用于普通可诊断错误，例如：

```text
从站掉出 PREOP / SAFEOP / OP
PDO 配置失败
DC 配置失败
SDO 维护任务失败且策略要求进入故障
```

初期建议只允许：

```text
故障态 -> Stop/Shutdown -> 初始化态
```

后续可增加：

```cpp
bool RecoverFromFault();
```

恢复目标建议优先回到：

```text
准备态 / PREOP维护
```

而不是直接回运行态。

### 12.2 紧急停止态

紧急停止态用于安全类严重事件，例如：

```text
外部急停输入
安全 IO 触发
严重通信异常
上层安全策略要求立即停止
```

初期建议只允许：

```text
紧急停止态 -> Stop/Shutdown -> 初始化态
```

不建议提供自动恢复。后续若要增加：

```cpp
bool ResetEmergencyStop();
```

也必须要求外部安全条件重新满足，并由人工确认。

## 13. 结论

主站状态机应该朝分层状态机演化。

推荐目标是：

```text
五大主状态：
初始化态 / 准备态 / 运行态 / 故障态 / 紧急停止态

三类过渡阶段：
进入中 / 稳定 / 退出中

准备态内部阶段：
网卡就绪 / 拓扑已发现 / PREOP维护 / SAFEOP就绪
```

这套模型能解决当前命名不直观、状态和动作混淆、准备流程细节与系统主状态混杂的问题，也能支撑后续 EEPROM、FoE、单站维护、故障恢复和急停处理等能力扩展。

实施上建议循序渐进：

```text
先修现有回退 bug
再引入只读分层状态
再显式表达进入/退出过程
最后替换旧接口和旧状态
```

这样既能保留当前已经验证通过的 PREOP 能力，又能让状态机架构逐步演化到更清晰、更适合长期维护的形态。
