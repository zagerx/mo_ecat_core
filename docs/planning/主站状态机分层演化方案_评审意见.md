# 《主站状态机分层演化方案》评审意见

## 1. 总体评价

**方向正确，建议按阶段实施。**

当前 `EcatController` 使用扁平的 `ControllerState` 描述主站生命周期，在 PREOP 验证阶段是够用的，但确实存在以下问题：

- 状态和动作命名混淆（`kScanned`、`kReadyToRun` 更像动作结果）；
- 准备态内部阶段与系统级大状态混在同一层；
- 进入/退出过程没有显式表达，导致 `BackToMaintenance()` 存在只改内存状态的 bug；
- 后续 EEPROM、FoE、单站维护、故障恢复等能力难以自然扩展。

方案提出将状态拆分为 `MasterMode`（五大主状态）、`TransitionStage`（进入/稳定/退出）、`PrepareStage`（准备态内部阶段）三层，并保留 Activity/任务层表达具体维护任务，这一分层思路清晰、可扩展，符合长期演进方向。

## 2. 方案优点

1. **分层语义清晰**
   - `MasterMode` 回答"主站在哪个大阶段"；
   - `TransitionStage` 回答"是否正在进入或退出"；
   - `PrepareStage` 回答"准备态内部 EtherCAT 走到哪一步"；
   - `Activity` 回答"当前正在执行什么任务"。
   每层只回答一个问题，避免了扁平枚举的膨胀。

2. **兼容现有接口**
   方案明确第一阶段保留旧接口（`InitializeAdapter`、`Scan`、`EnterMaintenance` 等），仅新增 `GetRuntimeState()` 只读接口，过渡期风险低。

3. **实施路径合理**
   五个阶段从"修复当前 bug"到"引入分层状态"再到"显式表达过渡阶段"，最后"替换旧接口并清理"，节奏稳健，符合"小步快跑"原则。

4. **Activity 与状态机边界明确**
   方案强调 EEPROM、FoE、单站诊断等不应新增主状态，应由 Activity 表达，这与当前 `ActivityScheduler` 的设计一致，避免了状态机被维护任务污染。

5. **风险意识到位**
   方案明确指出：
   - 不要一次性重构所有代码；
   - 进入/退出阶段不应长期停留，需超时进入故障态；
   - UI 显示应优先使用中文语义；
   - 底层 EtherCAT 状态仍必须可见。
   这些约束能够有效控制实施风险。

## 3. 发现的问题与风险

### 3.1 当前代码已存在一个明确 bug：BackToMaintenance 从 kReadyToRun 回退只改内存状态

在 `mo_ecat_core/src/ec_controller/ec_controller.cpp` 的 `DoAction()` 中：

```cpp
case MasterAction::kBackToMaintenance:
    if (state_ == ControllerState::kReadyToRun) {
        state_ = ControllerState::kMaintenance;   // 仅改内存状态
        LOG_INFO << "EcatController -> Maintenance";
        return true;
    }
```

`kReadyToRun` 对应从站实际处于 SAFEOP，但此处直接设置状态为 `kMaintenance`（语义上是 PREOP 维护），并未请求从站回到 PREOP 并确认。

**建议**：方案阶段一任务 1 应作为最高优先级修复，验收标准明确为"所有从站真实处于 PREOP"。

### 3.2 当前代码尚未开始演化，方案实施需要从头引入三层状态

调研确认当前代码中完全不存在 `MasterMode`、`TransitionStage`、`PrepareStage` 和 `MasterRuntimeState`。所有内部模块（`ProcessDataEngine`、Activity、`MoEcatMaster` 映射）仍依赖扁平 `ControllerState`。

**风险**：虽然方案分五阶段，但阶段二引入分层状态后，会有一小段时期内旧状态和新状态并存，需要维护两套映射，容易产生不一致。建议在阶段二就定义清晰的映射函数，并尽早让 `ProcessDataEngine` 和 Activity 切换到基于 `MasterRuntimeState` 的判断。

### 3.3 旧文档与代码不一致，可能误导后续开发

`mo_ecat_core/docs/others/状态机说明文档.md` 中仍描述为 `TransitionTo(ControllerState target)`，但实际代码已改为 `Dispatch(MasterAction action)`。此外，文档中 `kError` 已改为代码中的 `kFault`。

**建议**：在阶段一或阶段二同步更新 `状态机说明文档.md`、`主站状态设计方案.md` 等相关文档，避免新旧开发者被旧文档误导。

### 3.4 UI 当前显示英文枚举名，与方案中文语义要求不符

`app/core_bridge/ecat_worker.cpp` 将 `mo_ecat::MasterState` 一一映射到 `UiMasterState`，`MainWindow::OnMasterStateChanged()` 直接显示英文状态名（如 `ReadyToRun`）。

**建议**：阶段二引入 `MasterRuntimeState` 后，UI 应优先显示中文语义，例如：

```text
初始化态
准备态 / 网卡就绪
准备态 / 拓扑已发现
准备态 / PREOP维护
准备态 / SAFEOP就绪
运行态
故障态
紧急停止态
```

可考虑在 `core_bridge` 或 GUI 层增加 `ToDisplayString(MasterRuntimeState)` 转换函数。

### 3.5 过渡阶段（kEntering/kExiting）的实现需要小心设计

方案提出用 `TransitionStage` 表达进入/退出过程，这对 UI 和日志很有价值，但实现时容易出现以下问题：

- **状态组合爆炸**：`MasterMode × TransitionStage × PrepareStage` 的组合需要严格限制合法组合，避免 UI 出现"进入运行态 / SAFEOP就绪"这类无意义状态。
- **长时间停留风险**：如果某个进入/退出步骤阻塞，状态机会停在 `kEntering` 或 `kExiting`。必须设计超时机制，否则 UI 会显示"进入运行态"卡住。
- **与 Activity 的并发**：如果在 `kEntering` 阶段允许 Activity 启动，可能导致状态混乱。建议明确 `CanStart()` 在过渡阶段返回 false。

**建议**：在 `MasterRuntimeState` 的合法性校验函数中，显式枚举允许的组合；并在转换调度器中统一处理超时进入 `kFault` 的逻辑。

### 3.6 `PrepareStage` 与从站真实 EtherCAT 状态可能不一致

`PrepareStage` 是控制器内部阶段，但从站真实状态可能因为外部原因（如线缆抖动、从站自身故障）与控制器状态不同步。例如控制器认为自己在 `kPreOpMaintenance`，但从站可能已经掉回 INIT。

**建议**：方案应补充"状态同步/巡检"机制。当前 `StateInspectionActivity` 已具备读取从站真实状态的能力，可考虑定期触发巡检，若发现不一致则进入 `kFault` 或更新 `PrepareStage`。

### 3.7 故障态与紧急停止态的恢复策略需要更明确

方案提到：

> 故障态：可允许人工确认后受控恢复；
> 紧急停止态：禁止自动恢复，必须人工处理并重新建立安全条件。

但方案未详细说明：
- 恢复的入口接口是什么（例如 `RecoverFromFault()`、`ResetEmergencyStop()`）？
- 从 `kFault` 可以恢复到哪个状态（`kPrepare` / `kPreOpMaintenance` 还是 `kUninitialized`）？
- 从 `kEmergencyStop` 是否只能通过 `Stop()` 回到 `kUninitialized`？

**建议**：在阶段三或阶段四中补充恢复状态图和对应接口，避免实现时各模块对恢复语义理解不一致。

### 3.8 与 SOEM/IgH 替换的关联性

此前已评估过将协议栈从 SOEM 替换为 IgH 的可行性。状态机分层演化与该替换工作有一定关联：

- 若计划未来替换协议栈，建议在阶段二就将 `ControllerState` 中对 SOEM 常量（`EC_STATE_*`）的依赖解耦，避免状态机与 SOEM 强绑定；
- `MasterMode` / `PrepareStage` 是业务抽象，不应直接映射 SOEM 状态常量，而应由 `EcMaster` 内部完成协议栈-specific 的状态到业务状态的转换。

**建议**：在阶段二定义项目自己的状态常量，逐步替换 `EC_STATE_*` 的直接使用。

## 4. 具体修改建议

### 4.1 阶段一：修正当前状态机语义（立即执行）

1. **修复 `BackToMaintenance()` 从 `kReadyToRun` 回退的 bug**
   - 从 `kReadyToRun` 回退时，必须调用 `master_.RequestPreOpState()` 并确认所有从站到达 PREOP；
   - 若失败，进入 `kFault`；
   - 验收标准：回退后读取所有从站状态，均为 PREOP。

2. **同步更新文档**
   - 更新 `状态机说明文档.md`，将 `TransitionTo` 改为 `Dispatch(MasterAction)`；
   - 将 `kError` 统一改为 `kFault`；
   - 在注释中明确 `kMaintenance = PREOP维护`、`kReadyToRun = SAFEOP就绪`。

### 4.2 阶段二：引入分层状态只读模型

1. **新增类型定义**
   - `enum class MasterMode { kInit, kPrepare, kRun, kFault, kEmergencyStop }`
   - `enum class TransitionStage { kEntering, kStable, kExiting }`
   - `enum class PrepareStage { kNone, kAdapterReady, kTopologyDiscovered, kPreOpMaintenance, kSafeOpReady }`
   - `struct MasterRuntimeState { MasterMode mode; TransitionStage transition; PrepareStage prepare_stage; }`

2. **新增映射函数**
   - `ControllerState -> MasterRuntimeState` 映射（一一对应，见方案 4.3 节表格）；
   - `MasterRuntimeState -> 中文显示字符串`；
   - 合法性校验函数，禁止无意义组合。

3. **新增公共接口**
   - `MasterRuntimeState GetRuntimeState() const;`

4. **UI 优先使用新状态**
   - `EcatWorker` 将 `MasterRuntimeState` 转换为 UI 显示字符串；
   - `MainWindow` 显示中文语义。

### 4.3 阶段三：显式表达进入/退出过程

1. **在复合转换前后设置 `TransitionStage`**
   - 转换开始：`transition = kEntering`（进入新主状态）或 `kExiting`（退出当前主状态）；
   - 转换成功：`transition = kStable`；
   - 转换失败：`transition = kStable` 且 `mode = kFault`。

2. **统一超时处理**
   - 每个进入/退出步骤设置超时；
   - 超时后进入 `kFault` 并记录原因。

3. **限制 Activity 在过渡阶段启动**
   - 所有 Activity 的 `CanStart()` 应要求 `transition == kStable`。

### 4.4 阶段四：引入新接口并替换旧调用

1. **新增接口**
   - `bool EnterPrepare(const EcMasterConfig &config);`
   - `bool PrepareSafeOp();`
   - `bool EnterRun();`
   - `bool ExitRunToPrepare();`
   - `bool Shutdown();`
   - `void RaiseFault(const std::string &reason);`
   - `void TriggerEmergencyStop(const std::string &reason);`

2. **旧接口内部转调新接口**
   - 保持 `InitializeAdapter()`、`Scan()`、`EnterMaintenance()` 等短期可用；
   - 逐步在 `CLI`、`GUI`、`core_bridge` 中切换到新接口。

### 4.5 阶段五：清理旧扁平状态

1. **内部化或删除 `ControllerState`**
   - 仅在 `EcatController` 内部保留，或完全用 `MasterRuntimeState` 替代；

2. **`ProcessDataEngine` 切换到 `MasterMode::kRun`**
   - 当前判断 `controller_.GetState() != ControllerState::kOperational`；
   - 改为判断 `runtime_state.mode != MasterMode::kRun || runtime_state.transition != TransitionStage::kStable`。

3. **Activity 切换到 `MasterRuntimeState`**
   - `CanStart()` 入参从 `ControllerState` 改为 `const MasterRuntimeState &`；
   - 例如 SDO 诊断可在 `MasterMode::kPrepare && PrepareStage::kPreOpMaintenance && TransitionStage::kStable` 下执行。

## 5. 实施优先级建议

| 优先级 | 任务 | 原因 |
|--------|------|------|
| **P0** | 修复 `BackToMaintenance()` 从 `kReadyToRun` 回退的 bug | 当前已存在实现缺陷，影响状态机正确性 |
| **P0** | 同步更新 `状态机说明文档.md` | 避免新旧开发者被旧文档误导 |
| **P1** | 引入 `MasterMode` / `TransitionStage` / `PrepareStage` 和 `MasterRuntimeState` | 为后续分层状态机打好基础 |
| **P1** | 新增 `GetRuntimeState()` 并在 UI 显示中文语义 | 立即提升 UI 可理解性 |
| **P2** | 在 `ProcessDataEngine` 和 Activity 中切换到新状态模型 | 逐步减少旧 `ControllerState` 依赖 |
| **P2** | 显式表达进入/退出阶段并增加超时处理 | 提升日志、UI、故障定位能力 |
| **P3** | 引入新接口 `EnterPrepare` / `PrepareSafeOp` / `EnterRun` 等 | 长期目标，待新状态模型稳定后再做 |
| **P3** | 删除 `ControllerState` | 最后一步，确保上层完全迁移后再执行 |

## 6. 需要补充的内容

方案整体已经比较完整，但建议在正式实施前补充以下几点：

1. **状态转换时序图**
   - 画出 `kInit -> kPrepare(kAdapterReady) -> kPrepare(kTopologyDiscovered) -> kPrepare(kPreOpMaintenance) -> kPrepare(kSafeOpReady) -> kRun -> kPrepare(kPreOpMaintenance)` 的完整时序，标注每个阶段对应的 SOEM API 调用。

2. **Activity 权限矩阵**
   - 明确每种 Activity 在 `MasterMode` / `PrepareStage` / `TransitionStage` 下的启动条件，例如：

     | Activity | 允许条件 |
     |----------|----------|
     | StateInspection | `kPrepare` 任意稳定子阶段，或 `kRun` 稳定 |
     | SdoDiagnostics | `kPrepare` 且 `kPreOpMaintenance` 稳定 |
     | PdoMapping | `kPrepare` 且 `kPreOpMaintenance` 稳定 |
     | SdoParameter | `kPrepare` 且 `kPreOpMaintenance` 稳定 |
     | EepromUpdate | `kPrepare` 且 `kPreOpMaintenance` 稳定 |

3. **故障/急停恢复状态图**
   - 明确从 `kFault` 和 `kEmergencyStop` 可以恢复到哪个状态，以及对应接口。

4. **与协议栈解耦的常量定义**
   - 定义项目自己的 EtherCAT 业务状态常量，替代直接使用 SOEM 的 `EC_STATE_*`，为将来可能的 SOEM -> IgH 替换做准备。

## 7. 结论

《主站状态机分层演化方案》是一份方向正确、结构清晰的演化方案，能够有效解决当前扁平状态机命名混淆、扩展性差、进入/退出过程不可见等问题。

**建议在近期按阶段实施，优先修复 `BackToMaintenance()` 的当前 bug，然后引入分层状态只读模型和中文 UI 显示，再逐步让内部模块切换到新状态模型。** 实施过程中需特别注意：

- 文档与代码同步；
- 过渡阶段超时和 Activity 并发控制；
- `PrepareStage` 与从站真实状态的同步；
- 故障/急停恢复策略的明确定义。

若能按此节奏推进，主站状态机将演化为一个清晰、可扩展、适合长期维护的架构。
