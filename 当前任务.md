# 当前任务

## 目标

在主站状态机框架已完成、从站能稳定进入 **PREOP**、**SDO 通信已验证**的基础上，进一步**扩展 Activity 框架**并**补强状态机健壮性**，为后续更多维护类 Activity（单站复位、FOE 升级等）和 PDO/OP 阶段集成打好基础。

**当前阶段仍不进入 OP。**

---

## 当前重点：扩展 Activity 框架与状态机健壮性

### 1. 实现 `SdoParameterActivity`

目的：验证 Activity 框架不仅能"读"，还能"写"。

- 通过 SDO 向指定从站写入一个参数（如 `0x607A:00` 目标位置，或通用 index/subindex/value）。
- 写完后可选回读校验。
- 失败策略根据重要性选择，默认 `kKeepControllerState` 或 `kEnterError`。
- 允许在 `kPreOp` 及以上状态启动。

### 2. 实现 `StateInspectionActivity`

目的：验证 Activity 可以读取非对象字典类状态信息。

- 读取单个从站的 AL status、AL status code、当前 EtherCAT 状态等。
- 可以通过 `SlaveNode::RefreshActualState()` 或 `EcMaster` 提供的接口获取。
- 失败策略为 `kKeepControllerState`（只读操作）。

### 3. 扩展 `CheckSlaveStates()` 到 PREOP/SAFEOP

目的：提升状态机在非 OP 状态下的异常感知能力。

- 当前只在 `kOperational` 检测从站是否掉出 OP。
- 扩展为在 `kPreOp` 检测是否掉出 PREOP，在 `kSafeOp` 检测是否掉出 SAFEOP。
- 检测到异常时调用 `EnterErrorState()`。

## 验收标准

1. `SdoParameterActivity` 和 `StateInspectionActivity` 实现完成，能通过 `controller.ExecuteActivity()` 执行。
2. `main.cpp` 中演示"诊断 → 参数写入 → 状态巡检"的完整维护流程（或至少能独立验证每个 Activity）。
3. `CheckSlaveStates()` 支持 `kPreOp` / `kSafeOp` / `kOperational` 三种状态的掉线检测。
4. 所有新增 Activity 都有清晰的 `CanStart()`、`GetFailurePolicy()` 和日志输出。
5. 编译通过，`ecat_scan` 仍能让从站进入 PREOP 并执行 SDO 诊断。
6. 不进入 SAFEOP/OP，不启动 PDO 周期线程。
