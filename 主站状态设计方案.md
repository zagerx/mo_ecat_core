# 主站状态设计方案

## 1. 背景

`EcatController` 当前维护一条线性状态机，用于描述 EtherCAT 主站从网卡绑定到 OP 运行的正常运行流程：

```
Uninitialized → InitDone → Scanned → PreOp → PdoConfigured → SafeOp → DcConfigured → Operational → Error
```

该设计能够满足当前阶段“初始化到 PREOP”以及后续 PDO 周期运行的需求，但无法自然支撑 FOE 升级、单站复位、在线诊断等维护类操作。

本文描述当前设计的缺陷，并给出推荐的主站状态设计方案。

---

## 2. 当前状态机的缺陷

### 2.1 主站状态与从站状态混为一谈

当前 `ControllerState` 描述的是“整站”状态。所有从站必须同时处于同一 EtherCAT 状态，主站才能进入对应状态。这种设计暗含假设：所有从站永远同进同退。

实际维护场景中，这一假设不成立。例如 FOE 升级单站时，目标从站需要进入 BOOTSTRAP，而其他从站可能仍应保持 OP。

### 2.2 线性状态机无法表达维护活动

FOE 升级、SDO 诊断、单站复位等操作不是主站生命周期中的“下一个状态”，而是**在某一状态下临时执行的横向活动**。把它们硬塞进线性状态机会导致：

- 状态枚举爆炸（如 `kFoEUpgrading`、`kDiagnostics` 等）
- 状态转换关系混乱
- 正常流程和维护流程相互耦合

### 2.3 缺少从站级独立控制

当前 `SlaveNode::RequestState()` 虽然可以请求单个从站状态，但 `EcatController` 的状态机假设所有从站同步。没有清晰的机制支持：

- 单个从站进入 BOOTSTRAP
- 升级完成后单个从站重新上线
- 其他从站在维护期间保持运行

### 2.4 PREOP 阶段无法检测从站掉线

当前 `CheckSlaveStates()` 只在 `kOperational` 时判断从站是否掉出 OP。在 PREOP 等中间状态下，从站掉线不会被检测到，因为状态检查逻辑与生命周期状态绑定，且没有独立的监控机制。

---

## 3. 推荐方案：主站生命周期状态 + 独立维护活动

推荐采用“**主站生命周期状态 + EcatActivity 独立维护活动**”的二维设计，而不是把维护操作塞进线性状态机。

### 3.1 核心思想

- **主站状态机（纵向）**：保持简单、清晰的线性生命周期，只描述主站正常运行阶段。
- **维护活动（横向）**：把 FOE 升级、SDO 诊断、单站复位等操作封装为独立的 `EcatActivity`，在主站某个状态下异步执行，不污染主站状态机。
- **从站独立状态**：每个从站可以独立进入 BOOTSTRAP、INIT、PREOP 等状态，不受其他从站束缚。

### 3.2 主站状态机保留线性流程

```cpp
enum class ControllerState {
    kUninitialized,
    kInitDone,
    kScanned,
    kPreOp,
    kPdoConfigured,
    kSafeOp,
    kDcConfigured,
    kOperational,
    kError
};
```

状态转换仍通过 `TransitionTo()` 统一入口管理。主站状态只回答一个问题：**主站当前处于正常生命周期的哪个阶段？**

### 3.3 引入 EcatActivity 抽象

```cpp
class EcatActivity {
public:
    virtual ~EcatActivity() = default;

    // 执行活动，返回是否成功
    virtual bool Execute() = 0;

    // 活动名称，用于日志
    virtual const char *GetName() const = 0;

    // 是否允许在给定主站状态下启动
    virtual bool CanStart(ControllerState state) const = 0;

    // 活动结束后期望的主站状态（通常与启动时相同）
    virtual ControllerState GetTargetState() const = 0;
};
```

典型活动：

```cpp
class FoEUpgradeActivity : public EcatActivity {
public:
    FoEUpgradeActivity(SlaveNode &node, const std::string &filename,
                       const std::vector<uint8_t> &firmware);
    bool Execute() override;

private:
    SlaveNode &node_;
    std::string filename_;
    std::vector<uint8_t> firmware_;
};
```

### 3.4 活动执行器

`EcatController` 增加一个活动执行器，用于在后台运行维护活动：

```cpp
class EcatController {
public:
    // 提交一个维护活动，在后台执行
    bool StartActivity(std::unique_ptr<EcatActivity> activity);

    // 查询是否有活动正在执行
    bool IsActivityRunning() const;

private:
    void ActivityLoop();

    std::unique_ptr<EcatActivity> current_activity_;
    std::thread activity_thread_;
    std::atomic<bool> activity_running_{false};
};
```

活动执行器保证：
- 同一时刻只能有一个活动运行
- 活动开始前检查 `CanStart(state)`
- 活动结束后主站回到原状态或目标状态

### 3.5 从站独立状态控制

`SlaveNode` 提供完整的单站状态控制能力：

```cpp
class SlaveNode {
public:
    // 请求任意 EtherCAT 状态，包括 BOOTSTRAP
    bool RequestState(uint16_t state);

    // FOE 写入（升级固件）
    bool FoEWrite(const std::string &filename, const std::vector<uint8_t> &data,
                  int timeout_us = 60000);

    // 等待从站重新上线（升级后复位）
    bool WaitForReboot(int timeout_us = 30000000);
};
```

### 3.6 活动与主站状态的协调规则

为了安全和简单，定义以下协调规则：

| 规则 | 说明 |
|---|---|
| **运行时活动互斥** | 一个活动运行期间，不允许启动另一个活动 |
| **状态前置检查** | 活动通过 `CanStart()` 声明允许启动的主站状态 |
| **禁止破坏主站状态** | 活动内部可以改变单个从站状态，但不能私自改变 `ControllerState` |
| **异常进 Error** | 活动失败时，可请求 `EcatController` 进入 `kError` |
| **Stop 等待活动** | `Stop()` 时需等待当前活动结束，再执行 `DoShutdown()` |

## 4. FOE 升级场景示例

假设当前主站处于 `kOperational`，4 个从站都在 OP。收到升级从站 2 的指令：

```cpp
auto *node = controller.GetSlaveNodeManager().GetNode(2);
auto activity = std::make_unique<FoEUpgradeActivity>(*node, "firmware.bin", firmware_data);
controller.StartActivity(std::move(activity));
```

`FoEUpgradeActivity::Execute()` 内部流程：

1. 保存从站 2 升级前的状态（OP）。
2. 请求从站 2 进入 SAFEOP。
3. 请求从站 2 进入 INIT。
4. 请求从站 2 进入 BOOTSTRAP。
5. 通过 FOE 写入固件文件。
6. 校验写入结果。
7. 发送复位命令，等待从站重新上线。
8. 重新配置该从站（PDO 映射等）。
9. 请求从站 2 回到 OP。
10. 其他 3 个从站全程保持 OP 不变。

整个过程中：
- 主站状态始终是 `kOperational`
- 只有从站 2 经历 OP → SAFEOP → INIT → BOOTSTRAP → INIT → PREOP → SAFEOP → OP
- 升级失败时进入 `kError`，由上层决定 Stop 或重试

## 5. PREOP 阶段从站掉线检测的增强

除了维护活动，还建议增强 `CheckSlaveStates()`：

```cpp
void EcatController::CheckSlaveStates()
{
    master_.CheckSlaveStates();

    switch (state_) {
    case ControllerState::kPreOp:
        if (!master_.CheckAllSlavesInState(EC_STATE_PRE_OP)) {
            EnterErrorState("Slave dropped out of PREOP");
        }
        break;
    case ControllerState::kSafeOp:
        if (!master_.CheckAllSlavesInState(EC_STATE_SAFE_OP)) {
            EnterErrorState("Slave dropped out of SAFEOP");
        }
        break;
    case ControllerState::kOperational:
        if (!master_.CheckAllSlavesInState(EC_STATE_OPERATIONAL)) {
            EnterErrorState("Slave dropped out of OPERATIONAL");
        }
        break;
    default:
        break;
    }
}
```

并由 `EcatController` 内部启动一个状态监控线程，周期调用 `CheckSlaveStates()`。这样 PREOP 阶段也能及时发现从站掉线。

## 6. 与当前代码的衔接

推荐按以下顺序落地：

1. **保持当前状态机不变**：`ControllerState`、`TransitionTo()`、`DoXxx()` 都保留。
2. **新增 `EcatActivity` 基类**：不改动现有主站状态逻辑。
3. **新增活动执行器**：`EcatController::StartActivity()`、`IsActivityRunning()`。
4. **增强 `SlaveNode` 单站状态控制**：确保 `RequestState()` 支持 BOOTSTRAP。
5. **实现 `FoEUpgradeActivity`**：作为第一个具体活动验证设计。
6. **增强 `CheckSlaveStates()` 并增加监控线程**：补齐 PREOP 掉线检测。

这样可以在不推翻现有状态机的前提下，逐步获得维护能力。

---

## 7. 为什么选这个方案，而不是其他方案

### 为什么不把维护状态塞进主状态机

如果把 `kFoEUpgrading`、`kDiagnostics` 等作为 `ControllerState` 的新值，主状态机会变成一张复杂的网，正常流程和维护流程相互纠缠。每新增一种维护操作，都要修改状态转换表和 `DoXxx()`，长期不可维护。

### 为什么不采用层次化状态机（HSM）

HSM 很适合复杂维护流程，但会显著增加代码复杂度。当前项目阶段还不确定维护流程的细节，过早引入 HSM 容易导致过度设计。Activity 方案可以在 HSM 变得必要时平滑迁移过去。

### Activity 方案的优势

| 方面 | 说明 |
|---|---|
| **侵入性小** | 主站状态机保持简单线性，新增维护活动不需要改状态转换表 |
| **扩展性好** | FOE 升级、SDO 诊断、单站复位各自一个 Activity 类 |
| **职责清晰** | 主站状态管生命周期，Activity 管维护操作，从站状态管单站行为 |
| **可测试** | 每个 Activity 可以独立单元测试 |
| **与现有架构兼容** | 不需要推翻 `SlaveNode`、`SlaveNodeManager` 等已有设计 |

---

## 8. 代价

- 需要新增 `EcatActivity` 基类、活动执行器、监控线程等基础设施。
- `SlaveNode` 需要补充 FOE 相关接口和更 robust 的单站状态控制。
- 活动和主站状态之间的并发协调需要仔细处理（如 `Stop()` 时等待活动结束）。

总体而言，这是当前阶段**复杂度与扩展性最平衡**的方案。
