# MoDriverPC_EtherCAT 架构说明文档

> 版本：v3.1（增加 switch-case 主循环与 6 状态生命周期）  
> 目标：基于 SOEM 构建面向多从站 EtherCAT 主站，支持命令驱动的主站生命周期与维护 Activity。  
> 运行平台：Linux（x86_64 / ARM64）  
> 编程语言：C++17  
> 通信协议：EtherCAT

---

## 1. 项目目标

本项目作为上位机 EtherCAT 主站，用于驱动支持 EtherCAT 的伺服驱动器、IO 模块等设备。当前阶段重点：

1. **命令驱动的主站生命周期管理**：通过命令逐步推进 `AdapterReady → Scanned → Maintenance`。
2. **维护 Activity 框架**：在 `Maintenance` 状态下执行 SDO 诊断、参数写入、状态巡检等维护操作。
3. 后续再扩展 PDO 周期运行、CiA402 状态机等电机控制逻辑。

---

## 2. 设计原则

| 原则 | 说明 |
|------|------|
| **分层解耦** | 硬件访问、总线通信、从站管理、电机语义、命令交互分层独立。 |
| **主状态单一维护** | `ControllerState` 是主站生命周期的唯一状态源，`EcatApplication` 基于它判断命令合法性。 |
| **状态只表示稳定工况** | PDO 配置、SafeOp、DC 同步等过渡步骤不暴露为 `ControllerState`。 |
| **Activity 与生命周期正交** | 维护操作封装为 `EcatActivity`，不改变 `ControllerState`。 |
| **一从站一节点** | 每个扫描到的从站对应一个 `SlaveNode`，独立管理通信状态、PDO 映射和数据。 |
| **电机语义与通信分离** | `ServoAxis` 只描述电机命令/反馈，不感知 EtherCAT/PDO/SDO。 |
| **命令通道可替换** | 当前用 `stdin` 交互，后续可替换为命名管道 / socket / RPC，不影响主状态逻辑。 |
| **异步日志** | 日志系统不阻塞业务线程。 |

## 4. 核心模块职责

### 4.1 EcMaster（EtherCAT 主站层）

`src/ec_master/ec_master.h`

职责：

- 初始化 SOEM、绑定网卡
- 扫描从站、配置 PDO 映射、DC 时钟
- 全局/单站 EtherCAT 状态切换
- PDO 收发
- SDO 读写

> **注意**：Init / PreOP / SafeOP / OP 四个 EtherCAT 状态由 `EcMaster` 管理，`SlaveNode` 可以请求单个从站的状态切换，但底层切换能力来自 `EcMaster`。

---

### 4.2 EcatController（总线应用控制层）

`src/ec_controller/ec_controller.h`

职责：

- 维护 `ControllerState` 线性生命周期状态机（6 个稳定状态）
- 提供细粒度生命周期接口：`InitializeAdapter()` / `Scan()` / `EnterMaintenance()` / `StartOperation()` / `Stop()`
- 提供 `Initialize()` 便捷接口，内部顺序执行 `InitializeAdapter → Scan → EnterMaintenance`
- 持有 `EcMaster` 和 `SlaveNodeManager`
- 提供 `ExecuteActivity()` 执行同步维护 Activity
- 提供 `RunOneCycle()` / `CheckSlaveStates()` 供 PDO 阶段使用

关键接口：

```cpp
class EcatController {
public:
    bool Initialize(const EcMasterConfig &config);           // 到 Maintenance 的便捷接口
    bool InitializeAdapter(const EcMasterConfig &config);    // 到 AdapterReady
    bool Scan();                                             // AdapterReady -> Scanned
    bool EnterMaintenance();                                 // Scanned -> Maintenance
    bool StartOperation();                                   // Maintenance -> Operational
    void Stop();                                             // 回滚到 Uninitialized

    ControllerState GetState() const;
    size_t GetSlaveCount() const;
    static const char *StateToString(ControllerState state);
    std::string GetLastError() const;

    void RunOneCycle();
    void CheckSlaveStates();

    bool ExecuteActivity(std::unique_ptr<EcatActivity> activity);
    SlaveNodeManager &GetSlaveNodeManager();
};
```

周期流程（PDO 阶段启用）：

```cpp
void EcatController::RunOneCycle() {
    slave_node_manager_.UpdateAllOutputs();  // 所有 SlaveNode：ServoAxis → IOmap
    master_.RunOneCycle();                    // PDO 收发
    slave_node_manager_.UpdateAllInputs();   // 所有 SlaveNode：IOmap → ServoAxis
}
```

### 4.3 EcatApplication（命令分发器 + 状态机主循环）

`src/app/ecat_application.h`

职责：

- 持有 `EcatController`
- 从 `stdin` 读取命令
- 使用 `Run()` 中的 switch-case 主循环，根据 `ControllerState` 分发命令和周期任务
- 不维护独立应用状态，直接基于 `ControllerState` 判断命令是否允许执行
- 把命令映射为两类操作：
  - **状态迁移**：`scan` / `config` / `start` / `stop`
  - **维护 Activity**：`diagnose` / `param` / `inspect`

当前支持的命令：

| 命令 | 允许的 ControllerState | 操作 |
|---|---|---|
| `scan` | `kAdapterReady` | `controller.Scan()` |
| `config` | `kScanned` | `controller.EnterMaintenance()` |
| `diagnose` | `kMaintenance` | 对所有 SlaveNode 执行 `SdoDiagnosticsActivity` |
| `param <idx> <sub> <val>` | `kMaintenance` | 对所有 SlaveNode 执行 `SdoParameterActivity` |
| `inspect` | `kMaintenance` | 对所有 SlaveNode 执行 `StateInspectionActivity` |
| `start` | `kMaintenance` | `controller.StartOperation()` |
| `stop` | 非 `kUninitialized` | `controller.Stop()` |
| `help` | 任意 | 打印帮助 |
| `exit` / `quit` | 任意 | 退出程序 |

> **命令通道可替换**：当前从 `stdin` 读取，后续只需替换命令读取逻辑，`EcatApplication::Run()` 中的状态分发不需要改动。

主循环结构：

```cpp
void EcatApplication::Run() {
    while (!shutdown_requested_) {
        bool has_command = ReadCommand(command, 100);

        switch (controller_.GetState()) {
        case kAdapterReady:  HandleAdapterReadyState(...);  break;
        case kScanned:       HandleScannedState(...);       break;
        case kMaintenance:   HandleMaintenanceState(...);   break;
        case kOperational:   HandleOperationalState(...);   break;
        case kError:         HandleErrorState(...);         break;
        }
    }
}
```

### 4.4 Activity（维护活动层）

`src/activity/activity.h`

`EcatActivity` 是维护活动基类，封装在主站某个状态下执行的操作：

```cpp
class EcatActivity {
public:
    virtual const char *GetName() const = 0;
    virtual bool CanStart(ControllerState state) const = 0;
    virtual bool Execute() = 0;
    virtual ActivityFailurePolicy GetFailurePolicy() const;
};
```

当前已实现：

| Activity | 作用 | 失败策略 |
|---|---|---|
| `SdoDiagnosticsActivity` | 读取 `0x1000`、`0x1018:01/02/03` 验证 SDO 通信 | `kKeepControllerState` |
| `SdoParameterActivity` | 通过 SDO 写入一个 32 位参数，可选回读校验 | `kEnterError` |
| `StateInspectionActivity` | 读取从站实际 EtherCAT 状态和 AL status code | `kKeepControllerState` |

Activity 设计原则：

- **不改变 `ControllerState`**：Activity 只能改变单个从站状态或执行读写，主站状态只能由 `TransitionTo()` 改变。
- **失败策略可配置**：每个 Activity 声明失败后主站如何处理（保持状态 / 进 Error / Shutdown）。
- **同步执行**：当前阶段 `ExecuteActivity()` 同步执行，一个 Activity 未完成前不能启动另一个。

---

### 4.5 SlaveNode（从站节点）

`src/slave_node/slave_node.h`

每个扫描到的从站对应一个 `SlaveNode`。它负责：

- 维护本从站的 EtherCAT 通信状态
- 维护本从站的 PDO 映射
- 提供 SDO 读写能力
- 挂载对应的 `DeviceProfile`，在周期循环中把业务语义量编码成 PDO 字节，或把 PDO 字节解码成语义量

当前阶段（`Maintenance`）`SlaveNode` 是**占位对象**，只保存 `SlaveInfo`，支持 SDO 和状态查询，不分配 PDO buffer，不创建 `DeviceProfile`。

---

### 4.6 SlaveNodeManager（从站节点管理器）

`src/slave_node/slave_node_manager.h`

职责：

- 根据从站数量创建所有 `SlaveNode`
- 通过 `SlaveTypeDetector` 识别设备类型，通过 `SlaveConfigFactory` 生成对应配置
- 按索引/名字查找从站
- 批量更新所有从站 PDO
- 批量请求状态切换

---

### 4.7 ServoAxis（电机语义对象）

`src/servo_axis/servo_axis.h`

纯电机语义对象，**不感知 EtherCAT / PDO / SDO**。后续 CiA402 状态机可以在 `ServoAxis` 基础上再包一层实现，不影响现有框架。

### 4.8 CyclicRunner（通用周期任务调度器）

`src/cyclic/cyclic_runner.h`

独立的周期线程封装，当前阶段未启用。后续 PDO 阶段由 `main` 创建并管理。

---

### 4.9 Logger（异步日志）

`src/utils/logger.h`

- 五级日志：Debug / Info / Warn / Error / Fatal
- 输出带时间戳、级别、源文件行号
- 默认输出到 `logger/ecat.log`，自动创建目录
- **异步**：调用方只入队，后台线程负责写入，不阻塞业务线程

## 5. 线程模型

当前阶段为**单线程命令处理模型**：

```text
main 线程
    ├── 初始化 EcatApplication 到 AdapterReady
    ├── EcatApplication::Run()
    │       ├── ReadCommand() 每 100ms 读取 stdin
    │       ├── switch (ControllerState)
    │       │       ├── AdapterReady:  scan / stop
    │       │       ├── Scanned:      config / stop
    │       │       ├── Maintenance:  CheckSlaveStates() + diagnose/param/inspect/start/stop
    │       │       ├── Operational:  RunOneCycle() + CheckSlaveStates() + stop
    │       │       └── Error:        stop
    │       └── 状态迁移 / ExecuteActivity()
    └── 收到 exit / EOF / SIGINT 后调用 app->Shutdown()
```

说明：

- 所有 `ControllerState` 迁移和 Activity 执行都在 `main` 线程串行发生。
- `EcMaster` 内部仍有互斥锁保护 SOEM 上下文，但当前单线程模型下不会并发访问。
- 后续 PDO 阶段再引入周期线程，由 `main` 创建并管理。

## 6. 数据流

### 6.1 命令驱动状态迁移

```text
用户命令
    │
    ▼
EcatApplication::Run()
    │
    ├── scan  ──► controller.Scan()              ──► ControllerState: kAdapterReady -> kScanned
    ├── config ──► controller.EnterMaintenance() ──► ControllerState: kScanned -> kMaintenance
    ├── start ──► controller.StartOperation()    ──► ControllerState: kMaintenance -> kOperational
    ├── diagnose ──► ExecuteActivity(SdoDiagnosticsActivity)
    ├── param   ──► ExecuteActivity(SdoParameterActivity)
    ├── inspect ──► ExecuteActivity(StateInspectionActivity)
    └── stop   ──► controller.Stop()             ──► ControllerState: -> kUninitialized
```

```text
上层应用
    │
    ▼ 设置目标位置、使能等

    │

    │
    ▼ 按 PdoMapping 写入/读出 buffer

    │
    ▼ WriteOutput / ReadInput 与 IOmap 交换

    │
    ▼ EtherCAT 帧

```

## 9. 关键技术决策

### 9.1 为什么把命令分发和主循环交给 EcatApplication 而不是 main？

`main.cpp` 只负责命令读取、信号注册和程序生命周期，具体命令如何映射到状态迁移 / Activity 由 `EcatApplication` 负责。这样命令通道改变时（如从 stdin 换为 socket），业务逻辑不需要改动。

### 9.2 为什么不维护独立应用状态？

`ControllerState` 已经能完整描述主站生命周期阶段，额外增加 `ApplicationState` 会造成状态冗余和维护负担。命令合法性直接基于 `ControllerState` 判断，架构更简洁。

### 9.3 为什么 Activity 不改变 ControllerState？

如果把维护操作（如 FOE 升级、SDO 诊断）也塞进状态机，状态枚举会爆炸，转换关系会混乱。Activity 机制让维护操作横向扩展，主状态机保持清晰线性。

### 9.4 为什么把过渡状态从 ControllerState 中删除？

`kPdoConfigured` / `kSafeOp` / `kDcConfigured` 是 `StartOperation()` 的内部子步骤，不是稳定工况。暴露为状态会增加命令校验、状态显示、文档说明的复杂度，但给用户带来的价值很小。隐藏后状态机更清晰。

### 9.5 为什么 ServoAxis 不直接操作 PDO？

把“电机语义”和“通信细节”分离后：

- 更换驱动器或 PDO 映射时，`ServoAxis` 不需要改。
- 后续加 CiA402 状态机时，不会影响现有的 PDO 数据桥接。

### 9.6 为什么一从站一 SlaveNode？

每个从站的 PDO 映射、EtherCAT 状态、SDO 参数都可能不同。独立节点后：

- 可以单独控制某个从站。
- 可以单独更换某个从站的配置。
- 多从站代码清晰、可维护。
