# MoDriverPC_EtherCAT 架构说明文档

> 版本：v3.2（增加 ActivityScheduler / ProcessDataEngine，完成边界收缩阶段 1~3）  
> 目标：基于 SOEM 构建面向多从站 EtherCAT 主站，支持命令驱动的主站生命周期与维护 Activity。  
> 运行平台：Linux（x86_64 / ARM64）  
> 编程语言：C++17  
> 通信协议：EtherCAT

---

## 1. 项目目标

本项目作为上位机 EtherCAT 主站，用于驱动支持 EtherCAT 的伺服驱动器、IO 模块等设备。当前阶段重点：

1. **命令驱动的主站生命周期管理**：通过命令逐步推进 `AdapterReady → Scanned → Maintenance`。
2. **维护 Activity 框架**：在 `Maintenance` 状态下执行 SDO 诊断、参数写入、状态巡检等维护操作。
3. 后续再扩展 PDO 周期运行、CiA402 状态机等电机控制逻辑。

---

## 2. 设计原则

| 原则 | 说明 |
|------|------|
| **分层解耦** | 硬件访问、总线通信、从站管理、电机语义、命令交互分层独立。 |
| **主状态单一维护** | `ControllerState` 是主站生命周期的唯一状态源，`EcatApplication` 基于它判断命令合法性。 |
| **状态只表示稳定工况** | PDO 配置、SafeOp、DC 同步等过渡步骤不暴露为 `ControllerState`。 |
| **Activity 与生命周期正交** | 维护操作封装为 `EcatActivity`，不改变 `ControllerState`。 |
| **一从站一节点** | 每个扫描到的从站对应一个 `SlaveNode`，独立管理通信状态、PDO 映射和数据。 |
| **电机语义与通信分离** | `ServoAxis` 只描述电机命令/反馈，不感知 EtherCAT/PDO/SDO。 |
| **命令通道可替换** | 当前用 `stdin` 交互，后续可替换为命名管道 / socket / RPC，不影响主状态逻辑。 |
| **边界收缩** | `EcatController` / `ActivityScheduler` / `ProcessDataEngine` / `EcatApplication` 严格划分职责，只通过最小、稳定的接口通信。 |
| **异步日志** | 日志系统不阻塞业务线程。 |

## 4. 核心模块职责

### 4.1 EcMaster（EtherCAT 主站层）

`src/ec_master/ec_master.h`

职责：

- 初始化 SOEM、绑定网卡
- 扫描从站、配置 PDO 映射、DC 时钟
- 全局/单站 EtherCAT 状态切换
- PDO 收发
- SDO 读写

> **注意**：Init / PreOP / SafeOP / OP 四个 EtherCAT 状态由 `EcMaster` 管理，`SlaveNode` 可以请求单个从站的状态切换，但底层切换能力来自 `EcMaster`。

---

### 4.2 EcatController（总线应用控制层）

`src/ec_controller/ec_controller.h`

职责：

- 维护 `ControllerState` 线性生命周期状态机（6 个稳定状态）
- 提供细粒度生命周期接口：`InitializeAdapter()` / `Scan()` / `EnterMaintenance()` / `StartOperation()` / `Stop()`
- 提供 `Initialize()` 便捷接口，内部顺序执行 `InitializeAdapter → Scan → EnterMaintenance`
- 持有 `EcMaster` 和 `SlaveNodeManager`
- 提供 `ForEachSlaveNode()` 安全遍历节点，不对外暴露 `SlaveNodeManager`
- 提供 `RequestErrorState()`，供 `ActivityScheduler` / `ProcessDataEngine` 等调度/监控类请求进入错误状态
- **不再负责** Activity 调度、PDO 周期、从站状态监控（已剥离到 `ActivityScheduler` / `ProcessDataEngine`）

关键接口：

```cpp
class EcatController {
public:
    bool Initialize(const EcMasterConfig &config);           // 到 Maintenance 的便捷接口
    bool InitializeAdapter(const EcMasterConfig &config);    // 到 AdapterReady
    bool Scan();                                             // AdapterReady -> Scanned
    bool EnterMaintenance();                                 // Scanned -> Maintenance
    bool StartOperation();                                   // Maintenance -> Operational
    void Stop();                                             // 回滚到 Uninitialized

    ControllerState GetState() const;
    size_t GetSlaveCount() const;
    static const char *StateToString(ControllerState state);

    // 安全遍历节点，不暴露 SlaveNodeManager
    void ForEachSlaveNode(const std::function<void(SlaveNode &)> &callback);

    // 仅供 ActivityScheduler / ProcessDataEngine 调用
    void RequestErrorState(const std::string &reason);

    // 仅供底层周期模块使用，EcatApplication 禁止调用
    EcMaster &GetEcMaster();
    SlaveNodeManager &GetSlaveNodeManager();
};
```

> **边界规则**：`EcatController` 只做状态迁移和状态保护。`RunOneCycle()` / `CheckSlaveStates()` / `ExecuteActivity()` 已分别移入 `ProcessDataEngine` 和 `ActivityScheduler`。

---

### 4.3 ActivityScheduler（Activity 调度器）

`src/activity/activity_scheduler.h`

职责：

- 负责 `EcatActivity` 的启动检查、同步执行和失败处理
- 读取 `controller_.GetState()` 判断 `CanStart()`
- 根据 `GetFailurePolicy()` 决定失败后行为：
  - `kKeepControllerState`：保持当前状态
  - `kEnterError`：调用 `controller_.RequestErrorState()`
  - `kShutdown`：调用 `controller_.Stop()`
- 不直接访问 `EcMaster` / `SlaveNodeManager` / `SlaveNode`（除非通过 Activity）

关键接口：

```cpp
class ActivityScheduler {
public:
    explicit ActivityScheduler(EcatController &controller);
    bool Execute(std::unique_ptr<EcatActivity> activity);
    bool IsRunning() const;
};
```

---

### 4.4 ProcessDataEngine（周期任务引擎）

`src/cyclic/process_data_engine.h`

职责：

- 根据 `ControllerState` 执行 PDO 周期收发和从站状态监控
- `RunOnce()`：仅在 `kOperational` 下执行一次 PDO 周期
- `CheckSlaveStates()`：在 `kMaintenance` / `kOperational` 下检测从站是否掉线
- 发现异常时调用 `controller_.RequestErrorState()`
- 不发起任何状态迁移

关键接口：

```cpp
class ProcessDataEngine {
public:
    ProcessDataEngine(EcatController &controller, EcMaster &master,
                      SlaveNodeManager &node_manager);
    void RunOnce();
    void CheckSlaveStates();
};
```

周期流程：

```cpp
void ProcessDataEngine::RunOnce() {
    if (controller_.GetState() != ControllerState::kOperational) return;

    node_manager_.UpdateAllOutputs();  // 所有 SlaveNode：ServoAxis → IOmap
    master_.RunOneCycle();              // PDO 收发
    node_manager_.UpdateAllInputs();   // 所有 SlaveNode：IOmap → ServoAxis
}
```

### 4.5 EcatApplication（状态机单步执行器）

`src/app/ecat_application.h`

职责：

- 持有 `EcatController`、`ActivityScheduler`、`ProcessDataEngine` 的引用
- 通过 `CommandReader` 接口读取命令，不依赖具体输入通道
- 提供一个单步执行接口 `Run()`，由上层按周期调用
- 在 `Run()` 内部使用 switch-case 根据 `ControllerState` 分发命令、Activity 和周期任务
- 不维护独立应用状态，直接基于 `ControllerState` 判断命令是否允许执行
- 不拥有自己的线程，不阻塞调用方
- 把命令映射为三类操作：
  - **状态迁移**：`scan` / `config` / `start` / `stop` → `EcatController`
  - **维护 Activity**：`diagnose` / `param` / `inspect` → `ActivityScheduler`
  - **周期任务**：`CheckSlaveStates()` / `RunOnce()` → `ProcessDataEngine`
- 通过 `controller_.ForEachSlaveNode()` 安全遍历节点，不直接访问 `SlaveNodeManager`

当前支持的命令：

| 命令 | 允许的 ControllerState | 操作 |
|---|---|---|
| `scan` | `kAdapterReady` | `controller.Scan()` |
| `config` | `kScanned` | `controller.EnterMaintenance()` |
| `diagnose` | `kMaintenance` | 对所有 SlaveNode 通过 `scheduler_.Execute()` 执行 `SdoDiagnosticsActivity` |
| `param <idx> <sub> <val>` | `kMaintenance` | 对所有 SlaveNode 通过 `scheduler_.Execute()` 执行 `SdoParameterActivity` |
| `inspect` | `kMaintenance` | 对所有 SlaveNode 通过 `scheduler_.Execute()` 执行 `StateInspectionActivity` |
| `start` | `kMaintenance` | `controller.StartOperation()` |
| `stop` | 非 `kUninitialized` | `controller.Stop()` |
| `help` | 任意 | 打印帮助 |
| `exit` / `quit` | 任意 | 退出程序 |

> **命令通道可替换**：`EcatApplication` 依赖 `CommandReader` 接口。当前注入 `StdinCommandReader`，后续可替换为 `SocketCommandReader` / `PipeCommandReader` 等，无需修改 `EcatApplication::Run()` 中的状态分发逻辑。
>
> **边界规则**：`EcatApplication` 禁止直接调用 `TransitionTo()` / `DoXxx()` / `RequestErrorState()` / `GetSlaveNodeManager()` / `GetEcMaster()`，也不直接执行 Activity 或 PDO 周期。

单步执行结构：

```cpp
bool EcatApplication::Run() {
    // 非阻塞尝试读取命令
    auto result = command_reader_->Read(command, 0);

    if (result == ReadResult::kEof) {
        return false;  // 通知上层退出
    }

    bool has_command = (result == ReadResult::kOk);

    if (has_command && (command == "exit" || command == "quit")) {
        return false;  // 通知上层退出
    }

    switch (controller_.GetState()) {
    case kAdapterReady:  HandleAdapterReadyState(...);  break;
    case kScanned:       HandleScannedState(...);       break;
    case kMaintenance:   HandleMaintenanceState(...);   break;  // engine_.CheckSlaveStates()
    case kOperational:   HandleOperationalState(...);   break;  // engine_.RunOnce() + engine_.CheckSlaveStates()
    case kError:         HandleErrorState(...);         break;
    }

    return true;  // 继续运行
}
```

创建与调用方式：

```cpp
EcatController controller;
ActivityScheduler scheduler(controller);
ProcessDataEngine engine(controller, controller.GetEcMaster(),
                         controller.GetSlaveNodeManager());

auto app = std::make_unique<EcatApplication>(
    std::make_unique<StdinCommandReader>(), controller, scheduler, engine);

while (g_running) {
    if (!app->Run()) break;
    std::this_thread::sleep_for(std::chrono::milliseconds(1));
}
```

### 4.6 CommandReader（命令读取抽象）

`src/app/command_reader.h`

`CommandReader` 是命令读取器的抽象接口，把命令输入通道与 `EcatApplication` 解耦：

```cpp
enum class ReadResult { kOk, kTimeout, kEof, kError };

class CommandReader {
public:
    virtual ~CommandReader() = default;
    virtual ReadResult Read(std::string &command, int timeout_ms) = 0;
};
```

当前实现：

| 实现 | 文件 | 说明 |
|---|---|---|
| `StdinCommandReader` | `src/app/stdin_command_reader.h` | 从 stdin 读取，使用 `poll()` 实现超时；当前由 `Run()` 以非阻塞方式调用 |

注入方式：

```cpp
auto app = std::make_unique<EcatApplication>(
    std::make_unique<StdinCommandReader>(), controller, scheduler, engine);
```

设计收益：

- `EcatApplication` 不依赖 `stdin` / `poll()` 细节。
- 替换命令通道时不需要修改 `EcatApplication`。
- 单元测试可以注入 mock reader，按顺序返回命令。

### 4.7 Activity（维护活动层）

`src/activity/activity.h`

`EcatActivity` 是维护活动基类，封装在主站某个状态下执行的操作：

```cpp
class EcatActivity {
public:
    virtual const char *GetName() const = 0;
    virtual bool CanStart(ControllerState state) const = 0;
    virtual bool Execute() = 0;
    virtual ActivityFailurePolicy GetFailurePolicy() const;
};
```

当前已实现：

| Activity | 作用 | 失败策略 |
|---|---|---|
| `SdoDiagnosticsActivity` | 读取 `0x1000`、`0x1018:01/02/03` 验证 SDO 通信 | `kKeepControllerState` |
| `SdoParameterActivity` | 通过 SDO 写入一个 32 位参数，可选回读校验 | `kEnterError` |
| `StateInspectionActivity` | 读取从站实际 EtherCAT 状态和 AL status code | `kKeepControllerState` |

Activity 设计原则：

- **不改变 `ControllerState`**：Activity 只能改变单个从站状态或执行读写，主站状态只能由 `TransitionTo()` 改变。
- **失败策略可配置**：每个 Activity 声明失败后主站如何处理（保持状态 / 进 Error / Shutdown）。
- **同步执行**：当前阶段由 `ActivityScheduler::Execute()` 同步执行，一个 Activity 未完成前不能启动另一个。

---

### 4.8 SlaveNode（从站节点）

`src/slave_node/slave_node.h`

每个扫描到的从站对应一个 `SlaveNode`。它负责：

- 维护本从站的 EtherCAT 通信状态
- 维护本从站的 PDO 映射
- 提供 SDO 读写能力
- 挂载对应的 `DeviceProfile`，在周期循环中把业务语义量编码成 PDO 字节，或把 PDO 字节解码成语义量

当前阶段（`Maintenance`）`SlaveNode` 是**占位对象**，只保存 `SlaveInfo`，支持 SDO 和状态查询，不分配 PDO buffer，不创建 `DeviceProfile`。

---

### 4.9 SlaveNodeManager（从站节点管理器）

`src/slave_node/slave_node_manager.h`

职责：

- 根据从站数量创建所有 `SlaveNode`
- 通过 `SlaveTypeDetector` 识别设备类型，通过 `SlaveConfigFactory` 生成对应配置
- 按索引/名字查找从站
- 批量更新所有从站 PDO
- 批量请求状态切换

### 4.11 CyclicRunner（通用周期任务调度器）

### 4.12 Logger（异步日志）

`src/utils/logger.h`

## 5. 线程模型

当前阶段为**单线程周期调用模型**：

```text
main 线程
    ├── 创建并注入 EcatController / ActivityScheduler / ProcessDataEngine
    ├── 初始化 EcatApplication 到 AdapterReady
    ├── 周期循环（默认 1ms）
    │       ├── EcatApplication::Run()  // 单步执行
    │       │       ├── CommandReader::Read() 非阻塞读取 stdin
    │       │       ├── switch (ControllerState)
    │       │       │       ├── AdapterReady:  scan / stop
    │       │       │       ├── Scanned:      config / stop
    │       │       │       ├── Maintenance:  engine.CheckSlaveStates() + scheduler.Execute(Activity) / start / stop
    │       │       │       ├── Operational:  engine.RunOnce() + engine.CheckSlaveStates() + stop
    │       │       │       └── Error:        stop
    │       │       └── 状态迁移 / scheduler.Execute(Activity) / engine 周期任务
    │       └── std::this_thread::sleep_for(1ms)
    └── 收到 exit / EOF / SIGINT 后调用 app->Shutdown()
```

说明：

- `EcatApplication` 不拥有自己的线程，`Run()` 是单步执行函数。
- 调用方（当前为 `main` 线程）按固定周期调用 `Run()`，周期精度由调用频率决定。
- 所有 `ControllerState` 迁移和 Activity 执行都在调用线程串行发生。
- `EcMaster` 内部仍有互斥锁保护 SOEM 上下文，但当前单线程模型下不会并发访问。
- 后续如需要，可把周期调用逻辑提取到独立线程，而 `EcatApplication::Run()` 本身不需要改动。

## 6. 数据流

### 6.1 命令驱动状态迁移

```text
用户命令
    │
    ▼
EcatApplication::Run()
    │
    ├── scan  ──► controller.Scan()              ──► ControllerState: kAdapterReady -> kScanned
    ├── config ──► controller.EnterMaintenance() ──► ControllerState: kScanned -> kMaintenance
    ├── start ──► controller.StartOperation()    ──► ControllerState: kMaintenance -> kOperational
    ├── diagnose ──► scheduler_.Execute(SdoDiagnosticsActivity)
    ├── param   ──► scheduler_.Execute(SdoParameterActivity)
    ├── inspect ──► scheduler_.Execute(StateInspectionActivity)
    └── stop   ──► controller.Stop()             ──► ControllerState: -> kUninitialized
```

### 6.2 PDO 数据流（后续 PDO 阶段启用）

```text
上层应用
    │
    ▼ 设置目标位置、使能等
ServoAxis（电机语义，仅 Servo 类型）
    │
    ▼ 周期更新时编码为 PDO 原始值
DeviceProfile（ServoProfile / IoProfile / UnknownProfile）
    │
    ▼ 按 PdoMapping 写入/读出 buffer
SlaveNode（PDO 映射 + 数据桥接）
    │
    ▼ WriteOutput / ReadInput 与 IOmap 交换
EcMaster（SOEM 周期收发）
    │
    ▼ EtherCAT 帧
从站硬件
```

## 9. 关键技术决策

### 9.1 为什么把命令分发交给 EcatApplication 而不是 main？

`main.cpp` 只负责程序生命周期和周期调度，具体命令如何映射到状态迁移 / Activity 由 `EcatApplication::Run()` 负责。这样命令通道改变时（如从 stdin 换为 socket），业务逻辑不需要改动。

### 9.2 为什么 `Run()` 是单步执行而不是内部循环？

把 `Run()` 设计为单步执行函数后：

- `EcatApplication` 不拥有自己的线程，不阻塞调用方。
- 调用频率由上层决定，可灵活适配不同周期要求。
- 后续可把周期调用逻辑提取到独立线程，而 `EcatApplication` 本身不需要改动。
- 不需要在 `EcatApplication` 里维护 `shutdown_requested_` 或 `RequestShutdown()` 等退出控制逻辑。

### 9.3 为什么不维护独立应用状态？

`ControllerState` 已经能完整描述主站生命周期阶段，额外增加 `ApplicationState` 会造成状态冗余和维护负担。命令合法性直接基于 `ControllerState` 判断，架构更简洁。

### 9.4 为什么 Activity 不改变 ControllerState？

如果把维护操作（如 FOE 升级、SDO 诊断）也塞进状态机，状态枚举会爆炸，转换关系会混乱。Activity 机制让维护操作横向扩展，主状态机保持清晰线性。

### 9.5 为什么把过渡状态从 ControllerState 中删除？

`kPdoConfigured` / `kSafeOp` / `kDcConfigured` 是 `StartOperation()` 的内部子步骤，不是稳定工况。暴露为状态会增加命令校验、状态显示、文档说明的复杂度，但给用户带来的价值很小。隐藏后状态机更清晰。

### 9.6 为什么 ServoAxis 不直接操作 PDO？

把“电机语义”和“通信细节”分离后：

- 更换驱动器或 PDO 映射时，`ServoAxis` 不需要改。
- 后续加 CiA402 状态机时，不会影响现有的 PDO 数据桥接。

### 9.7 为什么一从站一 SlaveNode？

每个从站的 PDO 映射、EtherCAT 状态、SDO 参数都可能不同。独立节点后：

- 可以单独控制某个从站。
- 可以单独更换某个从站的配置。
- 多从站代码清晰、可维护。

### 9.8 为什么把 Activity 调度和周期任务从 EcatController 剥离？

`EcatController` 原本同时承担状态迁移、Activity 调度、PDO 周期、从站监控四项职责，违反单一职责原则。剥离后：

- `EcatController` 只维护状态机和状态保护，接口稳定。
- `ActivityScheduler` 只负责 Activity 执行与失败处理，后续可自然扩展为异步/优先级调度。
- `ProcessDataEngine` 只负责 PDO 和状态监控，可独立测试和替换调度策略。
- `EcatApplication` 通过最小接口编排三者，不越界访问底层节点/硬件。
